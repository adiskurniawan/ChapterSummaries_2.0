Summary
1. Benchmark is a rendered page with two static sample tables (Table1, Table2).
2. Current output is a unified app shell with no static tables and a dynamic pipeline (PasteToTable).
3. Both preserve a tables-viewer and a TOC but differ in structure, IDs, and initialization points.
DOM structure
1. Benchmark contains two concrete .table-wrapper blocks with id anchors #Table1 and #Table2 and visible rows.
2. Current output uses placeholders: #tables-placeholder, #page-area, and generates wrappers at runtime. No static sample rows present.
3. Many container nesting and class attributes are compatible but not identical.
Buttons / IDs
1. Benchmark exposes visible buttons like copyAllTablesBtn (visible), copyAllMdBtn hidden alias, convertBtn inline with handlers on elements in-place.
2. Current output centralizes controls: convertBtn, listBtn, saveBtn, saveMdBtn, copyAllTablesBtn (hidden), and moves paste textarea inside a header form. Some IDs differ (searchBox vs search / searchBox in benchmark).
3. Alias delegation script exists in both but current output duplicates it (kept for compatibility).
Paste / Convert workflow
1. Benchmark has no paste UI shown in header; convert assumed bound to a visible control like convertBtn in-page.
2. Current output provides #paste textarea inside a form #render-form with onsubmit prevented and convertBtn designed to call client convert or route to /api/render.
3. This changes where code must look for pasted content and where convert handler attaches.
TOC behavior
1. Benchmark TOC: #tocBar with direct anchors populated from static captions; #toc exists at top-level too.
2. Current: provides both #toc and #tocBar but #tocBarList and #tocList are empty placeholders for dynamic population. PTToc.init configuration added to cope with both.
Labels / captions (relevant to labels.json)
1. Benchmark has captions already present (.table-caption elements with ids Table1,Table2).
2. Current output relies on runtime fetch/apply logic (script.list.js, fetchAndApplyLabelsToDocument) to load labels.json and insert captions. Static sample captions are absent so labels.json is the source of captions at runtime.
3. Because current output does not include static tables, failure to fetch labels.json results in no applied captions. The labels loader must find a candidate path that matches deployed asset layout (see added /saved/... and origin-derived paths in revised script).
Scripts & load order
1. Benchmark loads assets/script.js (single bundle) plus worker.js and xlsx in specific order.
2. Current splits logic across script.toc.js, script.core.js, script.table.js, script.save.js, script.list.js. This modularization changes when labels loader runs relative to DOM population.
3. Ensure script.list.js runs after DOM elements that labels should target are inserted or that it retries when DOM changes.
Accessibility & attributes
1. Benchmark uses explicit caption elements preceding tables and anchors like id="Table1" for TOC links.
2. Current preserves ARIA roles and attempts to initialize PTToc with headingSelector: 'h3, .table-caption, .table-heading'. If labels are applied as caption elements inserted before tables, PTToc should pick them up when rebuilt.
Behavioral compatibility & safety
1. Benchmark assumes static content; current must handle appended HTML from server or client convert. Appending vs replacing is important (current routeHtmlToAreas tries to detect full-page HTML vs fragment).
2. If server returns HTML with static .table-wrapper nodes, script.list.js must run after insertion to apply labels. MutationObserver logic exists in current to reveal wrappers but ensure labels loader also retries after DOM mutations.

3. Ensure ENABLE_CACHE_BUSTER or query param not blocking expected path (debug with window.__ptt_list.setDebug(true)).
4. Confirm fetchWithTimeout has sufficient timeout and network allowed for labels fetch (revised to 8s).
