main.py — Per-Function Technical Breakdown
Module purpose & contract:

main.py is the sole entry point and orchestration layer for the DeckGenerator project. It owns the DeckBuilder class (pipeline coordinator), the create_presentation() public API function, the parse_args() CLI argument parser, and the main() top-level error handler.

Responsibility boundary (strict):
• OWNS: pipeline sequencing (load → validate → plan → render → save), CLI argument parsing, top-level error handling, Presentation() construction, .pptx file writing, output directory creation
• DOES NOT OWN: content parsing (deck.py:DeckSourceLoader), validation rules (deck.py:DeckValidator), metadata enrichment (deck.py:DeckPlanner), shape drawing (render.py:SlideRenderer), theme/layout configuration (config.py), type resolution (taxonomy.py)

Import surface:
• From deck.py: DeckValidator, DeckSourceLoader, DeckPlanner, SlideSpec, DeckValidationError, PresentationBuildResult
• From render.py: SlideRenderer, THEME, LAYOUT
• From python-pptx: Presentation, SlideLayout
• From stdlib: Path (pathlib), argparse, logging, Sequence / Optional / Union (typing)
• MUST NOT import from taxonomy.py or config.py directly — those are consumed transitively through deck.py and render.py

Module-level side effects:
• In the original monolith, logging.basicConfig(...) is called at module level guarded by if not logging.getLogger().handlers. In the 6-file split, this SHOULD move into main() or a dedicated logging_config.py to ensure import main is side-effect free for programmatic consumers (notebooks, test harnesses, batch scripts).
• The logger = logging.getLogger(__name__) call is safe at module level — it creates a logger but emits no output until a handler is attached.

Approximate size: ~130 lines (the smallest module in the project).
DeckBuilder.__init__(self, output_file, content_file) — Constructor, path resolution, dependency wiring

Purpose & contract:
Initialize the orchestrator with resolved output and input paths, and instantiate the three domain collaborators. Must resolve content_file through the fallback chain before storing. Must NOT perform any I/O, parsing, rendering, or logging at construction time — all heavy work is deferred to build() or save().

Signature:
def __init__(self, output_file: Union[str, Path] = OUTPUT_FILE, content_file: Union[str, Path] = CONTENT_FILE) -> None

Instance attributes created:
• self.output_file: Path — resolved output path (via Path(output_file))
• self.content_file: Path — resolved input path (via _resolve_default_content_path(Path(content_file)))
• self.validator: DeckValidator — stateless validation engine
• self.loader: DeckSourceLoader — stateless multi-format parser
• self.planner: DeckPlanner — stateless metadata enrichment engine

Primary invariants (must/shall):
1. self.content_file MUST be resolved through _resolve_default_content_path before assignment — the raw argument is never stored directly. This means builder.content_file may differ from the content_file argument if a fallback sibling file was found.
2. Construction MUST be side-effect free: no file reads, no logging beyond DEBUG, no network calls, no directory creation. This allows safe construction in REPL sessions, test fixtures, and batch loops.
3. The three collaborators MUST be stateless or safely re-entrant — DeckBuilder may be instantiated multiple times in tests or batch scripts without interference. No class-level mutable state is shared between instances.
4. output_file parent directory is NOT created at construction time — directory creation is deferred to save(). This prevents empty directories from being created when the pipeline fails before rendering.
5. Path(output_file) MUST accept both str and Path inputs. If output_file is None, Path(None) raises TypeError — this is an unrecoverable programming error and SHOULD propagate immediately.

Failure modes & recovery:
• content_file path does not exist after fallback resolution → invalid path stored silently; error surfaces at load_slide_specs() as DeckValidationError("Content file not found: {path}"). Intentional: allows callers to inspect builder.content_file before invoking the pipeline.
• output_file = "" → Path("") produces Path(".") (current directory) — this will fail at save() time when Presentation.save(".") raises IsADirectoryError. Not caught at construction.
• output_file = None → Path(None) raises TypeError at construction. Unrecoverable.
• content_file = None → Path(None) raises TypeError at construction. Unrecoverable.

Implementation notes:
• The three collaborators are instantiated unconditionally — there is no lazy initialization. This is correct because all three are lightweight (no I/O in their constructors) and always needed.
• The constructor does NOT validate that output_file has a .pptx extension. A user could pass "output.pdf" and python-pptx would write a .pptx-format file with a misleading extension. This is acceptable — the CLI help text specifies .pptx and the default has the correct extension.
• For batch processing (generating multiple decks from different content files), the recommended pattern is to create a new DeckBuilder per deck rather than mutating self.content_file between calls. The class is cheap to construct.

Cross-module interactions:
• Calls _resolve_default_content_path (own static method) which uses Path.exists() and Path.with_suffix() — stdlib only.
• Constructs DeckValidator(), DeckSourceLoader(), DeckPlanner() from deck.py — all have no-arg constructors with no side effects.
• Reads OUTPUT_FILE and CONTENT_FILE defaults from config.py (via import at module level).

Tests:
1. DeckBuilder() → output_file == Path(OUTPUT_FILE), content_file == Path(CONTENT_FILE) (or resolved fallback).
2. DeckBuilder(output_file="out.pptx", content_file="in.yaml") → correct paths stored.
3. DeckBuilder(content_file="missing.yaml") → no exception; content_file == Path("missing.yaml").
4. DeckBuilder(content_file="in.yaml") when in.json exists but in.yaml does not → content_file == Path("in.json").
5. DeckBuilder(content_file="in.md") when in.yaml exists but in.md does not → content_file == Path("in.yaml").
6. DeckBuilder(output_file=None) → TypeError.
7. DeckBuilder(output_file="") → no exception at construction; output_file == Path(".").
8. Two instances do not share mutable state: b1.validator is not b2.validator (separate instances).
9. No log output during construction (capture via caplog).
10. No files created during construction (verify via tmp_path inspection).
DeckBuilder._resolve_default_content_path(content_file: Path) -> Path — Static fallback chain for input file discovery

Purpose & contract:
Given a user-supplied content file path, return the first existing file in the fallback chain. Pure function (no side effects, no mutation, no logging). The chain supports the convention that a user can specify "slides" and the system finds "slides.yaml", "slides.json", or "slides.md" automatically.

Signature:
@staticmethod
def _resolve_default_content_path(content_file: Path) -> Path

Fallback chain (ordered, short-circuit on first hit):
1. content_file.exists() → return content_file immediately
2. If suffix is .yaml or .yml → try content_file.with_suffix(".json"); if exists → return it
3. If suffix is .md or .markdown → try content_file.with_suffix(".yaml"); if exists → return it
4. If suffix is .md or .markdown → try content_file.with_suffix(".json"); if exists → return it
5. If no suffix → try .yaml, .yml, .json, .md in order; return first that exists
6. Return content_file unchanged (defer error to loader)

Primary invariants (must/shall):
1. If content_file.exists() is True, MUST return it immediately — no further checks. This means an existing .txt file is returned as-is even though the loader may not support .txt.
2. Fallback checks are ordered and short-circuit on first hit — if both .json and .yaml siblings exist for a .md input, .yaml wins (checked first).
3. The method NEVER raises — if no file exists, it returns the original path unchanged, deferring the error to DeckSourceLoader.load() which produces a clear DeckValidationError.
4. Suffix comparison is case-insensitive: .YAML matches the .yaml branch. Implemented via content_file.suffix.lower().
5. MUST return a Path object in all cases — never None, never str.
6. MUST be a @staticmethod — no instance state accessed.

Failure modes & recovery:
• No failure modes — the method is total.
• The "failure" case (no file found) is handled by returning the original path. The downstream error message is: "Content file not found: {path}. Please ensure the file is in the correct location." — this message includes the unresolved path, which helps the operator understand what was searched for.
• If the filesystem is inaccessible (network drive disconnected), Path.exists() may raise OSError on some platforms. This is NOT caught — it propagates as an unhandled exception. In practice, Path.exists() returns False for inaccessible paths on Windows and Linux without raising.

Implementation notes:
• Path.with_suffix(".json") replaces the existing suffix: Path("data.yaml").with_suffix(".json") → Path("data.json"). It does NOT append: Path("data.tar.gz").with_suffix(".json") → Path("data.tar.json") (replaces .gz only). For the content file use case, single-suffix files are expected.
• The extensionless branch (step 5) supports the convention python main.py -i Examples_6_slides_content without specifying .yaml. This is a convenience feature for interactive use.
• The method does NOT check file readability (only existence). A file that exists but is not readable will fail at DeckSourceLoader._load_raw_payload with "Unable to read data file {path}: {exc}".

Cross-module interactions:
• None — pure stdlib Path operations. No imports from deck.py, render.py, config.py, or taxonomy.py.

Tests:
1. Existing file data.yaml → returned as-is.
2. Non-existing data.yaml, existing data.json → returns Path("data.json").
3. Non-existing data.md, existing data.yaml → returns Path("data.yaml").
4. Non-existing data.md, no data.yaml, existing data.json → returns Path("data.json").
5. Non-existing data.md, no siblings → returns Path("data.md") (original).
6. Non-existing extensionless data, existing data.yaml → returns Path("data.yaml").
7. Non-existing extensionless data, existing data.json (no .yaml) → returns Path("data.json").
8. Case-insensitive: non-existing data.YAML, existing data.json → returns Path("data.json").
9. Existing data.txt → returned as-is (no fallback for .txt).
10. Return type is always Path.
DeckBuilder._configure_presentation(prs: Presentation) -> None — Static PPTX metadata and dimension configuration

Purpose & contract:
Apply project-specific metadata and slide dimensions to a freshly constructed Presentation object. MUST be called exactly once per Presentation instance, immediately after construction and BEFORE any slides are added. This ordering is critical because python-pptx computes internal EMU coordinates relative to slide size at shape creation time — changing dimensions after slides exist corrupts all shape positions.

Signature:
@staticmethod
def _configure_presentation(prs: Presentation) -> None

Operations performed (ordered):
1. prs.slide_width = SLIDE_W — sets width to Inches(13.333) (12191712 EMU)
2. prs.slide_height = SLIDE_H — sets height to Inches(7.5) (6858000 EMU)
3. prs.core_properties.title = "NGO Financial Planning and Budgeting"
4. prs.core_properties.subject = "Executive month-end control deck"
5. prs.core_properties.author = "Finance Governance Team"

Primary invariants (must/shall):
1. Slide dimensions MUST be set before any add_slide call. Violation causes shape position corruption — shapes placed at Inches(5) on a 13.333"-wide slide would appear at a different relative position if the width is later changed to 10".
2. core_properties fields MUST be set to the project-specific strings. These appear in File → Properties in PowerPoint and in document metadata extractors.
3. This method MUST NOT add slides, modify layouts, touch the theme, or create shapes.
4. MUST be idempotent — calling twice with the same prs produces the same result without error.
5. MUST be a @staticmethod — no instance state accessed.

Failure modes & recovery:
• prs = None → AttributeError: 'NoneType' object has no attribute 'slide_width'. Programming error, propagates.
• prs is not a Presentation → AttributeError. Programming error, propagates.
• python-pptx internal XML corrupted → core_properties assignment may raise lxml.etree.Error. Unrecoverable, SHOULD propagate as RenderingError at the call site in render_presentation.
• SLIDE_W or SLIDE_H not Length instances → TypeError at assignment. Would indicate a bug in config.py.

Implementation notes:
• The hardcoded strings are project-specific (NGO finance theme). For multi-project reuse, these should move to config.py as DECK_TITLE, DECK_SUBJECT, DECK_AUTHOR constants.
• core_properties also supports category, comments, company, keywords, last_modified_by, revision — none are set here. PowerPoint sets last_modified_by and revision automatically on save.
• The default python-pptx Presentation() constructor creates a presentation with slide_width = Inches(10) and slide_height = Inches(7.5) (4:3 aspect ratio). This method overrides the width to 13.333" for 16:9. The height is already 7.5" but is set explicitly for clarity and safety.
• python-pptx stores dimensions as EMU (English Metric Units) internally: 1 inch = 914400 EMU. Inches(13.333) → 12191712 EMU. The slight imprecision from 13.333 vs 13.3333... is < 0.003" and matches PowerPoint's own 16:9 template.

Cross-module interactions:
• Reads SLIDE_W, SLIDE_H from config.py (imported at module level in the original; in the split, imported via render.py or directly).
• Operates on python-pptx Presentation object — no interaction with deck.py or taxonomy.py.

Tests:
1. Fresh Presentation() → after call: prs.slide_width == Inches(13.333), prs.slide_height == Inches(7.5).
2. prs.core_properties.title == "NGO Financial Planning and Budgeting".
3. prs.core_properties.subject == "Executive month-end control deck".
4. prs.core_properties.author == "Finance Governance Team".
5. Calling twice → idempotent, no error, same values.
6. Calling with None → AttributeError.
7. Calling after add_slide → dimensions change but existing shapes are NOT repositioned (known python-pptx behavior — this is why the invariant requires calling BEFORE add_slide).
DeckBuilder._resolve_blank_layout(prs: Presentation) -> SlideLayout — Static layout selection with quadruple fallback

Purpose & contract:
Select the blank (placeholder-free) slide layout from the presentation's layout collection. The rendering engine draws ALL shapes manually via SlideCanvas — it never uses layout placeholders. Therefore it requires a layout with zero placeholders to avoid unwanted default text boxes (e.g., "Click to add title") appearing on rendered slides.

Signature:
@staticmethod
def _resolve_blank_layout(prs: Presentation) -> SlideLayout

Fallback chain (ordered, 4 tiers):
1. Zero-placeholder scan: iterate prs.slide_layouts, return first layout where len(layout.placeholders) == 0. Wrapped in try/except per layout because some custom .potx templates raise KeyError or AttributeError during placeholder enumeration due to malformed XML.
2. Name-based match: iterate prs.slide_layouts, return first layout where layout.name.strip().lower() == "blank". Catches layouts that have placeholders but are named "Blank" (rare but possible in custom templates).
3. Index-based fallback: if len(prs.slide_layouts) > DEFAULT_LAYOUT_INDEX (6), return prs.slide_layouts[6]. Index 6 is "Blank" in the default Office template.
4. Last-resort: return prs.slide_layouts[-1]. Guarantees a valid return as long as the presentation has at least one layout (which python-pptx guarantees for default templates — always 11 layouts).

Primary invariants (must/shall):
1. MUST always return a valid SlideLayout — never None. The tier-4 fallback guarantees this.
2. Placeholder count check MUST be wrapped in try/except Exception per layout — some third-party .potx templates from SlideModel, Envato, or corporate brand kits have malformed placeholder XML that causes python-pptx to raise during len(layout.placeholders).
3. Name comparison MUST be case-insensitive AND stripped: layout.name.strip().lower() == "blank" matches "Blank", " blank ", "BLANK".
4. The method MUST NOT modify the layout or presentation — read-only inspection.
5. MUST be a @staticmethod — no instance state.
6. The returned layout is used for ALL slides in the deck — it is resolved once and passed to every prs.slides.add_slide(layout) call.

Failure modes & recovery:
• prs.slide_layouts is empty → IndexError on prs.slide_layouts[-1] in tier 4. Unrecoverable — indicates corrupted python-pptx installation or a programmatically constructed Presentation with no slide masters. In practice, Presentation() always creates 11 layouts from the default template.
• All layouts raise on len(placeholders) → tiers 1 and 2 fail, tier 3 or 4 catches. The returned layout may have placeholders, which would cause unwanted text boxes. Mitigation: the rendering engine draws shapes on top, so placeholder text is visually obscured but technically present in the XML. For clean output, use the default python-pptx template.
• layout.name is None → AttributeError on .strip(). Should be wrapped in try/except or guarded with getattr(layout, "name", ""). In the current implementation, the getattr guard is used.

Implementation notes:
• Default python-pptx template layout order (0-indexed):
0 = Title Slide (2 placeholders)
1 = Title and Content (2 placeholders)
2 = Section Header (2 placeholders)
3 = Two Content (3 placeholders)
4 = Comparison (5 placeholders)
5 = Title Only (1 placeholder)
6 = Blank (0 placeholders) ← tier 1 finds this
7 = Content with Caption (3 placeholders)
8 = Picture with Caption (3 placeholders)
9 = Vertical Title and Text (2 placeholders)
10 = Vertical Text (2 placeholders)
• With the default template, tier 1 always succeeds at index 6. Tiers 2–4 exist for custom templates where the blank layout may be at a different index or have a different name.
• DEFAULT_LAYOUT_INDEX = 6 is defined in config.py and is stable across Office versions (2013, 2016, 2019, 2021, 365).

Cross-module interactions:
• Reads DEFAULT_LAYOUT_INDEX from config.py.
• Operates on python-pptx Presentation.slide_layouts collection.
• No interaction with deck.py or taxonomy.py.

Tests:
1. Default Presentation() → returns layout at index 6 with len(placeholders) == 0.
2. Mock presentation where all layouts have placeholders but one is named "Blank" → returns that layout (tier 2).
3. Mock presentation where no layout has 0 placeholders and none named "Blank" → returns layout at index 6 (tier 3).
4. Mock presentation with 3 layouts, none matching → returns layout at index 2 (last, tier 4).
5. Layout that raises KeyError on len(placeholders) → skipped without error, next layout checked.
6. Layout with name = None → skipped in tier 2 without error.
7. Return type is SlideLayout.
8. Method does not modify prs (verify layout count unchanged after call).
DeckBuilder.load_slide_specs(self) -> list[SlideSpec] — Delegation to DeckSourceLoader with logging

Purpose & contract:
Load and normalize slide specifications from the content file. Delegates entirely to self.loader.load(self.content_file). Emits an INFO-level log line before loading. MUST propagate DeckValidationError from the loader without catching, wrapping, or transforming it — the caller (build() or save()) handles error reporting and exit codes.

Signature:
def load_slide_specs(self) -> list[SlideSpec]

Operations performed (ordered):
1. logger.info("Building slide specifications from source: %s", self.content_file) — lazy string formatting (the %s is only evaluated if INFO level is enabled)
2. return self.loader.load(self.content_file) — full delegation

Primary invariants (must/shall):
1. MUST log the source path at INFO level BEFORE delegating. This log line is the first visible output in a successful run and confirms which file is being read.
2. MUST NOT catch, suppress, wrap, or transform exceptions from the loader. DeckValidationError propagates as-is to build() → save() → main() where it is caught and reported.
3. MUST NOT cache results — each call re-reads the file from disk. This enables hot-reload workflows where the operator edits the YAML and re-runs without restarting the process.
4. The returned list MUST contain at least one SlideSpec — this is enforced by DeckSourceLoader.load() which raises DeckValidationError for empty results.
5. MUST NOT modify self.content_file or any instance state.

Failure modes & recovery (all from DeckSourceLoader.load):
• DeckValidationError("Content file not found: {path}. Please ensure the file is in the correct location.") — file does not exist after fallback resolution. Operator action: verify path, check working directory.
• DeckValidationError("Unable to read data file {path}: {exc}") — file exists but cannot be read (permission denied, locked by another process, I/O error). Operator action: check file permissions, close other editors.
• DeckValidationError("Unable to parse JSON file {path}: {exc}") — malformed JSON. Operator action: validate JSON syntax.
• DeckValidationError("Unable to parse data file {path}: {exc}") — malformed YAML. Operator action: validate YAML syntax (common issues: inconsistent indentation, unquoted colons, tab characters).
• DeckValidationError("Content file {path} was empty or contained no valid slides.") — file parsed successfully but contained no slide data. Operator action: add slide content.
• DeckValidationError("PyYAML is not installed. Please install it to read structured input.") — YAML file detected but PyYAML not available. Operator action: pip install pyyaml.
• All are terminal for the pipeline — no partial recovery is possible. The process exits with code 1 via main().

Implementation notes:
• This method exists as a named pipeline step for three reasons: (1) logging — the operator sees which file is being read; (2) testability — tests can override this method or mock self.loader to inject synthetic SlideSpec lists without touching the filesystem; (3) readability — build() reads as a clear four-step sequence.
• The lazy %s formatting in logger.info("...: %s", self.content_file) avoids string interpolation cost when INFO logging is disabled (e.g., in production with WARNING-level logging). This is a Python logging best practice.
• The method does NOT validate the returned specs — that is validate_slide_specs()'s responsibility in the next pipeline stage.

Cross-module interactions:
• Calls self.loader.load(self.content_file) → DeckSourceLoader.load in deck.py.
• DeckSourceLoader.load internally calls DeckSourceLoader._resolve_path, _load_raw_payload, normalize, and SlideFactory.create_from_sequence.
• Reads self.content_file (set in __init__).
• Uses logger (module-level logging.getLogger(__name__)).

Tests:
1. Valid YAML file with 3 slides → returns list[SlideSpec] of length 3.
2. Missing file → DeckValidationError with "Content file not found" in message.
3. Malformed YAML → DeckValidationError with "Unable to parse" in message.
4. Empty YAML (slides: []) → DeckValidationError with "empty or contained no valid slides".
5. INFO log line emitted with correct path (capture via caplog fixture: caplog.records[0].levelname == "INFO", path in message).
6. INFO log line NOT emitted when logging level is WARNING (verify via caplog at WARNING level).

DeckBuilder.validate_slide_specs(self, slide_specs: Sequence[SlideSpec]) -> list[SlideSpec] — Validation gate with warning emission

Purpose & contract:
Run structural, length, content-quality, and shape validations on the loaded slide specs. Emit WARNING-level log lines for each non-fatal issue. Raise DeckValidationError if any ERROR-level issues exist. Returns the input list unchanged (pass-through on success). This is a hard gate: the pipeline MUST NOT proceed to planning or rendering if validation fails.

Signature:
def validate_slide_specs(self, slide_specs: Sequence[SlideSpec]) -> list[SlideSpec]

Operations performed (ordered):
1. prepared = list(slide_specs) — shallow copy to prevent the validator from mutating the caller's sequence
2. result = self.validator.validate(prepared) — runs all 4 validation categories + aggregate checks
3. For each w in result.warnings: logger.warning("Slide %d: %s", w.slide_index, w.message) — emit all warnings
4. result.raise_if_invalid() — raises DeckValidationError if result.errors is non-empty
5. return prepared — pass-through on success

Primary invariants (must/shall):
1. MUST emit ALL warning-level log lines BEFORE calling raise_if_invalid(). This ensures operators see warnings even when the build ultimately fails due to errors. Warnings may provide context for why errors occurred (e.g., a warning about an unusually long title followed by an error about a missing subtitle on the same slide).
2. MUST call result.raise_if_invalid() which raises DeckValidationError with a multi-line message: one line per error in format "[{level}] Slide {slide_index}: {message}", prefixed with "Deck validation failed:\n".
3. MUST NOT modify, filter, reorder, or transform the input specs. The returned list contains the same SlideSpec objects (identity-preserving pass-through).
4. Validation is deterministic: same input always produces same warnings/errors. No randomness, no filesystem access, no time-dependent behavior.
5. The shallow copy list(slide_specs) ensures that if slide_specs is a generator or tuple, it is materialized into a list before validation. It also prevents the validator from accidentally mutating the caller's sequence (though DeckValidator.validate does not mutate its input).

Failure modes & recovery:
• DeckValidationError with multi-line message. Example:
"Deck validation failed:\n[ERROR] Slide 1: Title slide missing subtitle.\n[ERROR] Slide 3: Missing title.\n[ERROR] Slide 5: Comparison slide missing left/right points."
• This is terminal — the operator must fix the content file and re-run. There is no --force or --skip-validation flag.
• If slide_specs is empty → DeckValidator.validate returns a result with ERROR "No slide content found in the source." at slide_index 0 → raise_if_invalid() raises.

Implementation notes:
• The warning log format "Slide %d: %s" uses the 1-based slide_index from ValidationIssue, matching human slide numbering in PowerPoint. This allows the operator to open the content file and navigate to the correct slide entry.
• The DeckValidator is stateless — self.validator.validate(prepared) can be called multiple times with different inputs without interference.
• Warnings do NOT affect the return value — the same specs are returned whether or not warnings were emitted. Warnings are purely diagnostic.
• The validation categories and their severity levels are defined in deck.py:DeckValidator and documented in the deck.py technical breakdown. main.py does not define or modify any validation rules.

Cross-module interactions:
• Calls self.validator.validate(prepared) → DeckValidator.validate in deck.py.
• DeckValidator.validate internally calls _validate_structure, _validate_length, _validate_content_quality, _validate_shapes for each spec, then runs aggregate checks.
• Uses logger for warning emission.
• Raises DeckValidationError (defined in deck.py).

Tests:
1. Valid 3-slide deck → returns same list, no exceptions, no warnings.
2. Spec with missing title → DeckValidationError containing "[ERROR] Slide {n}: Missing title.".
3. Spec with title > 120 chars → warning logged ("Slide {n}: Title is unusually long."), no exception.
4. Multiple errors across slides → all listed in exception message, one per line.
5. Warnings logged BEFORE errors raised (verify via caplog record ordering: all WARNING records precede the exception).
6. Empty list → DeckValidationError with "No slide content found".
7. Return value identity: returned[0] is input[0] (same objects, not copies).
8. Input sequence is a tuple → materialized to list, validation succeeds.
9. No title slide in deck → warning "No title slide detected" logged, no exception.
10. Two title slides → warning "Multiple title slides detected: 2" logged, no exception.
DeckBuilder.render_presentation(self, slide_specs: Sequence[SlideSpec]) -> Presentation — PPTX construction and slide-by-slide rendering

Purpose & contract:
Construct a python-pptx Presentation, configure its dimensions and metadata, resolve the blank layout, and render each SlideSpec onto a new slide via the SlideRenderer dispatch. Returns the fully populated Presentation object. MUST NOT save to disk — that is save()'s responsibility. MUST render slides in the exact order of the input sequence.

Signature:
def render_presentation(self, slide_specs: Sequence[SlideSpec]) -> Presentation

Operations performed (ordered):
1. prs = Presentation() — construct fresh presentation from default template
2. self._configure_presentation(prs) — set dimensions and metadata
3. renderer = SlideRenderer(prs, THEME, LAYOUT) — construct renderer with strategy registry
4. blank_layout = self._resolve_blank_layout(prs) — resolve layout once
5. For each (page_no, spec) in enumerate(slide_specs, start=1):
a. slide = prs.slides.add_slide(blank_layout) — add blank slide
b. renderer.render(spec, slide, page_no) — dispatch to strategy
6. return prs

Primary invariants (must/shall):
1. Presentation() MUST be constructed fresh — never reuse a previous instance. python-pptx Presentation objects accumulate state (slides, relationships, media) and cannot be "reset".
2. _configure_presentation MUST be called before any add_slide. Violation corrupts shape positions.
3. _resolve_blank_layout MUST be called once and the same layout reused for all slides. Calling it per-slide would work but is wasteful and could theoretically return different layouts if the layout collection were mutated between calls (it is not, but the invariant prevents the risk).
4. Slides MUST be added and rendered in input order — page_no is 1-indexed and sequential. The page_no appears in the header band (top-right) and in error textboxes.
5. If a single slide's rendering raises, SlideRenderer.render catches it internally and draws an error textbox on that slide — the pipeline continues to the next slide. The Presentation is always returned, even if some slides have error placeholders. This is a deliberate resilience choice: a single malformed custom shape definition should not destroy a 20-slide deck.
6. This method MUST NOT write to disk. The Presentation object is returned in-memory.
7. The SlideRenderer is constructed once per call with the shared THEME and LAYOUT singletons from config.py. Strategy instances inside the renderer are stateless and reused across slides.

Failure modes & recovery:
• Presentation() construction fails → Exception propagates. Unrecoverable — indicates corrupted python-pptx installation or missing default template. In practice, this never fails with a standard pip install python-pptx.
• _configure_presentation fails → propagates. Would indicate config.py corruption (invalid SLIDE_W/SLIDE_H).
• _resolve_blank_layout fails → IndexError propagates. Would indicate empty layout collection (corrupted template).
• Individual slide rendering failure → caught by SlideRenderer.render, logged as ERROR "Failed to render slide {page_no} ({title}): {exc}", error textbox drawn. Build continues. The operator sees the error in the log and the error textbox in the output PPTX.
• prs.slides.add_slide(blank_layout) fails → propagates. Would indicate corrupted layout XML. Caught by SlideRenderer.render's try/except if it occurs during renderer.render, but add_slide is called BEFORE renderer.render in the loop, so this would propagate uncaught. Mitigation: this cannot happen with the default template.

Implementation notes:
• The SlideRenderer constructor builds the 18-entry strategy registry dict. This is O(1) and takes < 1ms. Strategy instances are created with (theme, layout) arguments and stored in the dict.
• prs.slides.add_slide(blank_layout) creates a new slide with the given layout. The layout determines which placeholder shapes are pre-created — since we use the blank layout (0 placeholders), no unwanted shapes appear.
• The for loop uses enumerate(slide_specs, start=1) for 1-based page numbering. This matches the page numbers shown in the header band and in PowerPoint's slide navigator.
• The method does NOT verify that the rendered slides match the input specs (no post-render validation). The Presentation is returned as-is. Post-render verification (e.g., re-opening the saved file and checking slide count) is left to the test suite.
• Memory: each slide adds ~10–100KB to the Presentation object's internal XML tree. A 20-slide deck uses ~2–5MB of memory. This is well within typical process limits.

DeckBuilder.build(self) -> PresentationBuildResult — Full pipeline orchestration: load → validate → plan → render

Purpose & contract:
Execute the complete four-stage pipeline and return a PresentationBuildResult containing the Presentation, the planned SlideSpec list, and the output file path. This is the primary programmatic API for batch scripts, tests, and the save() method. MUST execute stages in strict order with no stage skipped or reordered.

Signature:
def build(self) -> PresentationBuildResult

Pipeline stages (strict order, no skipping):
1. slide_specs = self.load_slide_specs() — read and normalize content file → list[SlideSpec]
2. slide_specs = self.validate_slide_specs(slide_specs) — structural and quality gate → same list[SlideSpec] or raise
3. planned_specs = self.planner.plan(slide_specs) — enrich with metadata → new list[SlideSpec] with meta populated
4. prs = self.render_presentation(planned_specs) — construct PPTX and draw slides → Presentation
5. Optional: if EXPECTED_SLIDE_COUNT is set and len(planned_specs) != EXPECTED_SLIDE_COUNT → logger.warning("Expected %s slides, but built %s.", EXPECTED_SLIDE_COUNT, len(planned_specs))
6. return PresentationBuildResult(prs, list(planned_specs), self.output_file)

Primary invariants (must/shall):
1. Stage order MUST be load → validate → plan → render. No stage may be skipped or reordered. Validation MUST occur before planning (planning assumes valid specs). Planning MUST occur before rendering (rendering uses planned metadata for adaptive behavior).
2. planner.plan() enriches each SlideSpec with computed metadata via dataclasses.replace(spec, meta=enriched_meta). The planned specs are NEW objects — the original loaded specs are not mutated. The following metadata fields are set (via meta.setdefault, so explicit values win):
• sequence_index (1-based position)
• sequence_total (total count)
• is_first / is_last (boolean position flags)
• slide_type_name (string enum value)
• section_name (section or section_label)
• content_density ("low" / "medium" / "high")
• slide_intent (e.g., "decision-first", "data-first")
• presentation_state (e.g., "opening", "synthesis")
• decision_mode (e.g., "binary", "comparative", "n/a")
• evidence_mode (e.g., "structured", "qualitative")
• composition_mode (e.g., "balanced", "dense", "hybrid")
3. PresentationBuildResult.slide_specs MUST contain the planned specs (with metadata), not the raw loaded specs. This allows downstream consumers to inspect the enriched metadata.
4. If EXPECTED_SLIDE_COUNT is not None and the count mismatches, a WARNING is logged but the build proceeds — this is a soft check, not a gate. The if EXPECTED_SLIDE_COUNT and ... guard means EXPECTED_SLIDE_COUNT = 0 or None disables the check.
5. build() MUST NOT write to disk. The Presentation is returned in-memory inside the result object.
6. The PresentationBuildResult is a @dataclass(slots=True) with three fields: presentation, slide_specs, output_file.

Failure modes & recovery:
• DeckValidationError from load_slide_specs() → terminal, propagates to caller. No Presentation is constructed.
• DeckValidationError from validate_slide_specs() → terminal, propagates to caller. No Presentation is constructed.
• planner.plan() raises → propagates. Should not happen with valid specs — indicates a bug in DeckPlanner. No Presentation is constructed.
• Rendering failures within render_presentation → caught per-slide by SlideRenderer.render. The Presentation is returned with error textboxes on failed slides. build() does NOT raise for individual slide failures.
• If the content file is modified between load_slide_specs() and render_presentation() (race condition in a long-running process), the loaded specs reflect the file at load time — the file is not re-read. This is correct behavior: the pipeline operates on a snapshot.

Implementation notes:
• The EXPECTED_SLIDE_COUNT check is placed AFTER planning and BEFORE the return, not after rendering. This means the warning is emitted even if rendering fails for some slides. The count check is about the content file, not the rendering output.
• list(planned_specs) creates a shallow copy for the result object, ensuring the caller cannot mutate the internal list by modifying result.slide_specs. The SlideSpec objects themselves are immutable (frozen dataclass), so deep copy is unnecessary.
• The method does NOT log a success message — that is main()'s responsibility after save() completes. build() is a pure pipeline step.
• For batch processing, build() can be called multiple times on the same DeckBuilder instance if the content file changes between calls (hot-reload). Each call constructs a fresh Presentation.

Cross-module interactions:
• Calls self.load_slide_specs() → DeckSourceLoader.load in deck.py.
• Calls self.validate_slide_specs() → DeckValidator.validate in deck.py.
• Calls self.planner.plan() → DeckPlanner.plan in deck.py.
• Calls self.render_presentation() → SlideRenderer.render in render.py.
• Reads EXPECTED_SLIDE_COUNT from config.py.
• Constructs PresentationBuildResult (defined in deck.py).

Tests:
1. Valid content file with 5 slides → PresentationBuildResult with len(result.slide_specs) == 5, len(result.presentation.slides) == 5.
2. EXPECTED_SLIDE_COUNT = 5, actual 5 → no warning.
3. EXPECTED_SLIDE_COUNT = 5, actual 3 → warning logged: "Expected 5 slides, but built 3." Build succeeds.
4. EXPECTED_SLIDE_COUNT = None → no count check, no warning regardless of actual count.
5. EXPECTED_SLIDE_COUNT = 0 → check disabled (falsy guard), no warning.
6. Invalid content → DeckValidationError propagates, no Presentation constructed.
7. result.slide_specs[i].meta["sequence_index"] == i + 1 for all i.
8. result.slide_specs[i].meta["sequence_total"] == len(result.slide_specs) for all i.
9. result.slide_specs[0].meta["is_first"] == True, result.slide_specs[-1].meta["is_last"] == True.
10. result.output_file == self.output_file.
11. result.presentation.slide_width == SLIDE_W.
12. No file written to disk after build() (verify via tmp_path).

DeckBuilder.save(self) -> Path — Build + write to disk

Purpose & contract:
Execute build(), create the output directory if needed, and write the Presentation to self.output_file via python-pptx's Presentation.save(). Returns the resolved output Path. This is the only method in the entire codebase that writes a .pptx file to disk.

Signature:
def save(self) -> Path

Operations performed (ordered):
1. result = self.build() — execute full pipeline
2. logger.info("Saving presentation to %s", result.output_file) — log the target path
3. result.output_file.parent.mkdir(parents=True, exist_ok=True) — create parent directories
4. result.presentation.save(str(result.output_file)) — write .pptx to disk
5. return result.output_file — return the path

Primary invariants (must/shall):
1. MUST call build() first — never render directly. This ensures the full pipeline (load → validate → plan → render) is executed before any file is written.
2. MUST create parent directories before saving: result.output_file.parent.mkdir(parents=True, exist_ok=True). This allows output paths like "output/decks/2026/Q3/report.pptx" where intermediate directories do not yet exist. The call is idempotent — existing directories are not modified.
3. MUST log "Saving presentation to {path}" at INFO level before writing. This is the third log line in a successful run (after "Building slide specifications..." and any warnings).
4. The write is performed by python-pptx's Presentation.save(str(path)). This is NOT atomic — there is no temp-file-then-rename pattern. If the process is killed mid-write (SIGKILL, power loss, disk full), the output file may be corrupted (truncated ZIP archive). The operator must delete the corrupted file and re-run.
5. MUST return the Path object, not a string. Callers use the return value for logging and verification.
6. MUST NOT verify the written file (no checksum, no re-open validation, no size check). This is a potential enhancement for regulated environments but is not currently implemented.
7. MUST NOT clean up partial files on failure. If save() raises mid-write, the partial file remains on disk. This is a known limitation.

Failure modes & recovery:
• DeckValidationError from build() → build failed, no file written, no directory created. Propagates to caller.
• PermissionError → output file is locked (open in PowerPoint on Windows) or directory is read-only. Most common failure on Windows. The error message from Python is: "[Errno 13] Permission denied: 'path\\to\\file.pptx'". Operator action: close the file in PowerPoint and re-run.
• IsADirectoryError → output path is a directory (e.g., output_file = "."). Operator action: specify a file path, not a directory.
• FileNotFoundError → parent directory does not exist AND mkdir failed (e.g., invalid drive letter on Windows). Operator action: verify the output path.
• OSError → disk full, path too long (> 260 chars on Windows without long path support), or I/O error. Operator action: free disk space, shorten path, enable long path support.
• All propagate to the caller. No retry logic — the operator must resolve the filesystem issue and re-run.

Implementation notes:
• python-pptx's Presentation.save() accepts a string path or a file-like object. We pass str(result.output_file) for compatibility with all python-pptx versions. Passing a Path object directly works in recent versions but str() is safer.
• The .pptx format is a ZIP archive containing XML files, media, and relationships. python-pptx uses Python's zipfile module internally. The write is a single zipfile.ZipFile open-write-close cycle.
• mkdir(parents=True, exist_ok=True) is safe for concurrent invocations — if two processes create the same directory simultaneously, exist_ok=True prevents FileExistsError. However, concurrent writes to the SAME output file are NOT safe and will produce corrupted output.
• The method does NOT open the output file for verification after writing. For regulated environments, a post-write verification step could be added: Presentation(str(output_file)) to verify the file is a valid .pptx (re-open and parse). This is not currently implemented.
• File size: a 20-slide deck produces a .pptx file of approximately 50–200KB depending on shape complexity. No media files (images) are embedded by the current rendering engine.

Cross-module interactions:
• Calls self.build() → full pipeline (load → validate → plan → render).
• Calls Presentation.save(str(path)) from python-pptx.
• Calls Path.mkdir(parents=True, exist_ok=True) from stdlib.
• Uses logger for the save log line.
• Returns result.output_file (set in __init__, passed through PresentationBuildResult).

Tests:
1. Valid build → file exists at output_file after call. Verify: output_file.exists() == True.
2. Output path with nonexistent parent dirs (tmp_path / "a" / "b" / "c" / "out.pptx") → dirs created, file written.
3. Read-only directory → PermissionError propagates. No file written.
4. Return value is a Path instance: isinstance(result, Path).
5. INFO log line emitted with correct path: "Saving presentation to" in caplog.text.
6. File is a valid .pptx: re-open with Presentation(str(output_file)), verify len(prs.slides) == expected_count.
7. File size > 0 bytes.
8. Output file is overwritten on second run (no append, no error): run save() twice → file exists, valid, same slide count.
9. build() failure → no file written, no directory created: verify not output_file.exists() after DeckValidationError.
10. Output path with .pdf extension → file written with .pdf name but .pptx format (no extension validation).

main(argv: Optional[Sequence[str]] = None) -> None — Top-level CLI entry point with structured error handling

Purpose & contract:
Parse arguments, invoke create_presentation(), and handle all top-level exceptions with appropriate exit codes and log messages. This is the only function that calls sys.exit (via raise SystemExit). MUST distinguish between three failure categories with different severity levels and exit codes.

Signature:
def main(argv: Optional[Sequence[str]] = None) -> None

Error handling tiers (ordered by specificity):
1. DeckValidationError → logger.error("Validation failed: %s", exc) → raise SystemExit(1) from exc
• Exit code 1 (general error)
• Logged at ERROR level
• from exc preserves the exception chain for debuggers
• The full multi-line validation message is included in the log
2. KeyboardInterrupt → logger.warning("Process interrupted by user.") → raise SystemExit(130)
• Exit code 130 (standard SIGINT: 128 + signal 2)
• Logged at WARNING level — NOT ERROR — because user interruption is not a system failure
• No from exc — no meaningful exception chain for a signal
3. Exception (any other) → logger.exception("Failed to generate presentation") → raise SystemExit(1)
• Exit code 1 (general error)
• logger.exception logs at ERROR level WITH full traceback
• Catches OSError, PermissionError, RenderingError, TypeError, and any other unexpected exception
• Does NOT catch BaseException (which would include SystemExit and KeyboardInterrupt if not already caught)

Success path:
• output = create_presentation(args.output, args.content)
• logger.info("Presentation created successfully: %s", output)
• Return normally (implicit None)

Primary invariants (must/shall):
1. DeckValidationError MUST be caught FIRST (before Exception) — it is a subclass of ValueError which is a subclass of Exception, so order matters. If except Exception came first, DeckValidationError would be caught by the generic handler and logged with a full traceback instead of the clean validation message.
2. KeyboardInterrupt MUST be caught SECOND (before Exception) — it is a subclass of BaseException, NOT Exception, so it would NOT be caught by except Exception. However, explicit catching ensures the clean warning message and correct exit code 130.
3. The except Exception block MUST use logger.exception(...) (not logger.error(...)) to include the full traceback. This is critical for diagnosing unexpected failures.
4. The success log MUST include the output path: "Presentation created successfully: {path}".
5. MUST NOT catch SystemExit itself — the raise SystemExit calls must propagate to the Python interpreter, which sets the process exit code.
6. MUST NOT perform any cleanup, file deletion, or state persistence — the function is a thin error-handling wrapper. Partial .pptx files from failed saves are NOT deleted.
7. MUST NOT retry on failure — the operator must diagnose and re-run manually.
8. MUST NOT call sys.exit() directly — uses raise SystemExit(code) instead. This is functionally equivalent but allows finally blocks and context managers to execute cleanly.

Failure modes & recovery:
• All failure modes are terminal — the process exits. There is no retry, no fallback output, no partial-file cleanup.
• If the process is killed between Presentation.save() starting and completing (SIGKILL, power loss), a corrupted .pptx may remain on disk. The operator must delete it manually. This is a known limitation of non-atomic writes.
• If create_presentation raises an exception NOT in the three tiers (e.g., MemoryError, RecursionError), it is caught by except Exception (both inherit from Exception). SystemExit and KeyboardInterrupt are NOT caught by except Exception — SystemExit propagates correctly, and KeyboardInterrupt is caught by its own handler.

Implementation notes:
• The from exc syntax on raise SystemExit(1) from exc sets SystemExit.__cause__ = exc, enabling debuggers and error tracking tools (Sentry, Rollbar) to see the root cause. Without from exc, the original DeckValidationError would be lost.
• Exit code conventions:
• 0 = success
• 1 = general error (validation failure, unexpected exception)
• 130 = SIGINT (Ctrl+C)
• 2 = argparse usage error (handled by argparse internally, not by main)
• The function does NOT flush logging handlers before exiting. Python's logging module flushes handlers at interpreter shutdown via atexit. If the process is killed with SIGKILL, log messages may be lost. For critical environments, add logging.shutdown() before raise SystemExit.
• The function does NOT set up signal handlers. KeyboardInterrupt is raised by Python's default SIGINT handler. If the process is running in a context where SIGINT is ignored (e.g., some CI runners), KeyboardInterrupt will not be raised and the process must be killed with SIGTERM or SIGKILL.

Cross-module interactions:
• Calls parse_args(argv) → own function.
• Calls create_presentation(args.output, args.content) → own function → DeckBuilder.save() → full pipeline.
• Uses logger for all log output.
• Raises SystemExit (stdlib) for process termination.
• Catches DeckValidationError from deck.py.

Thread safety:
• DeckBuilder is NOT thread-safe — it holds mutable path state (self.output_file, self.content_file) and the pipeline is sequential. Concurrent deck generation requires separate DeckBuilder instances.
• The SlideRenderer and strategy instances are stateless and theoretically thread-safe, but python-pptx's Presentation and Slide objects are NOT thread-safe — concurrent rendering to the same presentation is not supported.
• create_presentation() is safe for concurrent calls with different arguments (separate DeckBuilder instances, separate Presentation objects, separate output files). Concurrent calls with the SAME output file will produce corrupted output (last writer wins, partial writes possible).

Idempotency:
• Running main() twice with the same arguments produces the same .pptx file. python-pptx serialization is deterministic for the same input: same XML structure, same ZIP entry order, same timestamps (ZIP entries use the current time, so byte-identical output requires the same timestamp — in practice, files differ by ZIP metadata timestamps but are semantically identical).
• The output file is overwritten without warning. There is no --no-overwrite flag.

Performance budget:
• For a 20-slide deck, the full pipeline completes in < 2 seconds on modern hardware (SSD, Python 3.10+, python-pptx 0.6.21+).
• Breakdown: load (~10ms) + validate (~5ms) + plan (~20ms) + render (~300ms) + save (~200ms) = ~535ms typical.
• Bottleneck is python-pptx XML serialization during save() (~200ms for 20 slides) and shape creation during render (~300ms for 20 slides with ~200 total shapes).
• No performance optimization is needed at this scale. For 100+ slide decks, consider parallel rendering (not currently supported due to python-pptx thread safety limitations).

Memory budget:
• Presentation object: ~2–5MB for 20 slides (in-memory XML tree).
• SlideSpec list: ~50KB for 20 specs (frozen dataclasses with small strings).
• Peak memory during save(): ~10MB (XML serialization buffer + ZIP compression).
• No memory leaks: all objects are garbage-collected after save() returns.

Windows-specific concerns:
• File locking: PowerPoint locks .pptx files while open. Presentation.save() raises PermissionError if the file is locked. The operator must close PowerPoint before re-running. This is the most common failure on Windows.
• Path length: Windows has a 260-character path limit by default. Long output paths raise OSError. Mitigation: enable long path support via registry (LongPathsEnabled = 1) or use short paths.
• Path separators: Path handles \ vs / automatically. str(result.output_file) produces backslash-separated paths on Windows, which python-pptx accepts.
• Console encoding: logger output with Unicode characters (e.g., … ellipsis in error messages) may fail on Windows consoles with code page 437. Mitigation: set PYTHONIOENCODING=utf-8 or use Windows Terminal.

Security considerations:
• Path traversal: output_file and content_file are user-supplied CLI arguments. A malicious user could pass ../../etc/passwd as content_file — the loader would attempt to read it and fail with a parse error (not a security breach, but information disclosure about file existence). For a CLI tool running with the user's own permissions, this is acceptable. For a web service, input paths MUST be validated against an allowlist.
• YAML deserialization: DeckSourceLoader uses yaml.safe_load (NOT yaml.load), which prevents arbitrary code execution via YAML tags. This is critical — yaml.load with untrusted input is a remote code execution vector.
• No network access: the pipeline reads local files and writes local files. No HTTP, no DNS, no sockets. The attack surface is limited to the filesystem.
• No credentials: the pipeline does not read, store, or transmit credentials, API keys, or tokens.

Cause: output_file points to a directory, not a file.

SlideCanvas — Low-level drawing primitive layer

Purpose & contract:
Wrap a python-pptx Slide object with convenience methods for adding shapes, textboxes, bullet lists, cards, rule lines, and arrow connectors. This is the lowest-level rendering abstraction — every strategy and component draws through SlideCanvas, never directly through slide.shapes. This isolation means that if the drawing library changes, only SlideCanvas needs to be rewritten.

Constructor:
def __init__(self, slide: Slide, theme: ThemeTokens, layout: LayoutTokens)
• Stores self.slide = slide, self.theme = theme, self.layout = layout, self.shapes = slide.shapes
• MUST be constructed once per slide, per render pass.

Methods (8 public + 3 static helpers):

set_background(color: RGBColor) -> None
• Sets the slide background to a solid color via slide.background.fill.solid() + slide.background.fill.fore_color.rgb = color.
• MUST be called before any shapes are added (background is rendered behind all shapes in Z-order).

add_shape(shape_type, left, top, width, height, fill_color=None, line_color=None) -> Any
• Adds a python-pptx auto-shape via self.shapes.add_shape(shape_type, left, top, width, height).
• If fill_color is not None → shape.fill.solid() + shape.fill.fore_color.rgb = fill_color; else shape.fill.background().
• If line_color is not None → shape.line.color.rgb = line_color (wrapped in try/except); else shape.line.fill.background().
• Returns the shape object for further customization.

add_textbox(left, top, width, height, text, font_name, font_size, font_color, bold=False, italic=False, align=PP_ALIGN.LEFT, word_wrap=True, auto_size=MSO_AUTO_SIZE.TEXT_TO_FIT_SHAPE, vertical_anchor="top", line_spacing=None) -> Any
• Adds a textbox with full typographic control.
• Configures the text frame via _configure_text_frame(tf, word_wrap, auto_size): clears existing content, sets word wrap, sets auto-size mode, sets all four margins to Inches(0).
• Splits text into paragraphs via _split_paragraphs(text). Each paragraph gets alignment, spacing, and a single run with font properties.
• Vertical anchoring: if vertical_anchor in ("middle", "bottom") → _set_vertical_alignment(tf, "ctr" or "b") via XML manipulation of a:bodyPr@anchor.

add_bullet_list(left, top, width, height, bullets, font_size, font_color, spacing_after=Pt(6), bullet_prefix="• ") -> Any
• Renders a list of bullets as a single textbox with multiple paragraphs.
• BulletItem bullets get two runs in a single paragraph: Run 1 (bold title), Run 2 (regular description, 2pt smaller, min 12pt).
• Plain string bullets get a single run.
• Text truncation: titles to 120 chars, descriptions to 180 chars, plain bullets to 220 chars.
• List is clipped to MAX_BULLETS_PER_SLIDE (10) items.

add_card(left, top, width, height, fill_color, line_color, radius=True) -> Any
• Adds a rectangle or rounded rectangle shape representing a visual "card".
• radius=True → MSO_SHAPE.ROUNDED_RECTANGLE; radius=False → MSO_SHAPE.RECTANGLE.

add_rule_line(left, top, width, fill_color) -> Any
• Adds a thin horizontal rectangle (height Inches(0.015)) acting as a visual divider line.
• Used for footer rule line and timeline axis.

add_arrow_connector(start_x, start_y, end_x, end_y, color, width=Pt(2)) -> Any
• Adds a straight connector line with an arrowhead at the end (tail).
• Skips zero-length connectors with a logger.debug message and returns None.
• XML injection for arrowhead: python-pptx does not natively support arrowhead configuration. The method manually injects XML: locates/creates a:ln within p:spPr, sets a:solidFill > a:srgbClr, and creates a:tailEnd with type="triangle", w="med", len="med".
• Wrapped in try/except Exception — if XML structure is unexpected, connector is drawn without an arrowhead.
