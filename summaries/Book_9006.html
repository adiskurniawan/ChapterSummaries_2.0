Filename static/worker.js — lightweight Web Worker intended to index table HTML and run searches off-main-thread.
Primary purpose 1. Build an in-memory index of tables (headers + rows).
2. Execute text searches across indexed rows without blocking the UI.
3. Support cancellation and incremental yielding for very large tables.
Top-level constants / state - CHUNK_YIELD = 500 → number of rows processed before yielding to event loop.
- indexStore → { tables: [] }, each table entry: { id, headers: [], rows: [ [cell,..] ], rowsText: [], rowsTextLc: [] }.
- currentSearchToken → monotonically incremented token to cancel in-flight searches.
Sanitization helper sanitize(text) — normalizes whitespace, trims and converts to lower case for consistent matching.
Parsing utilities arraysEqual(a,b) — small helper used when deciding if first row equals detected headers.
HTML → tables parsing parseHtmlTables(html):
• Preferred path: uses DOMParser (if available in worker) to parse DOM and collect <thead>, <tbody>, or fallback logic (first row as header).
• Fallback path: regex-based best-effort parser when no DOMParser — extracts <table> blocks, finds <thead>/first <tr>, and pulls <td>/<th> text (strips inner tags).
• Returns array of { id, headers:[], rows:[ [...], ... ] }.
Index building addTableToIndex(id, headers, rows) → normalizes rows to arrays of strings, builds rowsText (joined row cell text) and rowsTextLc (sanitized lowercase string), then pushes the entry into indexStore.tables.
Index entry points buildIndex(msg) accepts:
• msg.tables — direct array of table objects {id, headers, rows} (fast path).
• msg.html — HTML string to parse then index.
• msg.url — fetch remote URL, parse returned HTML and index.
On success posts { type: 'status', status: 'indexed', tables: N }.
Search behavior handleSearch(msg) (async):
1. Increments currentSearchToken to allow cancellation of prior searches.
2. Reads msg.query, returns early if empty.
3. For each table: scans table.rowsTextLc and finds indexes where q occurs (substring match).
4. Builds snippets (context window up to ~160 chars) for up to perTableLimit rows.
5. Periodically yields (every CHUNK_YIELD rows) with await new Promise(r=>setTimeout(r,0)) so the worker stays responsive.
6. If token changes, posts { type: 'searchCanceled', requestId } and returns early.
7. On completion posts { type:'searchResults', query, results, requestId }.
Messages handled by the worker - 'index' → call buildIndex.
- 'search' → start handleSearch (cancellable).
- 'clear' → reset indexStore and respond {type:'status',status:'cleared'}.
- 'status' → respond {type:'status', status:'ready', tables: N}.
- 'ping' → respond {type:'pong'}.
- default → post error {type:'error', error:'unknown_message_type'}.
Responses / events posted - status (ready/indexed/cleared), searchResults (results + requestId), searchCanceled (requestId), error (various error codes), pong.
Error handling & robustness - Try/catch guards around parsing and message dispatch.
- Posts structured error messages back to main thread instead of throwing.
- Non-destructive: indexing and search do not touch DOM.
Performance considerations - Memory: index stores rowsText and rows arrays — can grow large for big tables.
- CHUNK_YIELD yields to event loop to keep worker responsive for huge tables.
- perTableLimit controls snippets returned per table (bounded).
Security & privacy notes - Worker parses HTML strings or fetches URLs — be mindful when indexing remote/untrusted pages.
- parseHtmlTables uses textContent extraction (DOMParser) or regex fallback — inlined scripts are not executed by DOMParser in a worker, but fetching remote URLs may expose request metadata. Validate URLs if needed.
Limitations & edge cases 1. Regex fallback is best-effort: fragile for complex/nested tables, tables with markup in cells, or unusual whitespace/encodings.
2. No fuzzy or tokenized ranking — search is simple substring match on joined row text.
3. Non-text content (images, complex HTML in a cell) is reduced to stripped text; important semantics may be lost.
4. Very large indexes may exhaust memory — no built-in eviction/streaming policy.
Suggested improvements (optional) - Allow configurable CHUNK_YIELD and perTableLimit via messages.
- Offer streaming index (emit progress messages) and a max-rows truncation to limit memory.
- Offer tokenized/fuzzy search, or precomputed n-grams, for better relevance.
- Use postMessage(..., [transferableArrayBuffer]) for binary transfer if indexes become large and need serialization.
Practical integration notes - Main thread should send {type:'index', html: document.documentElement.outerHTML} or pass structured tables for fastest, safest indexing.
- For responsiveness, main thread may send new search messages quickly — cancellation handled by token.
- requestId passthrough is supported so main thread can correlate results.
