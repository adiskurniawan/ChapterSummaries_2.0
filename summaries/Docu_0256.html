📦 deckgen/ Package

Core application package for deck generation and rendering.
• __init__.py – Declares deckgen as a package.
• config.py – Central configuration values and runtime settings.
• deck.py – Deck-level data structures and orchestration logic.
• render.py – Rendering pipeline for producing slide output.
🧠 deckgen/taxonomy/ Package

Defines taxonomy logic for matching, normalization, and rule-based classification.
• exceptions.py – Custom taxonomy exceptions.
• types.py – Shared taxonomy type definitions.
• normalize.py – Text normalization helpers.
• resolvers.py – Resolves taxonomy matches and references.
• registry.py – Central registry for taxonomy entries.
• matcher.py – Matching engine for taxonomy resolution.
• rules.py – Rule definitions and evaluation logic.
🔧 deckgen/taxonomy/shared/

Shared taxonomy primitives used across aliases, phrases, and matching logic.
• core.py – Core shared helpers and base definitions.
• intent.py – Intent-related shared logic.
• modifiers.py – Modifier handling utilities.
• stopwords.py – Stopword lists and filtering helpers.
• __init__.py – Package initializer.
🏷️ deckgen/taxonomy/aliases/

Alias-based taxonomy vocabulary organized by domain.
• core.py – Base alias definitions.
• narrative.py, argument.py, explanation.py, structure.py, lists.py, comparison.py, planning.py, process.py, data_analytics.py, strategy.py, product.py, technical.py, people.py, workshop.py, financial.py, risk.py, research.py, visual.py, reference.py, change.py, customer.py, innovation.py, legal.py, esg.py, partnership.py, operations.py, scenario.py, communication.py, meta.py – Domain-specific alias groups.
• __init__.py – Package initializer.
📝 deckgen/taxonomy/phrases/

Phrase-based taxonomy vocabulary organized in the same domain structure as aliases.
• narrative.py, argument.py, explanation.py, structure.py, lists.py, comparison.py, planning.py, process.py, data_analytics.py, strategy.py, product.py, technical.py, people.py, workshop.py, financial.py, risk.py, research.py, visual.py, reference.py, change.py, customer.py, innovation.py, legal.py, esg.py, partnership.py, operations.py, scenario.py, communication.py, meta.py – Phrase collections by use case.

🔑 deckgen/taxonomy/keywords/

Keyword grouping layer used to support taxonomy lookup and classification.
• groups.py – Keyword group definitions.
• shared.py – Shared keyword sets and helpers.
• __init__.py – Package initializer.
🧭 Functional Summary

build_deck.py drives the deck build process and likely acts as the main entry point.
build_deck_slides_content.yaml stores slide content configuration or source content for generation.
deckgen/ contains the core runtime package for deck creation, rendering, and taxonomy processing.
taxonomy/ provides the classification layer, including normalization, matching, registry management, domain vocabularies, and supporting shared modules.
Overall, the project is structured around converting slide content into generated deck output through a modular taxonomy-driven pipeline.

Recommended taxonomy expansions
Add a taxonomy/registry.py or catalog.py to centralize all category metadata in one place: canonical name, aliases, phrases, keywords, parent/child relationships, and priority.

Introduce subcategory layers. Current categories look broad. Add second-level groups such as strategy -> portfolio_strategy, competitive_strategy, go_to_market; research -> user_research, market_research, academic_research; operations -> planning, execution, optimization, governance; technical -> architecture, systems, api, data_platform.

Add confidence and precedence rules. When multiple taxonomy entries match, store match strength, tie-break priority, and a default fallback category.

Split by intent, not just domain. Many slide types are intent-based: explain, compare, decide, plan, persuade, summarize, diagnose, forecast, train, report. Intent usually improves slide generation more than topic alone.

Add negative or exclusion hints. Useful for overlapping groups. Example: strategy may match lots of business text, so add exclusions that prevent it from swallowing risk, financial, or legal unless those are primary.

Add richer phrase patterns. Expand each phrase file with synonyms, verb forms, noun forms, common business shorthand, and slide-specific phrases like “so what”, “next steps”, “key takeaways”, and “decision needed”.

Add cross-category bridge entries. Some content belongs in more than one bucket. Examples: customer + research, product + strategy, financial + scenario, operations + process, communication + workshop. These can be explicit bridge aliases instead of forcing a single bucket.

Create a few meta categories. Useful top-level groups: executive, analysis, planning, delivery, governance, reference, collaboration. This makes routing easier before narrowing into domain files.

Add taxonomy tests. Build a small fixture set: input phrase, expected canonical category, expected secondary matches, and disallowed categories. That will keep expansions from breaking old behavior.

Move repeated terms into shared base lists. Common words like “overview”, “summary”, “roadmap”, “framework”, “key points”, and “recommendation” probably belong in shared core files rather than duplicated in every category.

Best next expansion set: planning, decision, summary, diagnosis, forecast, governance, customer, operations, technical, research.
