My Responsibility

1. Define project goals clearly
- State the high-level purpose of the project (example: “render complex tables into styled HTML”).
- Break goals into smaller phases (example: “Phase 1: modularize rendering”, “Phase 2: optimize performance”).
- Keep the long-term vision consistent even if small tasks change.

1. Interpret and formalize goals
- Translate high-level goals into technical specifications.
- Map goals into actionable changes per file.
- Ensure all proposals preserve compatibility and future extendibility.

2. Provide current assets
- Share the latest versions of files before requesting an enhancement.
- Ensure files are complete (no truncated code).
- Organize shared snippets by filename so I can reassemble context.

2. Preserve and update assets
- Return the full updated file each time, not partial edits.
- Guarantee signatures, structure, and orchestration remain consistent with your existing project.
- Strip comments unless you request explanations.

3. Specify scope of each task
- Keep requests limited to one focused change (example: “optimize asset copy in coreassets.py”).
- Clarify boundaries (example: “do not modify corerenderer.py in this step”).

3. Deliver precise scope changes
- Modify only the targeted file(s).
- Apply exactly the requested change.
- Ensure no ripple changes spill into untouched modules unless explicitly asked.

4. Testing on your side
- Run the project after each change.
- Report console errors, traceback logs, or rendering issues.
- Provide failing input/output samples if something breaks.

4. Verification on my side
- Perform “check 10 times” before returning code: syntax, imports, edge cases, variables, test runs.
- Provide usage snippets for smoke testing.
- Point out any uncertainties that require your confirmation.

5. Provide style and policy rules
- Confirm rules (e.g., no comments, preserved function signatures).
- State coding preferences (e.g., PEP8 strict, short variable names).
- Clarify if third-party libraries are allowed.

5. Enforce style and policies
- Follow your constraints exactly.
- Default to minimal Python standard library unless you approve an external dependency.
- Maintain project conventions (folder layout, imports, naming).

6. Memory control
- Tell me explicitly when to remember a workflow, directory, or preference long-term.
- Request forget operations when rules or context change.
- Keep memory synced with project evolution.

6. Memory maintenance
- Store or delete memory items as instructed.
- Keep Prompt_03 as the baseline reference.
- Update memory when you finalize structural changes to the project tree.

7. Task approval
- Approve or reject each file I deliver.
- Provide feedback if changes are incomplete or incorrect.
- Request tweaks before final approval if needed.

7. Apply feedback
- Revise files quickly based on your corrections.
- Keep iterations short, precise, and cumulative.
- Avoid overwriting your assets during revision.

8. Repository management
- Decide when to merge changes into your Git repo or Dropbox folder.
- Create commits with clear messages (example: “core_assets.py: add atomic copy fallback”).
- Handle branching and release tags.

8. Provide commit-ready output
- Deliver changes in copy-pasteable format.
- Summarize diffs where helpful.
- Warn if a change introduces backward incompatibility.

9. Decide enhancements
- Choose priority (performance, readability, modularity, UX).
- Break big features into steps.
- Provide roadmap checkpoints.

9. Propose safe enhancements
- Suggest minimal, non-breaking changes for each step.
- Highlight trade-offs (speed vs. memory, clarity vs. brevity).
- Keep options modular for later rollback.

10. Communication
- Use clear and concise requests.
- Mark multi-file tasks with explicit filenames.
- Confirm when we move from one step to the next.

10. Communication
- Answer only in your chosen language.
- Keep replies short, factual, no fluff.
- Present everything in tables, lists, or code blocks for clarity.
