1. What this query is doing at the highest level
This Power Query is a full ingestion and transformation framework for a folder-based data pipeline. It is not just reading files. It is discovering files, filtering them by age and type, opening either text files or Excel workbooks, selecting the best worksheet or table, normalizing headers, coercing data types, harmonizing alternate column names, calculating business measures, joining optional dimension tables, assigning data quality scores, detecting duplicates, optionally keeping only the latest record per transaction key, and then returning a final reporting table.

In practical terms, the query behaves like a compact ETL system written in M. It is designed for a real operational environment where source files are inconsistent, naming conventions drift, dimensions may or may not exist, and the final output still needs to be trustworthy enough for reporting.
2. Why this is much more than a standard Power Query
A standard query usually does one or two things: load a table and maybe reshape a few columns. This one is architected like an ingestion platform. It has configuration, helper functions, workbook support, binary file parsing, schema repair, lookup enrichment, quality governance, duplicate handling, and final presentation. That is a fundamentally different design pattern.

The query is written as a layered pipeline. Each layer solves a different class of problem. That separation is one of the strongest signs that this is intended for repeated operational use rather than a one-off refresh.
3. The Config record is the control panel for the whole pipeline
The Config block defines the operational behavior of the query. It tells the pipeline where to read files from, which file extensions are acceptable, how to parse CSV files, what culture to use for number and date conversion, which worksheet names are preferred when opening Excel files, how far back to look for files, whether to keep only the latest row per key, whether to ignore archive folders, and whether to return a debug-style result that includes error rows.

This design is important because it separates environment settings from transformation logic. That makes the query easier to maintain, easier to reuse, and easier to adapt when the source folder, input format, or quality policy changes.
4. RefreshUtc and MinModifiedDate create a refresh-time boundary
RefreshUtc captures the exact moment the query starts. MinModifiedDate then computes a date threshold by subtracting the configured number of days from that refresh time. That means file discovery is not arbitrary; it is constrained to a rolling time window.

This is a practical incremental-ingestion technique. It reduces unnecessary file processing and lets the query behave like a rolling refresh rather than a full historical re-scan every time. In a large folder, that can make a significant performance difference.
5. fnTry is a reusable safety wrapper
This helper wraps a function call in a try record and returns either the result or a default value. It gives the rest of the query a simple way to tolerate operations that may fail without repeating the same error-handling structure everywhere.

Even when the query does not rely on this helper heavily in the visible flow, it signals a design style: errors should be handled deliberately and locally rather than allowed to break the whole pipeline when a fallback is acceptable.
6. fnCleanText is the base text sanitation layer
This helper converts any input to text if possible, trims whitespace, removes control characters, and converts empty strings to null. That is a very important early normalization step because text values in workbook and CSV environments are often messy.

Without this kind of cleaning, later comparisons and joins can fail for trivial reasons such as hidden spaces, line breaks, or blank-but-not-null values. The function helps stabilize every downstream text-based operation.
7. fnNormalizeName turns human-formatted names into machine-safe identifiers
This function lowercases the text, trims it, removes line breaks, replaces spaces and punctuation with underscores, strips brackets and quotes, collapses repeated underscores, and trims leading or trailing underscores. The result is a canonicalized identifier that can be compared reliably across sources even when the original formatting is inconsistent.

That is especially important for column names and worksheet names. In real files, those names rarely remain perfectly uniform. Normalization gives the query a stable basis for comparison and lookup.
8. fnNormalizeHeader is a semantic wrapper around general name normalization
The query defines a separate function for headers even though it reuses the general name normalizer. That may look redundant at first, but it improves readability. When a column header is being normalized, the code reads more clearly if the helper being used is named as a header function rather than a generic text cleaner.

This is a small but good design choice. It makes the intent of the code easier to follow in a long pipeline.
9. fnToNumber is built for messy real-world numeric values
This helper does not assume that a value arrives in a clean numeric state. It first tries direct numeric conversion. If that fails, it tries parsing the cleaned text under the configured culture. If that still fails, it tries alternate cleanup paths that remove thousands separators or swap decimal formatting.

This matters because different source systems and different locales format numbers differently. A single value may appear numeric to a human but still require careful parsing. This function is designed to salvage valid numeric intent from inconsistent input.
10. Why the numeric parser uses multiple conversion strategies
Excel exports, CSV extracts, and manual data entry do not always agree on decimal symbols or grouping separators. One file may use 1,234.56, another 1.234,56, and another may use a plain text string with formatting noise. The helper tries multiple reasonable interpretations before giving up.

This is the kind of flexibility needed in ingestion pipelines where the priority is to recover usable data whenever possible instead of failing on formatting alone.
11. fnToInt standardizes integer-like values
This helper converts a value to a numeric form, rounds down, and then casts it to a 64-bit integer. It is useful for values that should be whole numbers, such as line numbers, counts, or identifiers that may accidentally arrive as decimals.

That kind of coercion is important because downstream logic often assumes integer semantics. If such values remain floating point, later operations can behave unexpectedly or become harder to audit.
12. fnToLogical interprets common truthy and falsy text values
Not every source stores booleans as actual logical values. Some use yes/no, some use 1/0, some use x, and some use variants of true and false. This helper recognizes common truthy and falsy patterns and returns null when the input cannot be confidently interpreted.

That is a practical normalization strategy for source systems that are only loosely typed.
13. fnToDate, fnToDateTime, and fnToDateTimeZone are multi-culture parsers
These helpers first try direct type conversion. If that fails, they attempt to parse text under several cultures. This is useful because date formatting conventions vary by region and by export source. A string that fails in one culture may parse correctly in another.

The query is clearly designed for an environment where input data is not guaranteed to use a single date format. Supporting multiple parsing paths increases robustness substantially.
14. fnToCurrency is specialized for monetary strings
This helper strips a value down to digits, separators, signs, and parentheses, then interprets parentheses as a negative number if needed. That makes it suitable for currency-like fields that may include formatting symbols or accounting-style negatives.

In finance and sales files, monetary values are often formatted for human reading rather than machine parsing. This function closes that gap.
15. fnAddIfMissing is one of the core schema-stability tools
If a column does not exist, the helper adds it as a null column. That allows the rest of the pipeline to assume the schema exists even when a source file is missing certain fields. This is a key technique for handling schema drift safely.

Without this kind of helper, the query would be much more fragile whenever source fields vary from one file to another.
16. fnEnsureColumns establishes a canonical column contract
This helper first ensures that a specified list of columns exists, then reorders the table so that those columns appear in a stable order. That is useful both when standardizing source data and when shaping final output. It turns a variable schema into a predictable one.

This is one of the most important primitives in the query because so many later steps depend on known column names and stable column order.
17. fnDistinctNonNull is a small but useful hygiene helper
This utility removes nulls and deduplicates a list. It is a simple function, but it fits the broader style of the query: lists and metadata should be cleaned before they are used in important decisions. Even if it is not central in the main visible flow, it supports safer list-based operations in a reusable template.
18. fnContainsAnyText is a general-purpose text pattern checker
This function tests whether a text value contains any pattern from a list. It lowercases the input and each pattern for reliable matching. That makes it useful for heuristic classification, naming rules, or conditional routing based on file names, column names, or text fields.

It is one of the helper functions that makes this template extensible.
19. fnGetWorkbookTable safely reads a named table from the workbook
This helper attempts to retrieve a workbook table by name. If the table exists, it returns the content. If not, it returns an empty table, optionally with fallback columns. That means optional dimension tables can be used without becoming mandatory dependencies that break the whole query.

This is a very practical design decision for workbook-based pipelines, where supporting tables may be prepared later or might not exist in every version of the file.
20. fnStandardizeTable normalizes both headers and text values
This helper transforms column names into a canonical normalized form and then trims and cleans every text column. It is the generic cleanup step that can be applied to incoming files or lookup tables alike.

Its job is to make the shape and text content of a table more predictable before deeper business logic begins. That predictability is crucial in a pipeline with many later joins and comparisons.
21. The alias mapping is the semantic bridge between source variance and internal consistency
fnApplyAliases maps many possible incoming source names onto a smaller canonical vocabulary. Examples include multiple document number aliases, multiple date aliases, different customer key names, different product key names, and several amount-related naming variations.

This is one of the most important parts of the design. Without aliasing, every variation in source naming would require custom downstream handling. With aliasing, the rest of the pipeline can work against a single internal language.
22. The alias layer reduces friction caused by source naming inconsistency
Real files rarely use the same naming style forever. One system may export doc_no, another may use invoice_no, and another may call the same field voucher_no. The query normalizes all of those into document_no, which removes a major source of breakage.

That same principle applies to dates, customer codes, product codes, quantity fields, pricing, discounts, tax, and net amounts. It is a very strong foundation for schema drift handling.
23. fnPickBestItem selects the best worksheet or table inside Excel files
For binary Excel files, the query looks at visible items only, normalizes their names, prefers sheet names from the configured list, and otherwise falls back to the first visible item. This is a smart compromise between automation and predictability.

It helps the pipeline find the most likely data sheet even if the workbook contains multiple sheets or tables. That makes the query more resilient to workbook structure differences.
24. Why the preferred item search is normalized first
Sheet and table names often vary in case, punctuation, and spacing. Normalizing them before matching makes the selection much more forgiving. A sheet named Raw Data, for example, can still align with a preferred raw pattern after normalization.

This makes the workbook parser more robust in the face of human-authored file names.
25. fnLoadBinary is the file-format adapter
This function handles two major branches. For text-based files such as CSV or TXT, it uses Csv.Document, attempts to promote headers, and normalizes the column names. For Excel files, it opens the workbook, picks the best sheet or table, and normalizes the result. That means the query can ingest different file types through a common interface.

This is the core parser for the pipeline. It transforms a binary file into a structured table ready for standardization.
26. Why the CSV encoding is explicitly set to UTF-8
The configuration uses a specific encoding value for CSV parsing. That is a practical safeguard because source text may contain non-ASCII characters or multilingual content. Explicit encoding helps prevent garbled text and ambiguous parsing.

In file ingestion, encoding issues are a common source of subtle corruption, so setting it deliberately is a strong design choice.
27. fnStandardizeTable is applied after parsing but before metadata enrichment
The query first turns the file into a table, then standardizes the table, and only afterward adds metadata such as source file name and modified time. That is the right order because the business data should be cleaned before the lineage fields are appended.

This keeps the pipeline logically separate: parse, clean, then annotate.
28. The source metadata fields make the pipeline auditable
Every row receives source file name, file extension, folder path, full path, file size, file modified timestamp, ingest timestamp, load status, and load error. These metadata fields are critical because they allow the final data to be traced back to the exact file and refresh instance that produced it.

That lineage is important for troubleshooting, reconciliation, and governance.
29. source_full_path is especially valuable when folder structure matters
File name alone can be ambiguous in a large shared folder environment. Including the folder path makes it possible to identify the exact source location. That matters when archive folders, staging folders, and live folders all exist in the same overall structure.

1. What this query is doing in operational terms
This Power Query is a finance-grade ingestion and conformance pipeline for D365FO GeneralJournalAccountEntries. It does not merely import journal lines. It actively standardizes the source, filters the dataset to the intended legal entities and time window, derives accounting fields, reconstructs voucher-level control totals, scores row quality, enriches the rows with optional lookup dimensions, and then routes the result into one of three outputs: a clean fact set, a quarantine set, or the full set.

Operationally, the query behaves like a governed ETL process inside Excel or Power BI. It is designed to be refreshable, inspectable, and safe for reporting use.
2. The query is a controlled data pipeline, not a passive extract
The presence of configuration variables, reusable helper functions, dimension lookups, voucher control totals, row scoring, and final output routing shows that this is a full transformation layer. The query does not simply expose whatever the source system returns. It interprets the source through business rules and data quality rules before it produces output.

This is the key difference between a raw extract and a reporting pipeline. A raw extract is mechanical. This query is selective, opinionated, and built for downstream consumption.
3. Why the CONFIG section matters so much
The query starts with a deliberate configuration block. That is a strong design choice because it separates business intent from transformation logic. The source URL, entity name, company list, date range, allowed posting layers, allowed posting types, output mode, deduplication behavior, quality threshold, default currency, dimension delimiter, culture, and refresh timestamps are all clearly visible in one place.

That means the query can be adapted without rewriting the logic body. It also means a reviewer can understand the operating assumptions before reading the more technical sections.
4. BaseUrl and EntityName define the source contract
BaseUrl points to the Dynamics 365 Finance and Operations OData endpoint, and EntityName identifies the table-like entity being loaded. In this case, the entity is GeneralJournalAccountEntries. That tells you the query is pulling journal-entry-level accounting data rather than summary data or a prebuilt report.

This is important because journal line data is highly detailed and often requires careful quality checks before it is safe to use in reporting.
5. CompanyList is a scope control, not a convenience list
The company list limits the dataset to USMF, DEMF, and GBRT. That is a hard business boundary. It prevents the pipeline from pulling unrelated legal entities into the output. In a multi-company D365 environment, this is essential because journal data can otherwise become cross-company very quickly.

By filtering to known companies at the query level, the pipeline reduces noise, controls volume, and avoids accidental reporting leakage.
6. StartDate and EndDate create the reporting window
The query only accepts transactions dated between 2024-01-01 and 2026-12-31. That date window acts as a business filter and a performance filter at the same time. It makes the query predictable because the output always reflects the configured reporting span rather than the entire source history.

This is the correct approach for recurring finance reporting, especially when the source entity may contain a long and expanding transaction history.
7. AllowedPostingLayers controls which accounting layers survive
The query is not content to take every posting layer from source. It only accepts Current, Operations, and Tax, plus nulls. That means the author is actively limiting the ledger context to what is considered relevant for reporting.

Posting layers in D365 can be analytically important, so the filter is a major governance decision. It ensures the report does not mix in layers that would confuse the financial interpretation.
8. AllowedPostingTypes filters journal semantics
The query allows only Normal, Opening, Closing, Adjustment, and Reversal. Those are business-relevant accounting categories. The filter excludes everything else, which reduces the chance that exotic or unexpected posting types enter the fact output unnoticed.

This is an accounting-control choice, not a technical one. It says the pipeline is meant to report on a defined journal universe.
9. OutputMode gives the query three consumption behaviors
Fact, All, and Quarantine let the same query support three different use cases. In fact mode, only passing rows are returned. In quarantine mode, only suspicious rows are returned. In all mode, every row that survives the conformance pipeline is shown. That is a very efficient design because one query can support reporting, quality review, and troubleshooting.

It avoids maintaining separate extraction paths for good data and bad data.
10. KeepLatestByVoucher defines survivorship behavior
This flag controls whether the query collapses duplicate-like records by keeping the latest row for each transaction key. That is a very common enterprise data practice. Operational systems often emit revisions, reposts, retries, or duplicate-delivered rows. The query needs a rule for deciding which row is the surviving version.

Keeping the latest by voucher is a practical approximation of “most current truth” for reporting.
11. MinDQScore creates a reportability threshold
The data quality score is not merely decorative. The query uses the minimum score to decide whether a row is allowed into the fact output. That means the score has real operational consequences. Rows below the threshold are treated as needing quarantine rather than reporting approval.

This makes the pipeline measurable. It does not just say good or bad. It gives a score and then uses that score as a business gate.
12. DefaultCurrency is a fallback for incomplete data
Currency code can be missing or inconsistent in source data. Rather than allow that to break later reporting logic, the query assigns a default currency of USD where necessary. This keeps the output structurally complete and makes downstream modeling easier.

That said, defaulting a currency is a modeling decision and should be understood as a fallback rather than a claim about actual transaction currency.
13. DimensionDelimiter governs how composite dimensions are split
Many D365 dimension strings are compound values. The query uses a delimiter to split them into usable pieces. The configured delimiter is |, but the splitting function also normalizes alternative separators such as ~, -, /, and >. That means the query is prepared for messy display strings that do not always follow one exact convention.

This is valuable because ledger dimension strings are often not perfectly standardized across systems or exports.
14. DecimalCulture makes parsing locale-aware
The query uses en-US as the preferred parsing culture. This matters because finance data often arrives as text. A string like 1,234.56 parses differently from 1.234,56 depending on locale. By building a culture-aware parser, the query improves its chances of reading values correctly even when the source is inconsistent.

This is one of the most important robustness choices in the query.

The query captures the exact refresh time and derives a date-only version. These values later become audit fields in the output. That means each row not only shows accounting data but also carries the ingestion context that brought it into the model.

This is useful for lineage, troubleshooting, and refresh-cycle analysis. It lets analysts see when the dataset was produced.

The pipeline’s quality depends on its helper functions. This function converts any value to text, trims it, cleans control characters, and collapses blank strings to null. That seems small, but it is foundational. Later comparisons, merges, and alias lookups become much more reliable because stray whitespace and hidden control characters are removed early.

It also reduces the risk of false nonmatches and meaningless empty strings surviving in the pipeline.

The numeric parser first tries direct conversion. If that fails, it tries a culture-aware text parse. If that still fails, it attempts fallback transformations that remove separators or isolate numeric characters. This is not a minimalist converter. It is a rescue function for messy finance input.

That is appropriate here because D365-derived or Excel-transferred data can arrive in mixed formats. A single rigid parser would be too brittle.
19. fnInt ensures integer output where needed
The integer helper takes the parsed number, rounds it down, and converts it to a 64-bit integer. That is useful for fields like year, month, counts, and other whole-number contexts. It makes the model more explicit and reduces type ambiguity later in the query.

Using a dedicated integer helper also keeps intent clear when reading the transformation steps.
20. fnDate tries multiple paths to recover a date
Dates are one of the most common sources of parsing failure. This function first tries direct date conversion, then tries text parsing under several cultures. That makes it resilient when source values come in as localized strings or mixed types.

The same defensive idea is repeated across the query: trust the source as little as possible until it has been normalized.
21. fnDateTime does the same thing for datetime values
Datetime values can fail just as easily as dates. The function tries direct datetime conversion and several text parsing cultures before falling back to null. That makes timestamp fields like created_datetime and modified_datetime safer to use for sorting, deduplication, and auditing.

Without this, the deduplication logic would be much less reliable.
22. fnBool recognizes common true/false text patterns
This helper interprets strings like true, yes, 1, and on as true, and strings like false, no, 0, and off as false. That is useful in enterprise data because boolean-like fields often arrive as text rather than true logical values.

Even if the current pipeline does not lean heavily on booleans, the presence of this helper shows that the query was built with broader reuse in mind.
23. fnRoundCurrency normalizes money values to two decimals
Currency amounts are often best rounded to standard monetary precision. This helper handles that. It reduces noisy decimals and helps stabilize reporting values when amounts are later used in totals, comparisons, or exports.

It also helps prevent floating-point residue from becoming visible in the final dataset.
24. fnNormalizeName is one of the most important structural helpers
This function lowercases column names, trims them, removes control characters, replaces spaces and punctuation with underscores, removes brackets and quotes, compresses repeated underscores, and trims edge underscores. The result is a predictable canonical column naming pattern that is much easier to map and reference.

Without this step, the query would be much more vulnerable to source naming drift and formatting variation.
25. Why column-name normalization matters in D365 OData
D365 OData entities can expose long, technical, or inconsistent field names. Normalizing them upfront means the rest of the pipeline can work with cleaner names rather than the source’s raw naming style. That reduces the complexity of aliasing and improves maintainability.

In a finance query, that is not a cosmetic improvement. It is a structural one.

This helper adds missing columns as null if they are not already present, then reorders the table to match an expected column sequence. That is extremely useful in conformance pipelines because it lets the query tolerate source variation without breaking the downstream logic.

If the source omits a field that is expected later, the query can still proceed with a placeholder null column.
27. Why adding missing columns as null is powerful
This approach preserves pipeline continuity. Instead of crashing because a field is absent, the query creates the field and lets later steps deal with the null. That is often the right choice in reporting pipelines where some fields are optional and the report is better than a complete failure.

It also supports stable final schemas, which is valuable for refresh-based reporting.

The alias function maps normalized raw D365 field names to internal canonical names. Examples include accountingdate to transaction_date, voucher to voucher_number, ledgeraccount to ledger_account, amountcurdebit to debit_amount, amountcurcredit to credit_amount, and postinglayer to posting_layer.

This is a schema translation layer. It isolates the rest of the query from the raw source vocabulary.

Both description and txt map to the same canonical description field. That means the query can handle the source exposing narrative text under more than one possible name. This is a practical move because operational systems are often inconsistent about which field carries the line description.

That flexibility helps prevent accidental data loss during normalization.
30. Why aliasing is separate from normalization
Normalization makes names comparable. Aliasing makes names meaningful. The first step turns messy field names into predictable technical keys. The second step maps those technical keys to business-centric internal names. That separation improves maintainability because the canonical names are what the rest of the logic uses.

This is a very strong design pattern in Power Query.

This function trims and cleans all text columns, then converts empty strings to null. It helps normalize inputs after import and after type conversion. That keeps the dataset from being cluttered with visually blank but technically non-null strings.

It also improves comparison consistency, because null semantics are easier to reason about than empty-string semantics.

The query often needs to analyze dimensions that are packed into one display string. This function normalizes separator characters, splits the string into pieces, trims each part, and returns a record with dim_1 through dim_8. That makes the structure analyzable downstream instead of leaving it embedded in one opaque text value.

This is a very practical technique in finance reporting because composite dimensions are common.

The function does not assume that all dimension strings use the same separator. It converts ~, -, /, and > to the same delimiter before splitting. That tells you the author expects display strings from different formatting conventions or export paths.

Rather than fail on formatting diversity, the query normalizes it into a common shape.
