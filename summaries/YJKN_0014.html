Penjelasan Kalkulator PPh 21 Masa Pajak Terakhir
Link: Kalkulator PPh 21 Masa Pajak Terakhir
1) Input utama (apa yang diminta user / payroll)
- Status pegawai (pegawai tetap / bukan pegawai).
- Status NPWP (ada / tidak) — berpengaruh pada pengenaan tarif lebih tinggi jika tidak ada.
- Status kawin & jumlah tanggungan (untuk PTKP).
- Masa penghasilan (bulan kerja dalam tahun berjalan).
- Komponen penghasilan: gaji/pensiun, tunjangan pPh (tax allowance / gross-up), tunjangan lain/uang lembur, honorarium, premi asuransi yg dibayar pemberi kerja, natura/kenikmatan, tantiem/bonus/THR.
2) Penjumlahan bruto
- Kalkulator menjumlahkan semua komponen di atas → menghasilkan Penghasilan Bruto Masa (per masa / bulan atau kumulatif sesuai input).
3) Pengurangan yang diizinkan (Deductions)
- Biaya jabatan = 5% × Penghasilan Bruto tetapi dibatasi maksimum Rp500.000/bulan (atau Rp6.000.000/tahun).
- Iuran pensiun / iuran THT/JHT (sesuai kontribusi aktual).
- Zakat yang dibayarkan melalui pemberi kerja (jika ada).
- Hasil pengurangan ini = Jumlah Pengurang → Penghasilan Neto = Penghasilan Bruto − Jumlah Pengurang.
4) Menggabungkan penghasilan masa sebelumnya (jika ada)
- Jika pegawai sudah menerima penghasilan di beberapa masa dalam tahun berjalan, kalkulator mengakumulasi Penghasilan Neto Masa Sebelumnya dan menggabungkan dengan masa ini untuk mendapatkan posisi kumulatif.
5) Konversi ke basis setahun (annualisasi) — ketika “disetahunkan” dipakai
- Untuk kasus disetahunkan: Penghasilan Neto Setahun = Penghasilan Neto Bulanan × 12 (atau jika menggunakan kumulatif: Penghasilan Neto Kumulatif / jumlah bulan kerja × 12).
- Tujuan: mendapatkan basis tahunan untuk menerapkan tarif progresif Pasal 17 dan PTKP tahunan.
6) Hitung PTKP (Penghasilan Tidak Kena Pajak)
- Ambil PTKP sesuai status (mis. TK/0 = Rp54.000.000 per tahun; tambahan Rp4.500.000 untuk tiap tanggungan sampai max 3; tambahan untuk status kawin/istri digabung sesuai ketentuan).
7) Tentukan PKP (Penghasilan Kena Pajak) setahun
- PKP Setahun = Penghasilan Neto Setahun − PTKP (jika negatif → PKP = 0).
8) Hitung PPh terutang setahun (tarif Pasal 17, progresif)
- Terapkan lapisan tarif progresif ke PKP setahun: contoh lapisan umum saat ini: sampai Rp60 jt → 5%, >Rp60 jt s.d. Rp250 jt → 15%, >Rp250 jt s.d. Rp500 jt → 25%, >Rp500 jt s.d. Rp5 miliar → 30%, >Rp5 miliar → 35%.
- Hasilnya = PPh Pasal 21 Terutang Setahun.
9) Proporsionalkan dan rekonsiliasi ke masa pajak terakhir
- PPh disetahunkan untuk masa = PPh Terutang Setahun × (masa kerja dalam bulan / 12).
- Kemudian kurangi PPh yang sudah dipotong pada masa sebelumnya → hasilnya adalah PPh Terutang bulan ini (dapat menghasilkan kekurangan bayar atau kelebihan bayar/refund).
- Jika pilihan gross-up atau tunjangan pajak diaktifkan, kalkulator menyesuaikan agar beban pajak ditanggung pemberi kerja (menghitung gross-up algebraically sehingga penerima menerima netto yang diinginkan).
10) Faktor tambahan yang mempengaruhi hasil akhir
- NPWP tidak ada / NIK belum teraktivasi → pemotongan dapat dilakukan dengan tarif 20% lebih tinggi (yakni pemotongan = 120% dari jumlah PPh yang seharusnya).
- Pembulatan, plafon dan ketentuan PMK/Perdirjen (mis. perlakuan untuk bukan pegawai, tarif TER, dan perhitungan DPP untuk bukan pegawai) ikut memengaruhi langkah-langkah di atas.
11) Output yang ditampilkan oleh kalkulator
- Rincian: Penghasilan Bruto, Jumlah Pengurang, Penghasilan Neto (masa & setahun), PTKP, PKP, PPh Pasal 21 atas PKP, PPh Pasal 21 dipotong masa sebelumnya, PPh Terutang bulan ini, PPh yg sudah disetor dan indikasi kelebihan/kurang bayar.
12) Catatan implementasi & verifikasi
- Pastikan input sesuai bukti payroll (komponen bruto, kontribusi pensiun, zakat, status NPWP/NIK, masa kerja).
- Untuk kasus nyata gunakan angka riil; kalkulator hanya menerapkan aturan logika (annualisasi → tarif progresif → prorata → rekonsiliasi).

Technical User Guide: UpdateHR_From_StaffList (VBA macro)
Purpose & objective
Provide a step-by-step operational and technical user guide for the UpdateHR_From_StaffList macro (refactored variant). The macro automates updating HR and budget sheets from a named staff list: it iterates staff names, runs calculator logic (including an optional Gross-Up GoalSeek), writes results to HR_Calculation and Detail_Budget, and reports unmatched names and GoalSeek failures. The guide explains prerequisites, operation, error handling, auditing, maintenance and recommended controls so country/finance/HR teams can run and support the macro reliably.
Scope
Applies to the UpdateHR_From_StaffList macro and its refactored variant inside the workbook. Covers: environment preconditions, inputs/outputs, step-by-step run instructions, expected behaviour, failure modes, remediation, testing, deployment and change control. Excludes unrelated macros, external payroll systems and centralised treasury processes.
Key definitions & core recognition principle
• staff_list: workbook-level named range (single column) used as the input list of staff names.
• Calculator sheet: worksheet named Calculator containing cells used to compute values and optionally run GoalSeek logic (key cells: F2, F5, F29, F47, F50, G2, G36).
• HR_Calculation: worksheet where results are written to columns M and P for matched staff rows (match on column A).
• Detail_Budget: worksheet where results are written to column S for matched staff rows (match on column B).
• CleanName: canonical normalisation: convert NBSP→space, trim, collapse extra spaces, lower-case (used for robust matching).
Core principle: deterministic, read-only (input) → transform → idempotent write (output) with full state restore on error and clear audit trail of unmatched names and GoalSeek failures.
Core principles / Key principles
1. Preserve workbook/application state: screen updating, events and calculation mode are saved & restored.
2. Safe, reversible changes: original Calculator F5 is restored and temporary F29 formula is reconstituted and then converted to value per original behaviour.
3. Robust matching: use normalised name keys to avoid NBSP/case/extra-space mismatches.
4. Fail fast but recover gracefully: exceptions channelled to a single error handler that performs best-effort restore.
5. Auditability: collect dictMissing and dictGSFail and present a concise user message listing failures (first 25 shown).
6. Performance: build in-memory lookup dictionaries for HR_Calculation and Detail_Budget to avoid repeated Find calls.
Governance & responsibilities (RACI)
• Payroll/HR Lead (R): prepare staff_list, confirm expected name format and approve first-run results.
• Finance Analyst (A): validate output values in HR_Calculation (cols M/P) and Detail_Budget (col S); reconcile to prior month and GL samples.
• Workbook Owner / IT (C): maintain macro code, backup workbook, control macro signing/trust centre settings.
• Reviewer/Test Owner (R): run the test harness on a copy and sign off before production runs.
• End user (I): run macro and act on the summary message (resolve missing names / GoalSeek failures).
Systems & security (Excel, macro security, versioning)
• Store workbook in controlled repository (SharePoint/secured network folder) with versioning enabled.
• Use code signing where possible and require macros enabled only for trusted files.
• Maintain backups before running (recommended automated snapshot prior to production run).
• Restrict access to payroll/H R files and limit people who can run or modify the macro.
• Preserve an immutable change log for macro updates and store signed copies of prior versions.
Pre-requisites & environment
• Excel desktop client with macros enabled and trust for signed workbook.
• Worksheets: Calculator, HR_Calculation, Detail_Budget present and correctly named.
• Named range staff_list defined at workbook level and points to a single column of names (no header row expected, or remove headers before run).
• Key calculator cells must exist and be populated: F2, F5, F29 (VLOOKUP formula), F47, F50, G2, G36.
• Scripting.Dictionary available via CreateObject("Scripting.Dictionary") (no VBA reference required).
• Recommended: save a copy and close other add-in processes that might change calculation behaviour.
Inputs & supporting documentation
• Input: staff_list named range (staff names).
• Supporting: Calculator formulas, HR_Calculation master rows, Detail_Budget master rows, any lookup tables referenced by the Calculator.
• Output: updated HR_Calculation columns M and P; updated Detail_Budget column S; message box listing unmatched names and GoalSeek failures.
• Audit artifacts: retain workbook backup or Save-As copy before run; preserve the macro run log (capture dictMissing and dictGSFail results into an audit sheet on request).
Operation — pre-run, run, post-run
Pre-run:
1. Confirm staff_list is current and contains exact expected staff names.
2. Backup workbook (Save a copy with timestamp).
3. Ensure Calculator cell F29 formula is present if original behaviour expected (the macro clears and re-writes it during cleanup).
4. Notify reviewers if large run ( >500 rows ).
Run:
1. Open workbook and enable macros.
2. From Developer → Macros, run UpdateHR_From_StaffList_Refactored (or assigned button).
3. Macro will iterate staff names, place name into Calculator!F5, calculate sheet, optionally run GoalSeek, and update target sheets. The macro shows a message box at end if there are unmatched names or GoalSeek failures.
Post-run:
1. Review message box output and resolve unmatched names (fix master data or staff_list naming mismatch).
2. Spot-check 5–10 records for correctness (compare F47/F50/G36 values vs expected manual calculation).
3. Save workbook and archive the pre-run backup if run is accepted.
Matching & normalisation rules
• Matching uses CleanName: NBSP→space, WorksheetFunction.Trim (collapse multiple spaces) and LCase (case-insensitive).
• Exact row match: HR_Calculation match column A (key → row).
• Exact row match: Detail_Budget match column B (key → row).
• If your source contains accents, optional accent-stripping may be required (not in default CleanName).
GoalSeek behaviour & retry logic
• If Calculator!G2 = "Gross Up" the macro attempts GoalSeek on F2 to 0 by changing F29.
• First attempt: sheet-level calculation then GoalSeek; if it fails the macro does a full Application.Calculate then retries once.
• Failures are captured in dictGSFail and reported to the user. For persistent failures investigate dependent formula chains, circular references, or invalid starting values in F29.
Error handling & recovery
• Centralised ErrHandler performs best-effort restore: clears temporary cells, restores original F5, re-instates F29 formula and converts it to value (original macro behaviour), recalculates and restores Application state.
• On any unexpected error the macro shows an error message with number & description. Always inspect recovery actions after an error and restore from backup if necessary.
Logging, audit & traceability
• Summary message displays up to the first 25 names for each failure category.
• Recommended enhancement: write dictMissing and dictGSFail to an Audit_Log worksheet with timestamp, run user, workbook name and sample values (this preserves an audit trail beyond transient MsgBox).
• Always keep pre-run backups for audit sampling; include PayrollRun-like naming conventions for the backup file.
Testing & validation (recommended)
• Unit test: create a small test staff_list (3–10 names) with known expected outputs; run macro and verify target cell values exactly match expectations.
• Edge cases: empty names, names with NBSP, multi-space names, names with accents, names present in only one target sheet, duplicate names.
• Performance test: run on larger sample (500–2,000 rows) in a copy of production to measure runtime and memory behaviour.
• Regression test: re-run after any code change and verify F5 and F29 restoration behaviour remains identical.
Templates & repository (code & workbook storage)
• Store macro code in the workbook module and also maintain a text/copy in source control (Git/SharePoint) for version history.
• Recommended file naming for saved copies: Payroll_PR<YYYYMM>_Workbook_UpdateHR_<rundate>.xlsm.
• Keep a Macro_ChangeLog worksheet capturing version number, author, date, summary of changes and approval sign-off.
Common errors & mitigations
Common errors:
• staff_list named range missing or misdefined → macro aborts and displays message. Mitigation: validate named range before run and display clear instructions to the user.
• Name mismatch due to NBSP/case/extra spaces → entries appear in dictMissing. Mitigation: normalise input (CleanName) and run a pre-match diagnostic to list close matches.
• GoalSeek fails due to dependent calc state or poor initial guess → appears in dictGSFail. Mitigation: ensure Calculator formulas are stable, check F29 initial values, run full calculate before attempting GoalSeek or increase max iterations in Excel options.
• Application calculation mode left in Manual on crash → error handler restores state; if not, manually reset Application.Calculation = xlCalculationAutomatic.
Audit expectations & recommended audit pack
Include with each production run (retain in archive): the pre-run workbook copy, the post-run workbook, a run metadata note (user, date, staff_list snapshot), dictMissing log, dictGSFail log, and sample reconciliations showing how updated values map to GL or payroll control sheet. Provide auditors a cross-reference sheet mapping staff_list → HR row → Detail row → output cells.
Governance, escalation & approval thresholds
• Small corrections (typo name fixes): payroll/HR lead may update and re-run.
• Missing master data affecting >5% of list or GoalSeek failures for high-value staff: escalate to CFD and workbook owner for manual review.
• Persistent macro errors or unexpected output variance exceeding tolerance: freeze further runs, restore backup and perform root-cause analysis.

FAQ (short)
Q: What if the macro reports many names as missing?
A: Check for NBSP/extra spaces and run the normalisation diagnostic; confirm HR_Calculation column A and Detail_Budget column B contain the expected names; consider exporting both name lists and running a side-by-side diff.
Q: GoalSeek failed for high-value staff — what next?
A: Inspect dependent Calculator formulas, initial F29 value, and calculation options (iterations/precision). Run a manual GoalSeek to diagnose. Capture the failing staff name in the audit log and escalate if material.
Q: How to preserve the original F29 formula?
A: The macro re-writes F29 formula then converts to value to preserve original behaviour; if you need the formula kept as formula, modify cleanup to skip .Value = .Value step.
