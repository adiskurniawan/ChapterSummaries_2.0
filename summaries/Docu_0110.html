Table 2
Stage
Process
Description / Data Flow / Result
1. User Interaction Browser (index.html) User opens PTT and pastes raw table text. JavaScript (script.core.js / script.table.js) detects the paste event and begins parsing.
2. Client-Side Handling JavaScript Parsing JS may instantly parse small tables for preview. For server-side mode, it sends a POST request to /parse-table with raw text or JSON metadata.
3. Server Handling server.py → html_renderer.py / convert.py The server receives the request, calls the rendering module, normalizes text, detects table structure, and generates clean HTML with <thead>, <tbody>, unique IDs, captions, and metadata. Returns HTML or JSON.
4. Client Injection Browser DOM Update JS receives the server’s response and injects it into the DOM. If HTML → inserted directly. If JSON → iterated and rendered dynamically in the preview pane.
5. Interactivity Dynamic Table Features Rendered tables become interactive: users can copy Markdown, save tables, hover, scroll, and trigger extra JS enhancements.
6. Optional Saving /save-table Endpoint When the user clicks Save, JS posts table data to /save-table. The server writes output to saved_tables/ using atomic I/O helpers from core_io.py.
7. Key Points Learning and Integration - Python server is optional; client can handle parsing and rendering.
- html_renderer.py enables experimenting with Python-based parsing and caption logic.
- Server endpoints bridge browser and Python for real-time preview and testing.

Table 3
Mode
Files / Components
Description, Pros, and Cons
1. Client-Side Rendering (JS) Files: script.core.js, script.table.js How it works: User pastes text → JS parses it directly in the browser → generates <table> HTML → inserts into DOM → activates copy/save/hover/scroll features.

Pros:
• Instant preview, no server delay.
• Works fully offline.
• Direct DOM manipulation.

Cons:
• Complex table rules (caption, escaping, normalization) harder to maintain in JS.
• More difficult for advanced parsing logic.
2. Server-Side Rendering (Python) Files: server.py, html_renderer.py (or convert.py) How it works: Browser sends pasted text to /parse-table. Python receives and processes it, builds <table> HTML fragments, then returns them to JS for injection.

Pros:
• Easier handling of complex parsing and caption logic.
• Centralized control over table rules.
• Simplifies storage or preprocessing before client delivery.

Cons:
• Slower due to network round-trip.
• Requires running Python server.
• Still depends on JS for interactivity.
3. Combined Approach (Hybrid) Files: JS + Python integration How it works: JS provides instant preview and interactivity; Python optionally processes pasted data to produce clean, captioned, normalized HTML. JS then injects Python-generated HTML instead of parsing everything client-side.

Pros:
• Combines JS responsiveness with Python robustness.
• Enables learning and testing both sides.
• Reduces JS parsing complexity.

Cons:
• Slightly more complex coordination between JS and Python.
• Requires consistent data structure between client and server.
✅ Takeaway Recommended Setup - JS handles interactivity and instant rendering.
- Python focuses on reliable table generation, captions, and preprocessing.
- This hybrid model maximizes flexibility for learning and performance.

Table 4
Approach
Pros
Cons / Notes
1. PasteToTable (PTT) – JS-heavy, client-side • Instant preview: tables appear immediately without server round-trip.
• Fully interactive in-browser: copy, save, hover, dynamic adjustments handled by JS.
• Easy deployment: just open index.html; no Python server needed.
• Lightweight: no dependency on Python or render/ folder unless desired. 		• Parsing complexity: JS is harder for advanced text parsing, caption injection, or normalization.
• Consistency: multi-line cells or special Markdown can be tricky.
• Limited exposure to Python workflows or structured rendering logic like TV2.1.
2. Table Viewer 2.1 (TV2.1) – Python-heavy, server-side • Robust parsing: Python handles multi-line tables, captions, metadata, and escaping easily.
• Consistent HTML: outputs full-page, TV2.1-style tables with IDs, captions, and metadata.
• Reusable logic: render/ folder contains modular functions for any Python project.
• Great learning potential: tweak Python functions to understand structured table rendering. 		• Slower preview: requires server round-trip to render tables.
• Heavier setup: Python server must run; more dependencies.
• Interactivity (copy/save/DOM effects) still relies on client-side JS.
3. Hybrid Approach – PTT + TV2.1 • Fast client-side UX via PTT JS.
• Robust, consistent table HTML via TV2.1 Python rendering.
• Supports learning both JS and Python workflows.
• Captions, IDs, normalization handled exactly like TV2.1. 		• Slightly more complex setup: coordinate client-side preview with optional server-side rendering.
• Requires both JS and Python components to work together.
✅ Takeaway - Use PTT alone for speed and simplicity.
- Use TV2.1 alone for professional, consistent table rendering and learning Python.
- Combine PTT + TV2.1 for best of both worlds: instant UX plus robust, structured backend rendering. 		—

1. Locked TV2.1 Files Files in render/ (e.g., convert.py, core_renderer.py, core_table.py) remain untouched. They provide full table generation, caption injection, and metadata logic. • No edits required.
• Functions can be imported and called directly.
• Ensures PTT uses official TV2.1 rendering.
2. How PTT Uses Them Server receives pasted table text via POST /parse-table.
Server calls a wrapper (html_renderer.py) that internally invokes TV2.1 functions:
python<br>from render import convert, core_renderer<br>def render_tables_from_text(raw_text):<br> tables_html = convert.convert_text_to_tables(raw_text)<br> final_html = core_renderer.wrap_tables(tables_html)<br> return final_html<br>
Returned HTML fragments are sent to JS, which injects them into the DOM and manages interactivity (copy, save, hover). 		• TV2.1 logic is used as a read-only library.
• JS handles all user interactions without altering TV2.1 files.
3. Benefits Combines Python rendering with PTT’s client-side features.
Learning opportunity: you see how TV2.1 generates tables, captions, and IDs.
Future TV2.1 updates automatically improve PTT output. 		• Learn Python rendering exactly as TV2.1 does.
• Retains instant JS preview and interactivity.
• Minimal maintenance; automatic integration of TV2.1 updates.
