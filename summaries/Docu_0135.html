pph21_2025 — Excel (Pure-VBA) User Guide (verified ×10)
Quick verification note: This guide was reviewed and refined with repeated checks (10×) to ensure consistency with the provided VBA modules, sheet names, table schemas, and I/O behavior. Follow steps in order; each numbered item is a single actionable instruction or explanation.
1. Save workbook as macro-enabled — Save the file as pph21_2025.xlsm. Ensure Excel is set to allow macros for this file (Trusted Location or enable macros on open). Failure to enable macros prevents the automation from running.
2. Required worksheets (exact names) — Create these sheets exactly with the following names: Settings, Input, Rules_Brackets, Rules_Params, Outputs_BuktiPotong, Outputs_Per11, Logs. The code expects those sheet names (or a custom sheet name set via Settings keys).
3. Create the Input table (tblInput) — On the Input sheet insert an Excel Table (ListObject) and set its table name to tblInput. Required columns (headers exactly): nip, npwp, name, period, period_type, gross, npwp_status. Additional columns are allowed and will be preserved and copied into outputs. Use text format for IDs (nip/npwp) to avoid numeric rounding.
4. Rules — brackets table (tblBrackets) — On Rules_Brackets create a Table named tblBrackets with two columns: upper (numeric) and rate (decimal, 0..1). Rows must be ordered ascending by upper. The final row must contain a very large upper value to act as the top bracket (e.g., 9999999999). Each row is interpreted as a closed upper bound for that bracket.
5. Rules — params table (tblParams) — On Rules_Params create a Table named tblParams with columns key and value. Common keys: dtp_pct (decimal, e.g., 0.05), npwp_penalty_pct (decimal, e.g., 0.20), annualization_default (string: monthly, weekly, daily, yearly). Values are read as strings/numbers; numeric values will be converted in code.
6. Settings sheet keys & defaults — On Settings put keys in column A and values in column B. Required key: OUTPUT_BASE (absolute folder path where outputs will be written, e.g., C:\pph21_output). Optional keys: JOB_ID (if empty a job id is autogenerated), RULES_BRACKETS_SHEET (default Rules_Brackets), RULES_PARAMS_SHEET (default Rules_Params). The macro uses the first 100 rows of Settings.
7. JOB_ID format and behavior — If JOB_ID is left blank the macro auto-generates a job id in the form job_YYYYMMDD_HHNNSS (lowercase job_, timestamped). That JOB_ID is saved back to Settings for traceability. Each run uses OUTPUT_BASE\<JOB_ID>\ as the job folder.
8. VBA references & runtime notes — The code uses late binding for ADODB and Scripting.Dictionary but enabling Microsoft Scripting Runtime (Tools → References) is recommended to get Dictionary IntelliSense. If your environment lacks System.Collections.ArrayList (used as a simple dynamic array), the code falls back to array/Collection variants; if you see errors, change the collection implementation to VBA Collection or enable required runtime components.
9. Paste VBA modules (module names) — Open the VBA editor (ALT+F11). Insert Standard Modules and paste provided code into modules named exactly: modMain, modIO, modCalc, modManifest, modUtils. Save the workbook. Do not rename modules unless you also update any Public procedure calls that refer to them.
10. Example tblBrackets (sample rows) — Example bracket rows (upper → rate): 50000000 → 0.05, 250000000 → 0.15, 500000000 → 0.25, 1000000000 → 0.30, 9999999999 → 0.35. Ensure ascending upper; the last value should be a sufficiently large sentinel to capture incomes above the highest nominal bracket.
11. Example tblParams (sample entries) — Suggested params: dtp_pct → 0.05, npwp_penalty_pct → 0.20, annualization_default → monthly. dtp_pct is applied as a withholding relief fraction (tax*(1-dtp_pct)). npwp_penalty_pct is applied as a multiplicative penalty when npwp_status = "no".
12. Add a sample input row — Add a single test row in tblInput to validate: nip=12345, npwp=0012345678, name=John Doe, period=2025-11, period_type=monthly, gross=10000000, npwp_status=has. Use this to perform an initial functional test.
13. How to run the macro — Entry point: pph21_RunAll. Run from Excel: Developer → Macros → select pph21_RunAll → Run. Alternatively create a Form button on Settings and assign pph21_RunAll. The macro performs a full job run (read settings, load rules, process rows, write CSVs, write manifest, import CSVs to output sheets, log).
14. What a run performs (sequence) — High level: (a) validate settings and create <OUTPUT_BASE>\<JOB_ID>\; (b) read tblInput rows into memory; (c) load tblBrackets and tblParams; (d) per row: annualize gross (monthly→×12, weekly→×52, daily→×365), compute progressive tax via bracket piecewise algorithm, apply dtp_pct, apply NPWP penalty if npwp_status = "no", compute monthly withholding = final annual tax / 12; (e) assemble bukti and per11 records; (f) atomically write bukti_potong.csv and per11_payload.csv; (g) compute and write manifest.json containing SHA-256 checksums; (h) import CSVs to Outputs_BuktiPotong and Outputs_Per11; (i) append run entries to Logs.
15. Output file locations & names — Files are written to <OUTPUT_BASE>\<JOB_ID>\ with exact names: bukti_potong.csv, per11_payload.csv, manifest.json. The macro writes .tmp files first then renames them to ensure atomic writes and avoid partial files.
16. Default CSV schemas (what code produces) — The implementation currently produces these default columns (exact names): bukti_potong.csv → nip, npwp, name, period, period_type, gross, annual_gross, annual_tax, tax_after_dtp, tax_after_penalty, monthly_withholding, plus any extra input columns preserved verbatim. per11_payload.csv → nip, npwp, period, monthly_withholding, annual_tax (minimal per-payload fields). To change columns, update ProcessRow in modCalc.
17. Manifest structure & integrity — manifest.json contains job_id, generated_at, and an array files[] with entries {path, size, modified, sha256}. SHA-256 is computed in VBA and written as lowercase hex. Use the manifest to verify output integrity. On Windows you can independently confirm a checksum (e.g., certutil -hashfile <file> SHA256) to match the manifest.
18. Atomic write behavior — CSV and manifest writing use a .tmp → rename pattern: the macro writes to <filename>.tmp and then Name-moves it to the final filename. If the final exists it is removed first. On errors the .tmp file is deleted (best effort). This pattern reduces risk of partial/garbled output files on failure.
19. Logs and troubleshooting basics — The Logs sheet receives timestamped messages for each major action and any errors. If something fails, inspect Logs first. Typical quick fixes: enable macros and Scripting Runtime; ensure OUTPUT_BASE exists or allow macro to create it and verify permissions; confirm tblInput is nonempty; check that tblBrackets and tblParams are present and correctly formatted.
20. Common errors and resolutions — • Empty outputs: ensure tblInput has rows and required columns. • Write permission errors: confirm OUTPUT_BASE folder is writable by the current user and antivirus is not blocking. • Missing runtime objects: enable Microsoft Scripting Runtime or alter code to use VBA native Collection. • SHA-256 slow: VBA implementation is CPU-bound; for very large files consider verifying externally or using a dedicated tool for bulk checks.
21. Validation & parity testing (recommended) — Recommended tests: (A) Boundary bracket test — income exactly equals a bracket upper. (B) Large income test — income above top bracket. (C) NPWP missing test — npwp_status = no. (D) Period variations — test monthly, weekly, daily, yearly inputs and annualization_default. Compare annual tax and monthly withholding against a trusted implementation (Python or authoritative calculator).
22. Backup and version control — Before editing VBA or changing rule tables, save a copy: File → Save As → pph21_2025_backup_<YYYYMMDD>.xlsm. Keep versioned backups (date stamped) to allow reverting. If you integrate into a team workflow, store backups in a versioned repository (zip the .xlsm if necessary).
23. Extending rules & logic — To change tax logic or add new rule types: edit tblBrackets / tblParams for simple changes. For conditional/complex rules, implement additional tables and extend ProcessRow in modCalc to consult them. The codebase is modular: modCalc does core math; modIO handles table I/O and CSV atomic writes; modManifest computes checksums.
24. Performance guidance — The VBA pipeline is optimized for clarity and auditability, not for very large datasets. Practical target: hundreds to low thousands of input rows per run. For larger volumes: (a) split tblInput into batches and run multiple jobs; (b) migrate heavy workloads to Python while keeping Excel as UI; (c) avoid very large OUTPUT_BASE folders on slow network drives.
25. Security & environment considerations — Macros and file writes pose risk if the workbook is modified by untrusted parties. Restrict access to the workbook and to the OUTPUT_BASE folder. Do not run the macros on untrusted workbooks. The VBA implementation does not perform network I/O. ADODB.Stream and Scripting.Dictionary are used locally.
26. Re-running jobs and avoiding overwrite — If you want to re-run using the same JOB_ID, be aware that outputs in the same folder may be overwritten. To preserve historical runs, either specify a new JOB_ID in Settings before the run or leave JOB_ID blank to let the macro generate a fresh timestamped id.
27. Importing CSVs into sheets — After successful write the macro imports the generated CSVs into Outputs_BuktiPotong and Outputs_Per11 via QueryTables. The import treats columns as text to preserve identifiers. If you need different parsing behavior, adjust ImportCsvToSheet in modIO.
28. Testing tips & sample workflows — Create a hidden Test sheet containing example input rows and expected outputs. Implement a Test_RunAll macro that populates tblInput with test cases, runs pph21_RunAll, and asserts expected CSV values in Outputs_* sheets. This approach helps regression testing after code or rule changes.
29. Documentation & module comments — Each module contains inline comments and top-of-module documentation in the supplied code. Review modCalc to understand how ProcessRow maps inputs to bukti and per11 dictionaries, and update comments when you extend logic.
30. Support & feedback — If you need assistance, prepare a minimal reproducible example: include a copy of the workbook (with sensitive data redacted), the Settings values, sample tblBrackets and tblParams, and a few tblInput rows. Attach Logs output and the exact error message. For formal support follow your organization’s support channel.
31. Recommended operational checks before production runs — 1) Validate tblInput column names and data types. 2) Confirm tblBrackets has ascending upper values and a top sentinel. 3) Confirm OUTPUT_BASE exists or that the macro can create it. 4) Backup the workbook. 5) Run the macro on a small subset and inspect Logs.
32. Change control and auditability — For audit trails, record: the JOB_ID, the Settings snapshot, a copy of tblBrackets/tblParams used, and the generated manifest.json (SHA-256 values). Keep these artifacts together for reproducibility and audit reviews.
33. When to migrate to hybrid or external tooling — If you need high throughput, parallel processing, advanced rule languages (YAML/DSL), or faster checksum computation, consider using the original Python pipeline for heavy lifting and retain this Excel solution as a user interface or QA tool.
34. Additional housekeeping — Periodically clean up OUTPUT_BASE to archive or remove old job folders. If using network drives, prefer local temporary folders for atomic writes and then move final job folders to the network target to reduce partial-write risk.
35. Summary checklist before first run — ✔ pph21_2025.xlsm saved; ✔ macros enabled; ✔ sheets exist with exact names; ✔ tblInput, tblBrackets, tblParams created and populated; ✔ Settings has OUTPUT_BASE; ✔ modules pasted and saved; ✔ Scripting Runtime enabled (recommended); ✔ test row added; ✔ backup taken.
36. Final confirmation — After completing the checklist, run pph21_RunAll. Inspect Logs, the files written to <OUTPUT_BASE>\<JOB_ID>\, and the imported sheets Outputs_BuktiPotong and Outputs_Per11. If behavior differs from expectations, consult the Logs output and the manual validation tests above.

Technical Breakdown PPh 21 VBA
Overview & intent:

Pure-VBA reimplementation of the pph21_2025 processing pipeline designed to run entirely inside an Excel workbook (.xlsm). Eliminates Python dependency; loads rules from workbook tables; performs annualization, progressive tax (TER), DTP adjustments, NPWP penalties; produces bukti_potong and per11 CSVs; generates manifest.json with SHA-256 checksums; imports outputs back into sheets. Emphasizes atomic writes, auditability, offline operation.
Primary design goals:

Single-file deployment (.xlsm). Deterministic outputs. Editable rules via tables. Atomic file writes (.tmp then rename). Embedded SHA-256 integrity. Minimal external dependencies (ADODB.Stream, Scripting.Dictionary). Prioritize correctness & maintainability.
Workbook structure (required sheets):

Sheets: Settings, Input (tblInput), Rules_Brackets (tblBrackets), Rules_Params (tblParams), Outputs_BuktiPotong, Outputs_Per11, Logs. Settings stores key/value pairs. Input is the only required user data.
Settings content & named keys:

Settings sheet: Column A = keys, Column B = values. Required: OUTPUT_BASE. Optional: JOB_ID (autogenerated). RULES_BRACKETS_SHEET default Rules_Brackets. RULES_PARAMS_SHEET default Rules_Params. Accessed via GetSettingValue / SetSettingValue.
Input table schema (tblInput):

Columns: nip, npwp, name, period (YYYY-MM), period_type (monthly/yearly/etc), gross, npwp_status (has/no). Additional columns allowed; passed through to outputs if needed. Must be a ListObject.
Rules tables schema:

tblBrackets: upper, rate. Ascending by upper. Last row uses very large upper. tblParams: key/value pairs (dtp_pct, npwp_penalty_pct, annualization_default).
Top-level control module (modMain):

Entrypoint pph21_RunAll: read/validate settings; create job directory; read Input into array; load brackets/params; loop ProcessRow per input; collect bukti/per11 dictionaries; write CSVs atomically; write manifest.json atomically; import CSVs; log; fail-fast on critical errors.
I/O and CSV module (modIO):

TableToArray, RowToDict, ImportCsvToSheet, WriteCsvAtomic, CsvEscape, FileExists, EnsureFolderExists. CSV writer deduces header from dictionary keys. Atomic-write safety.
Calculation module (modCalc):

ProcessRow: read fields; Annualize (monthly×12, weekly×52, daily×365); CalcProgressiveTax; apply DTP; apply NPWP penalty; compute monthly withholding (taxAfterDTP/12); build bukti & per11 dictionaries; safe error logging with row skip.
Progressive tax algorithm:

Bracket-based: iterate brackets; taxable = min(income, upper) − lower; accumulate tax = Σ taxable×rate. If income > top bracket: apply top rate to excess. Rounding applied during CSV serialization.
Manifest & integrity (modManifest):

WriteManifest builds JSON: job_id, generated_at, files[{path, size, modified, sha256}]. Uses ReadBinaryFile + SHA256_Bytes + BytesToHex. Written atomically (tmp→final).
SHA-256 implementation details:

Pure-VBA SHA-256: block processing, padding, compression with emulated 32-bit ops (Long). Returns 32-byte array → hex. Works for typical file sizes; slow for very large files.
Utilities & logging (modUtils):

LogAdd appends timestamp; GetSettingValue / SetSettingValue scan first 100 rows; EnsureTrailingSlash; CreateScriptingDictionary. Logs errors; MsgBox only for critical failures.
Atomic-write pattern:

WriteCsvAtomic: write to path.tmp → delete final if exists → rename. Manifest uses same pattern. Prevents partial writes.
Error handling & robustness:

Each module uses On Error. ProcessRow logs errors & returns Nothing. pph21_RunAll aborts on fatal I/O failures. Folder creation checks. User notified via MsgBox and Logs.
Data types & serialization:

Internally numeric = Double. CSV serialization: gross raw; annual_tax, tax_after_dtp, monthly_withholding rounded to 2 decimals. CsvEscape quotes fields as needed.
Performance considerations:

Sequential row loop in VBA; practical scale: hundreds–low thousands. Brackets and params pre-loaded for speed. SHA-256 slow on large files.
Testing & validation:

Manual unit-style tests: bracket boundaries; zero incomes; NPWP missing; DTP changes; annualization_default; manifest checksum consistency. Compare against Python reference.
Security & environment notes:

ADODB.Stream and Scripting.Dictionary require standard Windows Excel trust. Outputs written to OUTPUT_BASE; user must ensure permissions. No external shell/network.
Limitations vs Python original:

Reduced rule expressiveness. No complex YAML branching. Only bracket rates & global params. Manifest has limited provenance.
Extensibility & maintenance:

Add rule tables as needed; update ProcessRow. Future tax changes via tblBrackets/tblParams. Complex rules may use formula-evaluator with VBA Evaluate. For heavy workloads, migrate back to Python backend.
Outputs & repo mapping:

OUTPUT_BASE/JOB_ID/bukti_potong.csv, per11_payload.csv, manifest.json. CSV schemas based on bukti/per11 dictionary keys. ImportCsvToSheet loads results into output sheets.
Developer guidance:

Preserve module boundaries: modMain, modIO, modCalc, modManifest, modUtils. Keep Scripting.Dictionary. When changing CSV schema, update header order logic. Add regression Test sheet and Test_RunAll macro.
