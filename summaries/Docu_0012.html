Logic / Structure
Conceptual Notes
Introduction The provided html_renderer.py is a merged, robust fallback/assembly module that:
1. Loads preferred shims (render.core_compat → core_renderer) and assigns render_html, render_table, render_page with clear diagnostics.
2. Exposes conversion helpers and graceful fallbacks (render_from_text, _fallback_render_html, _fallback_render_table, render_page_from_fragments).
3. Implements fragment extraction + concurrent fragment rendering (render_fragments_concurrently) with thread/process options, atomic writes, asset detection heuristics, and fragment metadata.
4. Assembles full page HTML with TOC placed directly under header (_assemble_page_html) and ensures &lt;table&gt; class injection.
5. Exports fragment helpers and a compact diagnostic API.
Key issues / incompatibilities 1. html_renderer.py expects render.core_assets to expose prepare_assets and write_overrides_css, but the provided render/core_assets.py exposes get_inline_css, get_inline_js, list_available_assets, copy_assets_to, _compute_sri.
2. Decide which side to adapt: add prepare_assets/write_overrides_css to core_assets.py, or modify html_renderer.py to call the existing get_inline_* and copy_assets_to.
3. html_renderer.py optionally imports .io_utils._atomic_stream_write — ensure render/io_utils.py provides that symbol or keep _atomic_write_text fallback (present).
4. Optional external dependencies: BeautifulSoup and pandas are attempted; behavior is safe but should be documented in README/tests.
5. Concurrency + pickling: ProcessPoolExecutor path checks pickleability of renderer — acceptable but note platform differences.
Risks 1. Breaking public API if we change names in core_assets that other modules expect.
2. Cross-platform atomic rename/mode/mtime behavior (Windows vs POSIX) when copying assets or writing fragments.
3. Large binary assets (e.g., SheetJS) — should be opt-in and possibly excluded from inline/copy by default.
4. Race conditions if multiple callers write to same tmp_frag_dir concurrently (document/guard).
5. Unhandled exceptions in third-party bridges may surface; current code logs and falls back (good).
Recommended next code changes (today) 1. Small, single-file change first: Update render/core_assets.py to add two thin helpers expected by html_renderer.py:
  - prepare_assets(dest_dir, assets=None, compute_sri=False, overwrite=False) — wrapper calling copy_assets_to with sensible defaults and returning a small map of copied assets + SRI.
  - write_overrides_css(dest_dir, overrides_text) — write an overrides.css file atomically into dest.
2. Add lightweight unit tests for core_assets behaviors (read/caching, copy_assets_to, _compute_sri, and the new helpers).
3. Integrate html_renderer.py with core_assets by confirming import paths (no change needed if we add those helpers).
4. Add a small integration test: run render_fragments_concurrently on a couple of sample inputs and call render_page_from_fragments to verify TOC placement and fragment assembly.
