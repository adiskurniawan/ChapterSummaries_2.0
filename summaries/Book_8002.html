`D:\Dropbox\Projects\tablesviewerv2_1\`

Root project folder. Contains all source code, assets, and outputs for the `tablesviewerv2_1` system. This project handles table rendering, Markdown-to-HTML conversion, sanitization, and front-end integration. Everything the program needs is organized here.

render\

Subfolder where the core rendering logic lives. It holds the Python modules that handle input text parsing, Markdown interpretation, HTML generation, and data sanitization. This is the "engine" of the project.

`render\core.py`

Central controller for rendering. Likely manages the pipeline: loading input, passing it through conversion, parsing Markdown, sanitizing, and producing HTML. If the HTML file is blank, this file is a prime suspectâ€”it might not be writing the parsed output correctly.

`render\convert.py`

Handles transformation of intermediate formats into renderable structures. Converts parsed Markdown tokens, table structures, or plain text into HTML blocks. Think of it as the "translator" from internal Python objects to HTML strings.

`render\markdown.py`

Specialized parser for Markdown. It interprets raw text (like headings, bold, italics, links, tables) and converts them into HTML tags. This is what makes your 2-column tables render correctly. If tables or formatting disappear, this file needs inspection.

`render\io_utils.py`

Input/output utilities. Could handle reading raw Markdown files, writing HTML files, or managing input/output streams. It acts as a helper for `core.py` to abstract file system operations.

`render\sanitize.py`

Cleans up HTML or Markdown to remove unsafe tags, attributes, or JavaScript. Prevents injection attacks (like ``) and ensures the final HTML is safe. If your output looks empty, over-aggressive sanitization here could be stripping too much.

`render\types.py`

Defines dataclasses, enums, or type structures used across rendering modules. Example: a `TableCell` object, a `MarkdownBlock`, or a `RenderResult`. Keeps type consistency across modules.

assets\

Front-end static assets. Contains CSS and JavaScript that style and power the output HTML. These files are copied to the output folder when rendering so the browser can use them.

`assets\style.css`

Primary stylesheet. Controls colors, spacing, font, and table formatting. Determines how the 2-column tables appear in the browser. If output is written but looks plain or broken, this file might be missing or not linked.

`assets\script.js`

Front-end JavaScript logic. Could handle interactive features like collapsible sections, copy-to-clipboard buttons, or dynamic table formatting. Not critical for plain HTML display, but important for advanced UX.

`assets\worker.js`

Web Worker script for handling heavy tasks in the browser (e.g., rendering math, syntax highlighting, or sanitization off-thread). Keeps the UI responsive. Optional but improves performance.

`assets\overrides.css`

Extra CSS overrides for styling. Lets you patch or adjust default styles without editing `style.css` directly. Useful for quick fixes or theme tweaks.

output\_combined\

Destination folder where rendered HTML files and copied assets are stored. This is what the user opens in a browser. If nothing shows up, check whether `core.py` actually wrote into this folder.

`output_combined\assets\`

Local copy of the `assets\` folder. Automatically copied during render so that the output HTML can link to its CSS/JS locally. Browser will load files from here instead of the source `assets\` directory.

`html_renderer.py`

Wrapper module to keep `main.py` compatible. Provides a higher-level API for rendering Markdown into HTML. Likely imports functions from `render\core.py` and handles integration with other code.

`utils.py`

General-purpose helper functions. Could include text cleanup, logging, path handling, or small string utilities. Supports other modules without being tied to rendering logic.

`helpers.py`

Another utility script, possibly older or domain-specific. May contain functions for debugging, testing table formatting, or preparing data for rendering.

`main.py`

Entry point of the project. Orchestrates the workflow: reads user input, calls `html_renderer.py` or `render\core.py`, and triggers output generation. Running `python main.py` is how you produce the final HTML file.
