Technical Breakdown
Overall model
1. Overall model — how the three sheets work together
Staff is the master employee reference, SC_Allocation is the month-by-month approval-code assignment map, and SCA_Data is the monthly cost output by person and approval code. Together, they form a full staff costing and allocation workflow.
2. Overall model — practical interpretation
In operational terms, the workbook first identifies who the staff member is, then determines how that staff member is split across approval codes in a given month, and finally calculates the payroll cost lines for that split.
3. Overall model — key dependency chain
If Staff is incorrect, identity and tax data become unreliable. If SC_Allocation is incorrect, the cost split by month will be wrong. If SCA_Data is incorrect, the financial output will not reconcile. Each sheet has a distinct role, but all three depend on each other.
Staff sheet
1. Staff — what this sheet is
This sheet is the employee master data table. It contains identity, payroll, tax, and administrative information for each staff member.
2. Staff — row grain
Each row represents one staff record. This is the reference sheet from which names, codes, tax details, and HR attributes are maintained.
3. Staff — key identifiers
Staff Code, D365 Staff Code, Name, and Tax ID Number are the main identity fields. These are the columns most likely used for lookup, validation, and system matching.
4. Staff — D365 mapping behavior
D365 Staff Code appears to be the ERP or Dynamics 365 reference. A value of #N/A in that field suggests the staff member is not mapped cleanly to D365 or the lookup failed.
5. Staff — payroll attributes
Gross Up (Y/N) and Marital Status are payroll and tax treatment fields. These values usually influence how salary, allowances, and withholding are calculated.
6. Staff — bank and reporting fields
Name in BNI is likely the bank-ready display name used for payment processing. Notes Sisca and Notes on Staff Code are operational comment fields for exceptions, admin notes, or special handling.
7. Staff — business role
This sheet functions as the master reference behind the allocation and payroll costing sheets. It provides the stable staff identity data that the monthly costing logic depends on.
8. Staff — relationship to the other sheets
SCA_Data uses Name and Approval code for cost allocation, while SC_Allocation uses Name and monthly approval mapping. The Staff sheet should be the source used to validate names, codes, tax IDs, and payroll flags before those costs are distributed.
9. Staff — data quality point
Because the costing and allocation sheets rely on names, code consistency is important. Any mismatch between Staff, SC_Allocation, and SCA_Data can create duplicate records, missing mappings, or incorrect payroll charging.
SC_Allocation sheet
1. SC_Allocation — what this sheet is
This sheet appears to be the allocation mapping table that drives how each staff member’s cost is assigned across months. It is a crosswalk between employee names and monthly approval codes.
2. SC_Allocation — row grain
Each row represents one person. Each month column shows which approval code applies for that person in that month. The sheet is therefore a time-based assignment matrix.
3. SC_Allocation — column structure
The columns run from JAN Approval Codes through DEC Approval Codes. That makes the sheet a full-year reference table rather than a transactional table.
4. SC_Allocation — single-code behavior
For many employees, a month contains one approval code repeated across the year. That means the person is fully assigned to a single approval bucket for those months.
5. SC_Allocation — split-code behavior
Some cells contain multiple approval codes with percentages, such as OCT2529080 | 90%; 247GR | 10%. That indicates a monthly split allocation, where one person’s cost is divided between two approval codes in fixed proportions.
6. SC_Allocation — operational role
This sheet likely acts as the source of truth for how monthly costs should be distributed when building SCA_Data. It tells the model whether a person belongs to one approval code or must be split across more than one.
7. SC_Allocation — relationship to Period
The month columns should line up with the Period values used in SCA_Data. For example, 202605 in SCA_Data should correspond to the May column in this allocation sheet.
8. SC_Allocation — why this sheet matters
Without this mapping, the cost rows in SCA_Data would not know how to distribute staff cost by approval code and month. This sheet is the allocation engine behind the payroll costing structure.
9. SC_Allocation — data quality point
The format inside each month cell must be consistent. If some cells use plain codes and others use split notation, any downstream automation must be able to parse both patterns correctly.
SCA_Data sheet
1. SCA_Data — what this sheet is
This sheet appears to be the monthly payroll cost fact table. It stores the financial result at a detailed allocation level for each person, period, and approval code combination. In practice, this is the sheet where payroll components are assembled into a month-by-month staff cost record.
2. SCA_Data — row grain
Each row is one monthly cost record for one person under one approval code. That means a single employee can appear multiple times in the same period if the payroll cost is split across approval codes or job allocations.
3. SCA_Data — key identifiers
The main keys are Period, Name, and Approval code. Together, these identify the cost line and tell you when the cost belongs, who it belongs to, and which approval bucket it is charged to.
4. SCA_Data — FTE logic
FTE in Month shows the share of the employee assigned to that record. In your sample, one person can have 100% on one code, or be split into 90% and 10% across two codes. This is the core allocation logic behind the sheet.
5. SCA_Data — compensation columns
Gaji, Tunjangan Asuransi, Tunjangan lain-lain, Tunjangan PPh 21, Tunjangan Hari Raya, and Severance Pay represent payroll and benefit components. These are the positive cost inputs that build total employment cost.
6. SCA_Data — statutory cost columns
JSHK, BPJSTK Employer, and BPJSKes Employer are employer-side contributions or payroll-related charges. These columns show the non-salary cost burden associated with the employee.
7. SCA_Data — tax handling columns
PPh 21 Payable, BPJSTK Payable, and BPJSKes Payable appear to represent payable offsets or deductions tied to payroll processing. The negative signs in the sample suggest these are treated as liabilities or contra entries in the cost build-up.
8. SCA_Data — THP meaning
THP is typically take-home pay. In this structure, it appears as a payroll result line rather than a pure gross salary input. It sits alongside the deduction and employer-cost fields, which suggests the sheet is tracking both employee net pay and employer cost impact.
9. SCA_Data — total staff cost
Total staff cost is the final roll-up column. It appears to combine salary, allowances, benefits, employer contributions, and tax-related items into one monthly cost figure per allocation line.
10. SCA_Data — split allocation behavior
The repeated name Iqbal Ridzi Fahdri Elyazar with two approval codes shows that one employee can be costed across multiple approval buckets in the same month. That is the strongest sign that this sheet is allocation-driven, not just a simple payroll list.
11. SCA_Data — business purpose
This sheet is likely used for cost reporting, budget allocation, and managerial chargeback by approval code. It is the detailed output sheet that downstream reports would use for monthly staff cost analysis.
12. SCA_Data — data quality point
Because the same person can appear multiple times, consistency between Name, Approval code, and FTE in Month is critical. Any mismatch here would distort monthly cost allocation.

Per-Sheet Data Model & Control Architecture
Staff (Master Identity & Attribute Layer) — Purpose, contract, invariants, PII controls, and reconciliation
Purpose & contract: Canonical master data layer establishing employee identity, statutory attributes, and ERP alignment. Serves as the immutable (within a period) reference for downstream allocation and costing. Must enforce a strict 1:1 relationship between surrogate key (Staff Code) and human entity.
Inputs & outputs: Consumes HRIS feeds or manual entries. Outputs canonical identity profiles (Staff Code, D365 Staff Code, Tax ID Number, Gross Up (Y/N), Marital Status, Name in BNI).
Primary invariants (must/shall):
1. Staff Code and Tax ID Number MUST be globally unique and non-null across the active dataset.
2. Name must be standardized (e.g., trimmed, consistent casing) to prevent fuzzy-match failures in downstream joins.
3. D365 Staff Code resolution failures MUST yield a deterministic #N/A or explicit exception flag; silent dropping of unmapped records is forbidden.
4. Payroll drivers (Gross Up, Marital Status) MUST be validated against statutory enums before downstream tax calculations consume them.
Observability & audit fields: Track changes to Tax ID and Gross Up flags with timestamp and operator ID. Emit staff.master.exception audit for unmapped D365 codes or missing statutory fields.
Failure modes & recovery: Duplicate names → enforce Staff Code surrogate keys for all joins. Missing Tax ID → block downstream costing and flag for HR review. Stale D365 mapping → trigger ERP reconciliation workflow.
Implementation notes & safe I/O: Lock identity columns during active payroll processing. Use exact-match (XLOOKUP with exact match or INDEX/MATCH) for downstream joins; prohibit approximate matching. Treat Notes Sisca as unstructured metadata; never parse it for computational logic.
Tests & validation rules: Uniqueness constraints on Staff Code/Tax ID. Null-check on mandatory statutory fields. Cross-reference D365 codes against ERP active directory dump.
SC_Allocation (Temporal Routing & Split Logic Engine) — matrix structure, split parsing, 100% invariants, and period mapping
Purpose & contract: Temporal routing engine defining monthly cost distribution rules. Translates static employee identity into time-bound financial assignments (single code or percentage splits). Acts as the deterministic bridge between master data and fact generation.
Inputs & outputs: Inputs: Staff Code/Name, Approval Code registry, monthly planning directives. Outputs: 12-month allocation matrix (JAN to DEC Approval Codes).
Primary invariants (must/shall):
1. The 100% Rule: For any employee in any active month, the sum of FTE allocation percentages MUST equal exactly 100.00%.
2. Syntax Determinism: Split allocations MUST follow a strict, parsable regex (e.g., ^(CODE1 \| \d{1,3}%; CODE2 \| \d{1,3}%)$). Free-text deviations or missing percentage signs are forbidden.
3. Temporal Continuity: Unassigned months must either inherit the prior month's rule (forward-fill) or be explicitly flagged as 0%/Unallocated; ambiguous blanks are prohibited.
4. Referential Integrity: Every Approval code referenced MUST exist in the canonical Chart of Accounts / Approval registry.
Observability & audit fields: Emit allocation.split.warning if sum != 100%. Track month-over-month variance in allocation codes. Log allocation.override when manual splits deviate from standard HRIS assignments.
Failure modes & recovery: Malformed split strings → reject row and flag for manual correction. Sum != 100% → block SCA_Data generation. Missing approval codes → fallback to default overhead code or halt pipeline.
Implementation notes & safe I/O: Use data validation lists for single codes to prevent typos. For splits, enforce strict string templates via conditional formatting or VBA validation. Avoid using Name as the primary join key; use Staff Code to insulate against name changes, displaying Name only for UX.
Tests & validation rules: 100% sum validation across all rows/months. Regex validation for split syntax. Orphaned code detection against the master approval registry.
SCA_Data (Fact & Costing Output Layer) — row grain, cost build-up, FTE application, and GL reconciliation
Purpose & contract: The authoritative fact table for monthly staff costing. Synthesizes master attributes, temporal allocations, and raw payroll components into fully burdened cost lines. Serves as the final ledger-ready output for GL posting and management reporting.
Inputs & outputs: Inputs: Staff (attributes), SC_Allocation (FTE %), Payroll Register (Gaji, Tunjangan, BPJS, PPh 21). Outputs: Granular cost fact table (Grain: Period + Staff Code + Approval code).
Primary invariants (must/shall):
1. Grain Strictness: Exactly one row per unique combination of Period, Employee, and Approval code.
2. Mathematical Integrity: Total staff cost MUST exactly equal the sum of all gross, statutory, and allowance components, multiplied by the FTE in Month, minus payable offsets.
3. Conservation of Cost: The sum of Total staff cost for an employee across all split allocation rows in a given Period MUST equal the employee's total un-split payroll cost for that period (zero-variance tolerance).
4. Liability Separation: Payables (PPh 21 Payable, BPJS Payable) MUST be distinctly separated from Employer Costs to ensure accurate GL posting and net-pay reconciliation.
Observability & audit fields: Reconciliation hashes per period. Emit costing.variance if split-sum != total payroll. Append costing.posted audit when exported to ERP.
Failure modes & recovery: FTE mismatch → halt GL posting. Missing payroll component → flag as incomplete_costing. Orphaned approval codes → route to suspense account.
Implementation notes & safe I/O: Pre-calculate un-split totals in a hidden staging area, then apply FTE in Month multipliers. Use explicit rounding rules (e.g., Banker's rounding) to prevent penny-off discrepancies across split rows. Never hardcode period dates; derive them from the Period column.
Tests & validation rules: Conservation of cost tests (Split Sum == Total). GL reconciliation tests. Component summation accuracy. FTE boundary checks (0 < FTE <= 1).
Cross-Sheet Dependency Chain & Pipeline Orchestration — data flow, join keys, and cascading failure modes
Purpose & contract: Defines the ETL-like data flow and referential integrity constraints across the three-sheet architecture. Ensures that updates to master data or routing logic propagate deterministically to the fact table without silent data loss.
Dependency Graph: Staff (Dimension) 1:N -> SC_Allocation (Bridge/Mapping) 1:N -> SCA_Data (Fact).
Primary invariants (must/shall):
1. Surrogate Key Propagation: All joins between sheets MUST utilize Staff Code (or a canonical hash of Name if codes are unavailable). Direct string matching on Name is forbidden for computational joins due to mutation risk.
2. Cascading Invalidation: If a Staff Code is deactivated in the master sheet, all corresponding future-month allocations in SC_Allocation MUST be flagged or zeroed out; historical SCA_Data MUST remain immutable.
3. Temporal Alignment: The Period column in SCA_Data (e.g., 202605) MUST deterministically map to the corresponding monthly column in SC_Allocation (e.g., MAY).
Observability & audit fields: Emit pipeline.orphan when SCA_Data contains an employee not found in Staff. Emit pipeline.unmapped when SC_Allocation contains an employee not in Staff.
Failure modes & recovery: Name mismatch causing dropped records → implement a fuzzy-match staging view for operator review before hard-joining. Period misalignment → enforce strict YYYYMM formatting and validation tables.
Implementation notes: Use Power Query (M) or structured SQL-like logic within the workbook to enforce join types (Inner vs. Left). Avoid volatile functions (INDIRECT, OFFSET) in the dependency chain to prevent calculation chain degradation.
Tests & validation rules: Referential integrity checks (Foreign Key constraints). Orphaned record detection. Period mapping validation.
Data Quality, Validation & Exception Handling (Module-level summary) — #N/A handling, split syntax parsing, and blank field semantics
Audit obligations: Every structural exception (unmapped D365 codes, split syntax errors, missing Tax IDs) must be surfaced in a dedicated Exceptions or Control dashboard sheet. Silent failures (e.g., #N/A resolving to 0 in a SUMIF) are strictly prohibited.
PII & evidence handling: Tax ID Number and Name in BNI (Bank details) are classified as PII/Sensitive. They MUST NOT be used as join keys in plain text if the workbook is shared broadly. Masking or tokenization should be applied for non-HR users.
Blank Field Semantics:
1. Mandatory Blanks: Blank Staff Code or Tax ID = Critical Failure (master.data.invalid).
2. Optional Blanks: Blank Notes Sisca = Normal state.
3. Allocation Blanks: Blank month in SC_Allocation = Must be interpreted via explicit policy (e.g., "Inherit Prior Month" or "Zero Allocation"). Ambiguity is forbidden.
Security & secrets policy: If the workbook pulls data via API or database queries, connection strings and credentials MUST NOT be stored in named ranges or visible cells. Use secure credential managers or OAuth flows.
Determinism & reproducibility: Cost calculations must be idempotent. Running the calculation engine twice on the same input data must yield bit-for-bit identical SCA_Data outputs. Rounding residuals in split allocations must be absorbed deterministically (e.g., assigned to the largest allocation slice).
Tests & CI rules: Static analysis of formulas to detect implicit intersections, unhandled #N/A errors, and volatile function abuse.
Security, PII & Governance Controls — Tax ID masking, bank data protection, and maker-checker workflows
PII Classification Matrix: Tax ID Number, Name in BNI, and Marital Status are classified as Restricted PII. Staff Code and Approval Codes are Internal Confidential. Total staff cost is Financial Confidential.
Access & Masking: Workbooks distributed to operational managers must have Restricted PII columns hidden or masked (e.g., showing only last 4 digits of Tax ID). Full PII visibility is restricted to HR/Payroll administrators.
Maker-Checker Workflows: Changes to SC_Allocation splits (especially mid-year reallocations) require a documented approval trail. The workbook should ideally integrate with a workflow tool or maintain a hidden Allocation_Change_Log sheet that records Timestamp, Operator, Old Value, and New Value via VBA or Office Scripts.
Statutory Compliance: PPh 21 and BPJS calculations must strictly adhere to current tax year regulations. Any update to tax brackets or statutory rates must be version-controlled in a hidden Parameters sheet, with the effective date explicitly tracked to prevent retroactive miscalculations.
Operator Runbook & Forensic Triage — troubleshooting missing costs, fixing D365 mappings, and resolving split mismatches
Common faults & mitigations:
• D365 #N/A Epidemic: Action: Export Staff sheet, cross-reference with D365 active directory dump, update mappings, and refresh. Do not manually override #N/A in downstream sheets.
• Split Allocation != 100%: Action: Filter SC_Allocation for sum errors. Identify if it's a syntax error (e.g., missing %) or a logical error (e.g., 90% + 20%). Correct at the source; do not normalize in SCA_Data.
• Conservation of Cost Failure (Penny Offs): Action: Check rounding logic in SCA_Data. Ensure Banker's rounding is applied after FTE multiplication, not before. Assign residual pennies to the primary cost center.
• Orphaned Cost Lines: Action: An employee appears in SCA_Data but not Staff. Check for trailing spaces in Name or Staff Code. Implement TRIM() and CLEAN() in the join staging area.
Operator triage checklist: 1. Check Exceptions dashboard. 2. Verify SC_Allocation 100% sums. 3. Run Conservation of Cost pivot (Total Payroll vs Sum of Allocated Cost). 4. Verify D365 mapping completeness.
Forensic artifacts to collect: Snapshot of Staff at period-end, version-controlled SC_Allocation matrix, raw payroll register input, and the final SCA_Data export. Compute SHA256 hashes of all CSV exports for chain-of-custody.
Audit, Archival & Period-End Close Procedures — versioning, snapshotting, and regulatory evidence packaging
Purpose & contract: Define the lifecycle of the workbook data from active processing to immutable archival. Ensures regulatory compliance for tax (PPh 21) and statutory (BPJS) reporting.
Period-End Close Sequence:
1. Freeze Master Data: Lock Staff sheet to prevent retroactive changes to tax status or ERP mappings for the closed period.
2. Snapshot Allocations: Copy SC_Allocation values (Paste Special -> Values) to a historical archive sheet (e.g., SC_Alloc_202605).
3. Finalize Fact Table: Lock SCA_Data. Generate control totals (Total Gaji, Total BPJS, Total PPh 21) and record them in the Control sheet.
4. Export & Hash: Export SCA_Data to CSV/Parquet for GL upload. Compute and log the file hash.
Retention & archival: Raw payroll inputs and final SCA_Data must be retained for the statutory period (e.g., 7 years for tax). SC_Allocation matrices must be versioned to explain historical cost distributions during audits.
Regulator workflows: In the event of a tax audit, the operator must be able to produce the Staff master (for tax status), the SC_Allocation (for project costing justification), and SCA_Data (for the mathematical proof of PPh 21 calculations) as a unified, mathematically consistent evidence package.
Error Code Catalog & Stable Exception Mapping — standardizing workbook errors for operational triage
MASTER_001 — Duplicate Identity: Staff Code or Tax ID appears more than once. Triage: Merge records or deactivate the obsolete one.
MASTER_002 — Missing Statutory Driver: Marital Status or Gross Up flag is blank. Triage: Halt costing; request HR update.
ERP_001 — D365 Mapping Failure: D365 Staff Code resolves to #N/A. Triage: Flag for ERP sync; route cost to default suspense account if policy allows.
ALLOC_001 — Split Sum Violation: Monthly allocation percentages do not sum to 100%. Triage: Reject row; require manual correction in SC_Allocation.
ALLOC_002 — Unparsable Split Syntax: Cell contains free-text instead of CODE \| %; CODE \| % format. Triage: Standardize string format.
COST_001 — Conservation of Cost Failure: Sum of allocated costs != Total Payroll Cost. Triage: Investigate rounding logic or missing FTE multipliers.
COST_002 — Orphaned Approval Code: SCA_Data references an approval code not in the master registry. Triage: Map to valid code or route to overhead.
UI/UX Mapping: Ensure all #REF!, #NAME?, or #VALUE! errors are caught by wrapper formulas (e.g., IFERROR(..., "ERR_MASTER_00X")) and surfaced in the Control Dashboard, never hidden in raw data columns.
Implementation & Maintenance Guidance — safe update patterns, formula protection, and structural integrity
1. Structured References: Always use Excel Tables (ListObjects) and structured references (e.g., Staff[Staff Code]) instead of A1:B100 ranges. This ensures dynamic expansion and prevents silent truncation when new staff are added.
2. Formula Protection: Lock all formula cells in SC_Allocation and SCA_Data. Only allow data entry in designated input columns (e.g., manual split overrides). Use sheet protection with hidden formulas for proprietary tax logic.
3. Staging Areas: Never join Staff directly to SCA_Data in a single massive formula. Use a hidden Staging sheet to perform the Staff + Payroll Register merge first, then apply the SC_Allocation FTE multipliers. This isolates calculation errors and improves performance.
4. Volatile Function Ban: Prohibit INDIRECT, OFFSET, TODAY, and NOW in the core calculation chain. Use INDEX for dynamic ranges and store the Period date as a static value in a control cell.
5. Error Trapping: Wrap all cross-sheet lookups in IFERROR or IFNA with explicit, trackable error codes (e.g., IFERROR(VLOOKUP(...), "ERR_ERP_001")). Never allow raw #N/A to propagate into summation columns.
6. Performance Optimization: For large workbooks (>10k rows), move the allocation splitting logic to Power Query (M) or Python in Excel. Native Excel array formulas handling 12-month splits across thousands of employees will cause severe calculation lag and memory bloat.
