Prompt_01 — Single-column Workflow
1. Purpose

Convert structured or unstructured text into a verified two-column table, then immediately produce a single-column Markdown layout with each left-column header as a bolded section header.
2. Input types

Accepts pasted text, document, or webpage content. If the source is a webpage, follow the extraction and copyright steps defined below.
3. Extraction phase (Phase 1)

Generate a strict two-column table:
1. Left column in bold.
2. Right column plain text.
3. Use <br> for numbering, sub-bullets, and multi-line points.
4. Maintain factual and numerical fidelity.
5. Perform 10× validation (structure, encoding, punctuation, and completeness).
4. Merge phase (Phase 2)

Immediately transform the verified table into a single-column Markdown layout, formatted so each left-column header becomes the heading cell followed by its full right-column content.
5. Output structure

Final Markdown table must match the style:

``markdown<br>| **Header** |<br>| ---------- |<br>| Full right-column text |<br>``

Each header–content pair occupies one cell in a single-column table.
6. Formatting rules

• Use Markdown tables.
• Bold left headers.
• Use <br> for line breaks and numbering.
• Preserve inline code, backticks, and HTML entities.
• Maintain UTF-8 encoding.
• Do not alter factual data.
• Remove all extra comments or metadata unless explicitly requested.
7. Verification (10× checks)

1. Source matches input.
2. Row count equals source.
3. Headers bolded correctly.
4. Right-column content preserved verbatim.
5. Encoding = UTF-8.
6. No lost HTML entities.
7. Markdown renders correctly.
8. Table alignment correct.
9. Line breaks valid.
10. Overall integrity confirmed.
8. Final output

Always deliver the merged single-column Markdown table as the default result.
No separate “approve for merge” step required.
Prepend: Checked 10 times.
9. Error handling

If any extraction or formatting fails, return the failed step number and short diagnostic. Do not proceed until resolved.

Prompt_02 – World-Class Multi-File Extraction & Two-Column Summary (Coding Project Edition)
Description: Workflow for reviewing multiple .py files via Dropbox direct links, chunked if necessary, summarized into a two-column Markdown table with ≥80% verbatim fidelity in the code column.
Purpose

Transform multiple .py source files into a high-fidelity, per-file two-column summary. Left column = extracted logic/summary. Right column = supporting details/verbatim snippets. Iterative approval ensures accuracy and preserves audit trail.
1️⃣ Inputs

- Source .py files via Dropbox direct links (max 50 links per batch).
- Each file up to ~15,000 characters per chunk (auto-split if larger).
- Optional: file-level metadata (path, purpose, module role).
2️⃣ Processing Steps

1. Silent Absorption – Load all .py content from links.
2. File Identification – Detect file names and module structure from paths.
3. High-Fidelity Extraction – Preserve ≥80% verbatim code text per file.
4. Two-Column Summarization – Left: concise logic/structure summary.
    Right: verbatim snippets, key functions, imports.
5. User Approval Workflow – Confirm after each file before moving to next.
6. Optional Metadata – Include file name, directory, notes.
7. Iteration – Process all files sequentially.
8. Optional Notes/Analysis – Auto-generate insights (dependencies, patterns).
3️⃣ Output

- Markdown table with two columns: Logic / Structure and Verbatim Code / Key Functions.
- Optional metadata row per file.
- Workflow-ready format for audit or refactoring.
4️⃣ Key Features

- Verbatim Preservation – ≥80% code fidelity.
- Approval Loop – User validates each file step.
- Two-Column Output – Clear logic vs code details split.
- Multi-File Support – Batch up to 50 Dropbox links.
- Metadata Integration – Track file source and notes.
- Optional Code Insights – Dependencies, helpers, patterns.
5️⃣ Workflow Guidelines

1. Paste Dropbox links to .py files.
2. Absorb content silently.
3. Extract and display the first file in two-column format.
4. Get approval before proceeding.
5. Continue until all files are processed.
6. Generate a consolidated two-column summary table.
7. Optionally append analysis (cross-file helpers, common utils).
6️⃣ Best Practices

- Always use Dropbox direct download links for .py files (?raw=1).
- Ensure consistent file naming in project directories.
- Use silent absorption for large batches.
- Maintain approval workflow to ensure accuracy.
- Keep summaries Markdown-compatible and exportable (Excel, research DB).
✅ Outcome

A professional, audit-ready, file-by-file two-column summary of your Python project. Preserves verbatim code snippets, clarifies logic structure, integrates metadata, and supports systematic approval.

Chapter Identification – Detect or confirm chapter titles and order.

Core Extraction – Identify key arguments, story beats, or themes.

Two-Column Summarization –
1. Left: distilled logic, concept, or event flow.
2. Right: key supporting quotes or factual detail.

User Approval Workflow – Confirm each chapter’s summary before next.

Optional Metadata Integration – Author, edition, thematic category.

Sequential Iteration – Continue through all chapters.

Optional Analysis – Thematic map, tone shifts, recurring motifs.
3️⃣ Output

• Markdown table with two columns per chapter:
• Column 1: Logic / Core Ideas
• Column 2: Supporting Quotes / Key Details
• One summary per chapter.
• Optional metadata header (title, chapter number, page range).
• Export-ready for research, notes, or teaching materials.
4️⃣ Key Features

• Two-Column Clarity – Ideas and quotes side by side.
• Approval Loop – User validates each summary.
• High-Fidelity Quote Retention – Preserve up to 50% verbatim phrasing.
• Multi-Chapter Handling – Batch process full books or partials.
• Metadata Awareness – Integrate book details seamlessly.
• Optional Thematic Analysis – Identify tone, pattern, structure shifts.

1. Paste or link chapter texts (Dropbox ?raw=1 or direct text).
2. Absorb content silently.
3. Present Chapter 1 in a two-column Markdown table.
4. Await approval before Chapter 2.
5. Continue sequentially through all chapters.
6. After final approval, generate full consolidated summary.
7. Optionally append theme or character analysis table.
6️⃣ Best Practices

• Use direct text or raw file links for clean ingestion.
• Keep chapter titles consistent (e.g., Ch1_Intro.txt).
• Maintain approval after each summary for accuracy.
• Use Markdown output for portability across note systems.
• Fiction: highlight character and plot transitions.
• Nonfiction: focus on argument chains.
✅ Outcome

A professional, structured, chapter-by-chapter two-column book summary. Captures logic, narrative, or argument flow alongside key supporting quotes. Suitable for research, study, curation, or AI-assisted literature review.

Prompt_06 — Conceptual Absorption
OVERVIEW

This workflow allows a user to share an entire coding project (Python, JS, or mixed) with the assistant in a safe, structured, and efficient way. Focus is on conceptual understanding rather than verbatim code echoing.
Key goals

- Efficient absorption of large multi-file projects.
- Conceptual mapping of each file (purpose, dependencies, control flow).
- Preserve function names, signatures, and modules without repeating full code.
- Guarantee approval after each file before moving on.
- Support upgrades, audits, and deep conceptual expansion.
USER GUIDE: STEP-BY-STEP

STEP 1 — PREPARE FILES
- Identify files to share (example: script.js).
- If large (>15,000 characters), split into chunks.
- Keep filenames clear and ordered (script.part01.js).
STEP 2 — CREATE DROPBOX DIRECT LINKS

- Upload each file or chunk to Dropbox.
- Use links ending in ?raw=1 for clean content streaming.
- Example: https:script.part01.js?raw=1
STEP 3 — SHARE BATCHES

- Up to 50 files per batch.
- ≤750,000 characters total per batch.
- Paste all links with a header:
PROMPT06 BATCH: script.js (5 parts)
STEP 4 — FILE-BY-FILE WORKFLOW — For each file:

1. Assistant absorbs the file silently.
2. Assistant produces a conceptual summary that represents a two-column conceptual table (Logic/Structure → Conceptual Notes) but returns it inside this single-cell output.
    Format example inside the cell:
    Logic/Structure: brief prose describing purpose, control flow, key functions.
    Conceptual Notes: design rationale, dependencies, edge cases, upgrade points.
3. Assistant pauses and asks: Do you approve this summary? before proceeding to the next file.
STEP 5 — CONSOLIDATION

After all files approved, assistant can:
- Generate consolidated conceptual map of project.
- Expand into long-form (~10k–15k chars per section).
- Suggest upgrades, optimizations, or audits.
STEP 6 — OPTIONAL MODES

- Analysis Mode → Two-column conceptual table only.
- Conceptual Expansion Mode → Detailed prose explanations.
- Upgrade Mode → Apply Prompt_01 rules (safe single-file upgrades).
- Audit Mode → Portability, performance, UX, and security notes.
ASSISTANT RULES (INTERNAL)

- Always absorb full code silently.
- Never echo full code unless explicitly requested.
- Right column = conceptual notes, not raw code.
- Always pause for user approval before next file.
- Maintain awareness of active code paths (avoid analyzing inactive modules).
- For upgrades, follow Prompt_01 (single-file, stepwise, approved).
EXAMPLE

User: Here is my first file: https:script.js?raw=1

Assistant conceptual output:

| Logic/Structure | Conceptual Notes |
| ----------------- | ------------------ |
| Initializes global config, sets up event listeners, defines UI rendering functions. | Uses modular IIFE pattern for scope isolation; central event bus handles state updates; DOM manipulations use querySelectorAll. |

Assistant: "Do you approve this summary?"
TROUBLESHOOTING

- File too large (>15k chars): Split into chunks.
- Preview page instead of raw file: Ensure ?raw=1 at the end.
- Assistant shows low fidelity: Provide declared character count and optional checksum.
- Need more depth: Switch to Conceptual Expansion Mode.
CONCLUSION

This refined Prompt_06 maximizes efficiency:
- Files absorbed silently.
- Conceptual tables ensure project-level understanding.
- No wasted output space on code repetition.
- Guarantees approval checkpoints and safe upgrades.

Prompt_07 — Comprehensive Multi-File Conceptual Absorption & Mapping Workflow
OVERVIEW

Prompt_07 is a high-fidelity workflow for absorbing, mapping, and analyzing multi-file projects (Python/JS/mixed). Focus on interlinked conceptual mapping, dependency tracking, concurrency understanding, and fallback handling. Emphasize conceptual structures not verbatim code.
PURPOSE

- Silently absorb multiple project files.
- Map concepts, control flow, dependencies, and data transformations.
- Handle optional imports, fallbacks, and error paths.
- Build interlinked module understanding and support concurrent workflows.
- Produce per-file conceptual summaries with approval checkpoints.
- Preserve function signatures and critical behaviors; highlight edge cases and optional features.
INPUTS

1. Project files (Python, JS, mixed), ≤750,000 chars per batch.
2. Up to 50 files per batch.
3. Split large files (>15k chars) into logical chunks.
4. Use clear filenames (core_fragments.py, core_table.py, etc.).
5. Prefer raw links (Dropbox ?raw=1).

Example header: PROMPT07 BATCH: core_fragments.py, core_table.py, io_utils.py (3 files)
WORKFLOW — STEP 1: SILENT ABSORPTION

1. Read each file silently.
2. Detect public API (functions/classes).
3. Detect private helpers (_underscore), optional dependencies (try/except imports), fallback paths, concurrency patterns (ThreadPoolExecutor, async).
4. Identify input/output formats (DataFrame, list-of-lists, Markdown, HTML) and core transformations.
5. Track inter-module calls and normalization steps.
WORKFLOW — STEP 2: CONCEPTUAL MAPPING

1. Produce a two-column conceptual table conceptually, but output as single-cell content: Logic/Structure and Conceptual Notes.
2. Include input types, output structures, normalization/sanitization, special handling (escaped pipes, fenced code, RTL), fallback paths, concurrency and atomic write mechanisms, metadata extraction.
3. Note edge cases: empty inputs, malformed Markdown, inline code with pipes, diagram-like text, Unicode normalization.
WORKFLOW — STEP 3: INTER-FILE LINKAGE

1. Track which file calls which function in other modules.
2. Distinguish optional helpers vs mandatory imports.
3. Map data flow: Input → Normalization → Extraction → Rendering → Fragment output.
4. Example flows: Markdown→HTML→Sanitized Table; DataFrame→HTML Table→File output.
WORKFLOW — STEP 4: USER APPROVAL CHECKPOINTS

1. After each file present the conceptual summary.
2. Ask: "Do you approve this conceptual summary?"
3. Wait for confirmation before next file.
4. If rejected accept feedback, update summary, re-present for approval.
WORKFLOW — STEP 5: CONSOLIDATED PROJECT MAP

1. After approvals generate full interlinked conceptual map.
2. Highlight core pipelines (rendering pipeline, fragment management, table extraction).
3. Include per-module purpose, public API, optional dependencies, fallback strategies, and edge-case handling.
WORKFLOW — STEP 6: OPTIONAL MODES

1. Analysis Mode → Two-column conceptual table only.
2. Expansion Mode → Long-form explanations (≥500 words/module).
3. Upgrade Mode → Safe single-file refactor suggestions (Prompt_01 rules).
4. Audit Mode → Security, performance, UX, portability notes.
5. Dependency Map Mode → Visual dependency graph generation.
WORKFLOW — STEP 7: HANDLING LARGE FILES & EDGE CASES

1. If >15k chars split into logical chunks and track chunk order.
2. Treat non-Python modules conceptually (JS, JSON, YAML).
3. Always log optional imports, try/except fallbacks, potential failure points, concurrency concerns.
OUTPUT STRUCTURE

1. Per-file conceptual table (Markdown, conceptual two-column content presented as single-cell).
2. Consolidated interlinked map for all files.
3. Edge-case log (warnings, optional modules).
4. Metadata per table/function (table ID, row count, section headers).
5. Total row counts for fragment-rendering modules where applicable.
BEST PRACTICES

1. Preserve function names and signatures; do not echo full code unless requested.
2. Mark optional helpers explicitly.
3. Identify control-flow forks (try/except, fallback renderers).
4. Map concurrent jobs conceptually (ThreadPoolExecutor, async patterns).
5. Treat normalization, Markdown parsing, and HTML sanitization as critical.
6. Respect placeholder stashing and restoration mechanisms.
USER INTERACTION

1. After each conceptual summary provide clear approval instruction.
2. Allow stepwise adjustments: add notes, correct misinterpretations, expand edge cases.
3. After approval move to next file automatically.
OUTCOME

- Audit-ready conceptual understanding of the entire project.
- Interlinked data and control-flow maps.
- Clear knowledge of I/O handling, normalization, sanitization, fallback mechanisms, concurrency, optional dependencies, and edge-case handling.
- Ready for refactoring, documentation, upgrade suggestions, dependency analysis, and research presentation.
EXAMPLE USAGE

1. User: PROMPT07 BATCH: core_fragments.py, core_table.py, io_utils.py, convert.py, sanitize.py.
2. Assistant silently absorbs core_fragments.py.
3. Assistant presents conceptual summary (Logic/Structure + Conceptual Notes) in single-cell format.
4. Assistant asks: "Do you approve this conceptual summary?"
5. On approval move to next file; after all files produce full interlinked project map.
KEY FEATURES

- Conceptual absorption without code echoing.
- Per-file detailed mapping with approval checkpoints.
- Fallback and optional module detection.
- Concurrency and atomic I/O mapping.
- Markdown/HTML table handling and inline/fenced code protection.
- Edge-case logging and dependency graph support.
COPY-BUTTON READY

- Integrates with ChatGPT workflows.
- Handles large multi-file projects with ≥80% fidelity to architecture and behavior.
- Produces Markdown tables compatible with Excel, HTML, or database ingestion.

Preserve public API and signatures. One-file safe commits. Strong sanitization by default. Exports remain plain-text unless HTML-export is explicitly added.
GOAL

Ensure robust preservation and safe rendering of diverse inline and block content inside table cells while keeping the existing public API (render_table, render_html, render_page) unchanged.
CONSTRAINTS

- Preserve public API and function signatures.
- One-file safe commits per workflow.
- Strong sanitization by default (no XSS regressions).
- Exports (CSV/JSON/XLSX) remain plain-text unless HTML export added behind an opt-in.
HIGH-LEVEL PHASES

1. Safety + baseline CSS
2. Format extensions (math, inline-block)
3. Renderer fixes (preserve inner HTML)
4. Optional rich media (gated)
5. Exports, tests, CI, release
STEPWISE STEPS (single-file)

1) Allow safe inline wrappers — render/sanitize.py.
2) Diagram & math CSS baseline — assets/style.css (append-only).
3) Preserve inline HTML in convert pipeline — render/convert.py.
4) Math wrappers in markdown renderer — render/markdown.py.
5) Preserve inner HTML when extracting tables — render/core_renderer.py.
6) Use sanitizer flags when rendering cells — render/core_table.py.
7) Worker/search: preserve code/math tokens — assets/worker.js.

9) Opt-in rich media gate — render/sanitize.py (allow_rich_media flag).
10) Exports server-side (optional) — core_renderer.py or small server script.
11) Tests, CI, accessibility checks — tests/, CI config.
12) Release & rollout — PR, feature flags.
VERIFICATION CHECKLIST (summary of key checks)

- Sanitizer robustness (strip script, on* attributes).
- <pre>/<code> preserved for blocks and inline code.
- .cell-diagram CSS present and applied.
- .cell-math wrapper present and KaTeX opt-in available.

- Search highlighting restores original HTML safely.

- Accessibility basics covered (semantic roles, focusable elements where required).

DELIVERABLES (per-phase)
