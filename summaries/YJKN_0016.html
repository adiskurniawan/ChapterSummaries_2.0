Introduction
BPJS Kesehatan — langkah penting (untuk pemberi kerja & karyawan)
- Persiapan dokumen (pemberi kerja): akta pendirian/NPWP perusahaan, NIB/SIUP/izin usaha, KTP penanggung jawab, surat domisili (jika diminta).
- Persiapan karyawan: fotokopi e-KTP, Kartu Keluarga, kartu keluarga/akta lahir, paspor untuk WNA.
- Daftarkan badan usaha / pemberi kerja di portal BPJS Kesehatan (pendaftaran badan usaha → setuju syarat & ketentuan). Setelah akun aktif, lakukan entri peserta: manual melalui aplikasi new e-DABU atau upload formulir 34 kolom (pendataan massal).
- Verifikasi & approval: BPJS memverifikasi data perusahaan dan peserta; pemberi kerja harus memastikan data valid sebelum submit. Setelah disetujui, peserta terdaftar dan mendapatkan nomor peserta.
- Penetapan kelas & perhitungan iuran: tentukan kelas peserta (I/II/III) sesuai ketentuan; hitung iuran karyawan dan pemberi kerja (kewajiban pemotongan dan kontribusi perusahaan). (Periksa tarif iuran terbaru pada sumber resmi sebelum bayar.)
- Pembayaran iuran: lakukan pembayaran bulanan melalui Virtual Account atau kanal pembayaran resmi BPJS Kesehatan; simpan bukti bayar untuk rekonsiliasi.
- Mutasi / penambahan / pengurangan peserta: laporkan karyawan baru, resign, atau mutasi (mutasi antar fasilitas/kota) melalui e-DABU atau menu perubahan data.
- Klaim pelayanan & rujukan: peserta menggunakan faskes tingkat pertama; rujukan ke rumah sakit mengikuti mekanisme rujukan dan sistem klaim INA-CBG di rumah sakit. (Pemberi kerja tidak mengurus klaim medis langsung kecuali untuk verifikasi administrasi peserta.)
- Sanksi & kepatuhan: pastikan laporan dan pembayaran tepat waktu untuk menghindari denda atau sanksi administratif.
BPJS Ketenagakerjaan — langkah penting (untuk pemberi kerja & karyawan)
- Persiapan dokumen (pemberi kerja): formulir pendaftaran pemberi kerja, NPWP perusahaan, NPP (jika sudah), KTP pemilik/pengelola, dokumen karyawan (KTP, KK).
- Daftar pemberi kerja & buat akun SIPP / portal BPJamsostek: daftar perusahaan (NPP) → buat user login di SIPP/ES BPJamsostek (sistem pendaftaran online untuk pemberi kerja dan entri data pekerja). Alternatif ada pendaftaran BPU untuk bukan penerima upah.
- Entri data pekerja: isi data tiap pekerja (upah, jabatan, NIK, jenis perlindungan yang diikutkan: JHT, JKK, JKM, JP, JKP). Upload formulir laporan iuran (laporan rinci) jika diperlukan.
- Penentuan iuran & potongan: hitung iuran sesuai persentase untuk program (mis. JHT, JKK, JP, JKP) — potongan/pemotongan gaji karyawan dan kontribusi pemberi kerja sesuai aturan terbaru. Selalu verifikasi tarif iuran terbaru pada sumber resmi.
- Pembayaran iuran & pelaporan bulanan: bayar iuran melalui kanal pembayaran resmi (VA/korporat) dan laporkan iuran bulanan melalui SIPP/ES. Simpan bukti pembayaran untuk audit.
- Klaim manfaat (JHT, JKK, JKM, JP, JKP): prosedur klaim berbeda per program — umumnya: persiapkan dokumen peserta (kartu peserta, KTP, KK, surat keterangan kerja/PHK), ajukan klaim melalui kantor cabang BPJamsostek atau aplikasi resmi (mis. JMO/JHT online bila tersedia). Untuk klaim pensiun/benefit berkala ada persyaratan tambahan.
- Mutasi/terminasi peserta: laporkan pemutusan kerja/terminasi sehingga status peserta diperbarui; untuk pekerja keluar pastikan data keluar (terminate) agar perhitungan iuran dan hak (JHT) dapat diproses.
- Kepatuhan & audit: sediakan dokumentasi lengkap; responif terhadap permintaan data dari auditor atau BPJS untuk menghindari sanksi atau denda.
Quick checklist operasional (ringkas)
- Siapkan dokumen perusahaan + dokumen karyawan.
- Daftarkan perusahaan → buat akun admin di portal masing-masing (e-DABU untuk Kesehatan; SIPP/ES untuk Ketenagakerjaan).
- Entri/upload data peserta (form 34 kolom atau laporan rinci).
- Hitung & bayarkan iuran bulanan melalui VA/kanal resmi; simpan bukti.
- Lakukan mutasi/terminate saat ada perubahan status; bantu karyawan proses klaim saat diperlukan.

Tujuan ringkasan:
Bagi HRD perusahaan yang ingin memastikan seluruh data pekerja telah sesuai untuk menerima BSU, menjelaskan tahapan pembaruan data di SIPP.
Ringkasan singkat proses:
1. Login ke situs resmi SIPP menggunakan akun perusahaan.
2. Pilih menu “BSU Tahun 2025”.
3. Akses sub-menu “Pengkinian Data BSU”.
4. Unduh template Excel yang tersedia.
5. Lengkapi informasi sesuai kolom yang diminta.
6. Upload kembali file Excel yang telah diisi ke dalam sistem.
7. Tunggu proses verifikasi dan persetujuan data dari sistem.
Peran dan akses yang diperlukan:
HRD / PIC perusahaan dengan akun perusahaan terdaftar dan hak untuk mengakses menu BSU/Pengkinian Data.
Template & format file:
Gunakan template Excel yang tersedia pada laman unduhan SIPP; isi sesuai kolom yang diminta oleh template.
Prasyarat sebelum update:
Akun perusahaan aktif dan terdaftar di SIPP; akses ke menu BSU Tahun 2025; file Excel terisi lengkap sesuai template.
Proses verifikasi:
Setelah upload, sistem melakukan verifikasi — menunggu persetujuan/konfirmasi dari sistem sebelum perubahan dianggap final.
Catatan penting:
Di laman unduhan tersedia panduan pengisian untuk menghindari kesalahan teknis saat input data; ikuti panduan tersebut sebelum upload.

IMPORT DATA
Tujuan ringkasan:
menjelaskan langkah, format, validasi, risiko, troubleshooting, dan best-practice untuk unggah/impor data peserta perusahaan ke sistem BPJS (penerapan umum untuk kedua layanan).
Ringkasan singkat proses:
1) persiapkan data sesuai template resmi
2) validasi & bersihkan (duplikat, NIK, tanggal)
3) uji di sandbox atau subset
4) unggah massal via portal resmi
5) cek hasil / perbaiki eror
6) monitoring & audit.
Peran dan akses yang diperlukan:
Akun admin perusahaan / PIC yang tercatat, hak untuk unggah massal, akses ke laporan hasil impor, dan otorisasi untuk mengoreksi data.
Prasyarat sebelum impor:
Akun perusahaan aktif; nomor pemberi kerja/registrasi (NPP/NIK perusahaan sesuai portal); akses ke template resmi; koneksi aman (TLS/HTTPS); backup data lama.
Template & format file:
Gunakan template Excel/CSV resmi dari portal; jangan ubah header; simpan dalam encoding UTF-8 (untuk CSV); jaga konsistensi format tanggal.
Nama file & versi:
Gunakan pola nama yang jelas, contoh: IMPOR_BPJS_.xlsx. Simpan salinan arsip sebelum unggah.
Kolom wajib (umum, per baris):
NIK; Nama Lengkap; Tgl Lahir; Jenis Kelamin; Tgl Mulai Kerja/Keaktifan; Status Karyawan (aktif/cuti/keluar); Upah (jika untuk ketenagakerjaan); Jabatan/Unit; Alamat; No. Telepon/email PIC.
Kolom sering diminta khusus BPJS Kesehatan:
Kelas perawatan; jenis peserta (pekerja utama/anggota); FKTP rujukan awal (jika ada).
Kolom sering diminta khusus BPJS Ketenagakerjaan:
Gaji/upah bulanan; kode cabang/UP; jenis penerima manfaat; tanggal PHK (jika keluar).
Format NIK:
16 digit numerik — pastikan tanpa spasi, tanpa tanda baca; verifikasi checksum dasar (panjang + numerik).
Format tanggal:
Tentukan format yang konsisten (disarankan YYYY-MM-DD) dan pastikan portal menyetujui format tersebut; beberapa portal menerima DD/MM/YYYY.
Validasi nama:
Hilangkan karakter kontrol; batasi panjang (mis. ≤100 char); cek karakter ilegal (emoji/karakter non unicode standar).
Validasi duplikat:
Deteksi NIK duplikat dan karyawan ganda; tandai untuk review manual sebelum unggah massal.
Validasi upah:
Rentang logis (tidak negatif, dalam mata uang IDR); jika ada aturan kelas upah per threshold — laporkan nilai outlier >3× rata-rata unit.
Mapping kolom:
Siapkan peta kolom (sheet terpisah) yang menunjukkan kolom Excel → field portal; susun dokumentasi mapping untuk tim IT.
Field optional vs mandatory:
Pisahkan kolom opsional (mis. NPWP, alamat lengkap) dari wajib; kosongkan opsional bila tidak tersedia.
Penanganan baris error:
Portal biasanya mengembalikan file laporan error; buat proses otomatis untuk mem-parsing laporan dan menandai baris yang gagal.
Contoh entri sampel (satu baris):
NIK=1234567890123456 | Nama=Budi Santoso | TglLahir=1985-05-20 | JK=L | TglMulai=2020-01-15 | Upah=7500000 | Kelas=III | Unit=Finance
Batas ukuran & jumlah baris:
Periksa batas portal (mis. max rows per upload / max filesize); jika besar → bagi menjadi batch lebih kecil.
Preflight checks (otomatis):
Cek header cocok template; jumlah kolom benar; semua mandatory terisi; NIK numerik 16 digit; tanggal valid; tidak ada duplikat internal.
Backup & rollback:
Sebelum unggah, simpan snapshot data peserta saat ini; jika import merusak, gunakan prosedur rollback/restore atau minta portal melakukan revert jika tersedia.
Uji sandbox:
Lakukan unggah percobaan pada subset data di environment uji (jika tersedia) untuk memeriksa pemetaan dan eror.
Logging & audit trail:
Simpan log upload (user, timestamp, file name, jumlah baris sukses/gagal, id job dari portal) untuk audit internal.
Komunikasi & SOP internal:
Tetapkan checklist persiapan, penanggung jawab koreksi, SLA untuk menindaklanjuti eror, dan saluran eskalasi ke pihak berwenang.
Keamanan & perlindungan data:
Enkripsi file sensitif saat transit (HTTPS) dan at-rest; batasi akses ke file mentah; hapus file temporer setelah sukses.
Privasi data:
Patuhi aturan perlindungan data karyawan (pembagian NIK/identitas cukup untuk tujuan administratif).
Notifikasi & konfirmasi:
Setelah upload, verifikasi status job (success/partial/failed) dan kirim ringkasan ke stakeholder (HR, payroll, compliance).
Penanganan anggota yang sudah terdaftar:
Jika NIK sudah terdaftar dengan nomor peserta, putuskan apakah melakukan update atau skip — siapkan rule prioritas (update if newer).
Merge vs Replace:
Tentukan apakah unggah massal harus men-merge (update sebagian field) atau replace penuh; hati-hati jika replace dapat menghapus data penting.
Error umum & solusinya:
Format tanggal salah → perbaiki format; NIK tidak valid → verifikasi KTP; kolom hilang → gunakan template resmi; ukuran file → pecah batch.
Monitoring pasca-unggah:
Cek laporan tagihan/kontribusi bulan pertama setelah perubahan; pastikan perubahan tercermin di laporan payroll dan potongan.
Test case minimal sebelum go-live:
1) baris sukses sederhana
2) baris dengan NIK duplikat
3) baris dengan format tanggal berbeda
4) baris gagal karena mandatory kosong
5) batch besar.
SOP koreksi:
Unduh laporan error → perbaiki di file sumber → ulang upload hanya baris yang gagal; jangan re-upload seluruh file kecuali dimaksudkan.
Komunikasi dengan pihak portal:
Siapkan bukti (log, file sumber, screenshot job id) saat eskalasi. Catat jam, pesan error, dan nomor tiket/ID job.
Integrasi payroll & HRIS:
Buat pipeline ETL yang men-generate file import dari HRIS secara otomatis setelah validasi, termasuk aturan mapping & normalisasi.
Automasi validasi:
Gunakan skrip (Python/VBA) atau tools ETL untuk checklists: NIK, tanggal, duplikat, karakter set, konsistensi upah, referensi kode cabang.
Dokumentasi:
Simpan manual operasional, template contoh, mapping, dan changelog tiap kali struktur kolom berubah.
Kontrol versi & change management:
Untuk template/skrip, gunakan version control (git) dan tandai release yang digunakan untuk setiap upload.
Rekomendasi pelaksanaan:
Lakukan unggah pada akhir hari kerja/low-traffic window; upload pertama pada subset kecil; gunakan batching; pastikan tim on-call saat upload besar.
Checklist quick-run sebelum upload:
backup data lama; jalankan preflight script; cek user access; beri tahu stakeholder; simpan nama file & versi; catat job id.
Risiko & mitigasi:
Kehilangan data karena replace → gunakan backup; korupsi karakter → gunakan UTF-8 & cek; saldo iuran tidak sinkron → cross-check laporan kontribusi.
Frequently Asked Points:
Selalu gunakan template resmi; jangan edit header; validasi NIK; simpan laporan hasil; koreksi hanya baris gagal bila memungkinkan.
Catatan akhir:
Selalu konfirmasi format terakhir di portal resmi sebelum unggah — template resmi bisa berubah; dokumentasikan setiap perubahan prosedur.

Form Excel 34 Panduan BPJS Kesehatan
Tujuan & sasaran
Form Excel 34 (Format 34 kolom) adalah template kolektif/kanonik yang digunakan pemberi kerja untuk mendaftarkan peserta BPJS Kesehatan baru (pegawai dan tanggungan yang belum memiliki nomor kepesertaan BPJS). Tujuan: menangkap seluruh data tingkat perusahaan dan peserta dalam skema tetap agar BPJS dapat memvalidasi, membuat record kepesertaan, menetapkan faskes dan memproses iuran. Gunakan Form_34.xlsx resmi dari e-DABU/SVBU dan ikuti sheet “kamus data 34 kolom” di template untuk definisi field yang otoritatif.
Siapa yang menyiapkan & mengunggah (peran & tanggung jawab)
Primer:
- Pemberi kerja (HR / PIC payroll) atau vendor HR/payroll berwenang dengan akses BU e-DABU.
Pendukung:
- Kantor cabang / account manager BPJS dapat membantu pengiriman awal atau kasus pengecualian.
Reviewer:
- reviewer independen HR/Finance (bukan yang menyiapkan) harus melakukan sample-check sebelum unggah.
Uploader:
- pengguna BU-account di e-DABU/SVBU yang melakukan unggahan secara aktual.

SIPP (BPJS Ketenagakerjaan)
SIPP (Sistem Informasi Pelaporan Perusahaan) BPJS Ketenagakerjaan adalah platform daring resmi untuk memudahkan perusahaan mengelola administrasi ketenagakerjaan: pendaftaran pekerja, pembaruan data upah/mutasi, serta perhitungan iuran JHT, JKM, JKK, dan JP secara real-time.
SIPP BPJS Kesehatan (Saluran Informasi & Pengaduan Peserta JKN-KIS)
SIPP BPJS Kesehatan adalah sistem yang disediakan untuk peserta JKN-KIS agar memperoleh informasi dan menyampaikan pengaduan seputar program JKN yang terhubung langsung ke Kantor Cabang. Saluran ini dibuat untuk memberikan kemudahan, kecepatan dan kepastian layanan informasi serta penanganan pengaduan peserta yang dapat diakses kapan saja.
Checklist operasional cepat (tugas satu-baris)
1. Dapatkan Form_34.xlsx resmi dari e-DABU.
2. Baca sheet kamus data.
3. Pemetaan export payroll → 34 kolom.
4. Format field ID sebagai Text (NIK/KK).
5. Validasi NIK vs export Dukcapil.
6. Validasi kode faskes/kecamatan sesuai kamus data.
7. Simpan .xlsx; unggah via e-DABU.
8. Perbaiki error laporan unggah portal; unggah ulang sampai bersih.
9. Arsipkan file final + laporan unggah + bukti pembayaran.
Header (field perusahaan) — yang muncul di atas sheet
Nama Instansi/Badan/Perusahaan (sesuai terdaftar); Virtual Account / kode VA; Bank tempat pembayaran iuran; Tanggal Registrasi / TMT (DD/MM/YYYY); Nomor PKS / Kode PKS / Unit Kerja / Kode Cabang (jika diminta). Informasi ini mengidentifikasi BU dan menentukan konteks unggahan. Gunakan data resmi perusahaan (NPWP, NIB) untuk ejaan persis.
Ringkasan field per-field (34 kolom tipikal — gunakan kamus data untuk urutan tepat)
Gunakan sheet kamus data di Form_34.xlsx resmi untuk nama/urutan kolom yang tepat. Berikut kolom yang umum muncul dan catatan praktis:
• No / Serial — nomor baris internal.
• No. KK (Kartu Keluarga) — 16 digit; set format sel Text untuk mempertahankan leading zero.
• NIK / KITAS / KITAP — 16 digit untuk NIK; untuk WNA gunakan passport/KITAS/KITAP sesuai kasus.
• Nama Lengkap — sesuai KTP/passport.
• Hubungan keluarga — kode (1=Peserta,2=Istri,3=Suami,4=Anak, dst.).
• Tempat Lahir / Tanggal Lahir (DD/MM/YYYY).
• Jenis Kelamin (1=Lk,2=Pr).
• Status Kawin (kode sesuai kamus data).
• Alamat lengkap; RT/RW; Kode Pos.
• Kode Kecamatan / Nama Kecamatan; Kode Desa / Nama Desa (gunakan kode dari kamus data).
• Kode & Nama Faskes Tk.I (primer) — kode adalah otoritatif; pilih puskesmas/klinik/GP (bukan rumah sakit) untuk faskes primer.
• Faskes dokter gigi (opsional di beberapa template).
• Telepon / Email (jika tersedia).
• NPP / No. Pegawai / Jabatan / Status Pegawai (identifier internal perusahaan).
• Kelas rawat (I/II/III) — ikuti kebijakan perusahaan dan ketentuan undang-undang; basis gaji dapat mempengaruhi kelas.
• Gaji pokok + tunjangan — digunakan untuk unggahan mutasi perubahan gaji bila diperlukan.
• Kewarganegaraan / Passport (untuk WNA).
• NPWP / Asuransi lain / No Polis / Catatan.
• Kolom 26–34 — flag/lampiran/catatan internal tergantung versi template. Selalu ikuti kamus data untuk nilai dan panjang yang diizinkan.

Kamus data & validasi kode
Buka sheet kamus data di Form_34.xlsx: daftar nilai valid untuk hubungan keluarga, jenis kelamin, status kerja, kode faskes, kode kecamatan/desa dan lookup lainnya.
Gunakan kode tersebut persis — validasi portal berbasis kode, bukan nama.
Buat tabel lookup internal dari kamus data untuk memvalidasi export sebelum unggah.
Langkah persiapan file (praktis, rinci)
1. Unduh Form_34.xlsx dari akun BU e-DABU (Referensi → Download File Excel).
2. Baca kamus data untuk identifikasi kolom wajib per tipe mutasi.
3. Pemetaan field export payroll → kolom template; buat script mapping/transform ETL yang dapat diulang.
4. Format NIK/KK sebagai Text; set kolom tanggal ke DD/MM/YYYY; hilangkan formula (paste as values).
5. Validasi internal: panjang NIK = 16; KK ada untuk tanggungan; kode faskes benar; kolom wajib terisi sesuai kode mutasi.
6. Split batch besar untuk mematuhi batas baris/ukuran portal; uji unggah dengan batch kecil (10–50 baris).
7. Simpan file final sebagai .xlsx; simpan salinan kerja tanpa tanda tangan dan arsip final.
Prosedur unggah (e-DABU / SVBU / cabang)
• Preferensi: unggah via e-DABU (Peserta → Upload Peserta) dan pilih tipe template 34-kolom; tentukan unit kerja jika diminta portal.
• Validasi: portal mengembalikan laporan unggah (error tingkat-baris). Perbaiki baris yang dilaporkan di sumber dan unggah ulang sampai tidak ada error. Simpan laporan unggah dan screenshot untuk audit trail.
• Offline: jika BU tidak dapat mengakses e-DABU, serahkan Form_34.xlsx ke kantor cabang BPJS Kesehatan sesuai instruksi cabang; cabang akan memberi arahan pemrosesan dan lampiran yang diperlukan.
Kesalahan unggah umum & remediasi
- NIK mismatch/invalid: koreksi NIK agar cocok dengan catatan Dukcapil atau lampirkan dokumen pendukung dan unggah ulang.
- Kode faskes invalid: ganti dengan kode dari kamus data (kode adalah otoritatif).
- Format tanggal salah / formula masih ada: konversi ke DD/MM/YYYY dan paste values.
- Struktur template berubah: kembalikan urutan/nama kolom ke format asli dari Form_34.xlsx resmi.
- Batas ukuran / baris: bagi menjadi batch lebih kecil dan unggah ulang.
Lampiran wajib & kasus khusus
- Penonaktifan (terminate): lampirkan paklaring / surat pemutusan / akta kematian / surat cerai bila relevan.
- Koreksi NIK/KK: sertakan dokumen koreksi Dukcapil jika tersedia.
- Entri WNA: isi kolom passport/KITAS di kolom passport/KITAS.
- Pemberi kerja besar: koordinasikan dengan account manager BPJS untuk pengiriman volume besar dan kemungkinan migrasi bertahap.
Siapa bertindak & RACI singkat (disarankan)
HR/Payroll (PIC): siapkan mapping, isi Form_34 dan jalankan validasi.

Pemetaan 34-kolom kanonik — rincian otoritatif (diperluas)
1) RowID / No. — indeks numerik berurutan untuk referensi internal; berguna pada laporan kesalahan dan diff pemetaan. Sertakan referensi sumber asli (mis. source_row_id) untuk traceability dua-arah.
2) Nomor Kartu Keluarga (KK) — 16 digit (format Text); identik untuk baris keluarga. Validasi integritas grup: tandai grup KK yang kehilangan kepala keluarga atau yang memiliki beberapa kepala keluarga bertentangan kebijakan. Simpan kk_group_consistency di pra-proses dan kk_group_id untuk partisi unggah aman.
3) NIK / KITAS / KITAP — identitas primer: NIK 16 digit untuk WNI; untuk WNA gunakan KITAS/KITAP/paspor sesuai aturan template. Simpan sebagai Text untuk mencegah pemotongan numerik. Jika rekonsiliasi registri tersedia, lakukan lookup batch dan tangkap dukcapil_response_code, dukcapil_name, dan dukcapil_match_confidence. Untuk pengecualian, lampirkan dokumen pendukung dan catat pernyataan operator.
4) Nama Lengkap — nama resmi persis seperti ID; pertahankan diakritik menggunakan normalisasi UTF-8 NFC; tegakkan panjang maksimum sesuai template dan dokumentasikan kebijakan pemotongan. Simpan name_normalized untuk algoritma pencocokan (mis. strip_accents(lowercase)) hanya untuk pencocokan, bukan untuk pengiriman.
5) Hubungan (relationship) — kode enumerasi: 1=Peserta, 2=Istri, 3=Suami, 4=Anak, dll. Gunakan kode kamus untuk memastikan attachment hierarkis; validasi persis satu peserta primer per grup keluarga bila kebijakan mengharuskan. Terapkan validasi tingkat keluarga: pastikan DOB dependent dan hubungan konsisten (mis. cek plausibilitas usia pasangan/anak).
6) Tempat Lahir — tempat lahir (kota/kabupaten) — simpan label dan kode admin opsional; normalisasi via pemetaan kamus bila tersedia. Untuk nama ambigu sertakan place_of_birth_confidence untuk menangkap status pemetaan fuzzy.
7) Tanggal Lahir (DOB) — dd/mm/yyyy; normalisasi dan validasi tanggal tidak mungkin (DOB di masa depan, DOB < 1900) dan hitung usia turunan untuk logika kelayakan cakupan. Untuk pendaftaran bayi baru lahir gunakan jendela kebijakan (mis. izinkan TMT dibackdate X hari) dan dokumentasikan pengecualian.
8) Jenis Kelamin — kode (1=Laki-laki, 2=Perempuan). Jika template menyertakan kode lain atau semantik unknown, tangkap sumber asli dan peta secara konservatif, tandai untuk review HR bila diperlukan.
9) Status Perkawinan — kode per kamus (mis. 1=Belum Kawin, 2=Kawin, 3=Cerai). Cross-check dengan hubungan dependent dan gender NIK sesuai kebijakan bila diperlukan.
10) Alamat Lengkap — alamat tempat tinggal sesuai catatan HR; pertahankan tanda baca bila perlu; normalisasi singkatan (Jl, Jln) konsisten. Simpan raw_address di staging untuk audit; gunakan address_parsed untuk geocoding dan pemetaan kamus.
11) RT / RW — kode lingkungan (simpan sebagai Text untuk mempertahankan leading zero). Validasi terhadap hierarki kecamatan/desa.
12) Kode Pos — kode pos 5 digit; verifikasi terhadap tabel pos bila tersedia dan tandai outlier untuk verifikasi manual.
13) Kode Kecamatan — kode admin resmi; peta via lookup kamus dan validasi konsistensi kecamatan–desa.
14) Kode Desa / Kelurahan — kode desa yang cocok dengan kecamatan; pastikan validasi hierarki dan gunakan fallback geo jika pemetaan gagal (antrian review manual).
15) Kode Faskes Tk.I — kode fasilitas primer (puskesmas/klinik/GP). Kode bersifat otoritatif — portal menggunakannya untuk menetapkan faskes primer; peta ketat ke kamus dan tangkap faskes_mapping_confidence untuk alur review manual.
16) Nama Faskes Tk.I — opsional; kode adalah primer. Jangan mengandalkan nama saja. Jika kode & nama disertakan, verifikasi konsistensi dan tandai mismatch untuk tindakan operator.
17) Kode Faskes Dokter Gigi — jika template meminta; peta via kamus dan tangani optionality sesuai tipe mutasi.
18) Telepon / Email — field kontak peserta opsional; format telepon sebagai Text untuk mempertahankan tanda plus dan kode negara; validasi email dengan regex pragmatis (RFC-5322-lite) dan catat kontak invalid sebagai peringatan (bukan kegagalan kritis kecuali dipersyaratkan oleh kebijakan).
19) Status Pegawai — kode status kerja (1=Tetap, 2=Kontrak, 3=Harian/Perjam dll.). Peta status internal perusahaan ke kode template dan simpan tabel pemetaan versi kontrol dengan metadata approver.
20) NPP (No. Pegawai) — nomor pegawai internal untuk rekonsiliasi dengan payroll; pastikan unik dalam perusahaan dan simpan identifier sebelumnya untuk alur mutasi. Gunakan npp_history untuk link audit identifier lama.
21) TMT Kerja (start date) — dd/mm/yyyy tanggal efektif kerja; diperlukan untuk hitung cakupan & jendela kelayakan. Cross-check TMT vs DOB dan Gaji bila aturan bisnis memerlukan (mis. masa percobaan mempengaruhi pilihan kelas).
22) Kelas Rawat — I/II/III; hitung deterministik dari band gaji jika perusahaan menetapkan kelas, atau kosongkan jika portal menentukan menurut aturan. Selalu simpan kelas_decision_source (mis. company_policy_v3 + signature approver).
23) Gaji Pokok (monthly) — angka bulat, tanpa pemisah. Gunakan gross penggajian atau base yang dideklarasikan pemberi kerja untuk kontribusi; konfirmasi aturan regional untuk basis kontribusi dan terapkan aturan pembulatan/pemotongan konsisten. Tangkap contribution_basis_components jika beberapa elemen menyusun basis.
24) Tunjangan / Komponen Upah Lainnya — kolom opsional untuk tunjangan jika template menyertakannya dan bila komponen tersebut menentukan basis kontribusi; dokumentasikan komponen yang termasuk dan versi mapping ini.
25) NPWP — ID pajak (format Text); validasi struktur dan opsional cross-check dengan otoritas pajak bila integrasi ada. Tandai NPWP yang malformed tapi perlakukan non-blok kecuali dibutuhkan untuk mutasi tertentu.
26) No Passport — untuk WNA; sertakan indikator negara penerbit jika diminta. Simpan passport_expiry untuk cek kelayakan & pembaruan.
27) Kewarganegaraan — kode kewarganegaraan (1=WNI, 2=WNA) per kamus. Untuk dual-national capture kedua pengenal dan atur prioritas sesuai kebijakan.
28) Kode Mutasi / Operation — kode operasi: DAFTAR_BARU, PERUBAHAN_IDENTITAS, PERUBAHAN_FASKES, PERUBAHAN_GAJI, PENONAKTIFAN, dll. Masing-masing kode mengaktifkan kolom wajib — otomasi mask kolom wajib per tipe mutasi dan pastikan engine validasi menerapkan mask ini sebelum unggah.
29) Tanggal Mutasi / Pelaporan — dd/mm/yyyy tanggal efektif unggah/mutasi; pastikan penyelarasan timezone dan periode pelaporan dengan cut-off payroll. Untuk pemrosesan batch sertakan reporting_period_month dan reporting_cutoff artifacts untuk bukti kelancaran throughput.
30) Kode Alasan Penonaktifan — kode alasan (resign, termination, death). Lampirkan referensi dokumen pendukung sesuai pemetaan manifest dan pastikan retensi bukti sesuai jendela audit kebijakan.

Tangkap attachment_sha256 dan attachment_storage_path dalam manifest untuk pemeriksaan integritas.

Aturan validasi tingkat-field & cek best-practice (diperluas)
- NIK/KK: simpan sebagai Text; tegakkan regex ^\d{16}$ dan pastikan numeric-only. Implementasikan cross-check batch via registri kependudukan bila tersedia dan simpan catatan audit setiap pengecualian dan dokumen pendukung. Tandai NIK yang ada tetapi terasosiasi nama berbeda di registri untuk review manual; buat workflow DCA (discrepancy confirmation attestation) untuk bukti dari karyawan.
- Konsistensi antar-field: validasi Hubungan vs DOB (mis. "Anak" harus lebih muda dari kepala keluarga dalam margin wajar); pastikan grup KK punya tepat satu baris "Peserta" primer kecuali kebijakan multi-peserta berlaku. Bangun rule-set tingkat keluarga untuk mendeteksi dependent yatim dan komposisi keluarga yang tidak konsisten.
- Faskes: peta menggunakan kode fasilitas resmi dari kamus; update daftar kanonik ketika portal menerbitkan versi kamus baru dan snapshot kamus per unggah untuk reproduksibilitas. Untuk faskes yang tidak ditemukan buat alur tiket faskes_exception dengan SLA yang disyaratkan dan mapping manual interim bila diizinkan.
- Gaji → Kelas mapping: enkode aturan band-gaji-ke-kelas di table sumber-kebenaran tunggal dengan tanggal efektif; selalu catat versi mapping efektif dan approver. Sertakan prosedur rollback saat logika mapping berubah.
- Validasi tanggal: gunakan parser tanggal andal dan aturan bisnis (DOB < hari ini; TMT tidak boleh di masa depan kecuali diizinkan; tanggal mutasi dalam jendela pelaporan). Sertakan cek tahun kabisat dan pastikan locale (DD/MM/YYYY) ditegakkan. Normalisasi serial date Excel ke string sebelum upload bila portal mengharuskan tanggal format teks.
- Encoding karakter: tegakkan UTF-8 NFC. Hapus karakter kontrol (ZWJ, ZWNJ, BOM) sebelum unggah dan normalisasi whitespace. Ganti karakter non-printable dengan placeholder dan tandai baris untuk review operator. Pertahankan file encoding_errors untuk audit.

2. Normalisasi: normalisasikan Unicode ke NFC; trim whitespace leading/trailing; ganti whitespace berulang jadi satu spasi; hapus karakter kontrol non-printable dan normalisasi tanda baca (kutip pintar, dash panjang). Jalankan pass sanity_stats untuk menangkap distribusi kolom kunci (mis. jumlah NIK unik, persentase email kosong, kode pos invalid).

Gunakan id_coercion_report.csv untuk menampilkan record yang dipaksa dan approval operator bila padding otomatis diterapkan.
4. Jika geocoding gagal, coba inferensi unit admin dari kode pos lalu set address_inference_reason.

Pertahankan faskes_override_log untuk mencatat override manual dengan who, why, timestamp, dan nilai sebelumnya.
6. Normalisasi tanggal: parse semua input tanggal dan format ke DD/MM/YYYY, deteksi tanggal mustahil dan tambahkan ke file errors; sertakan sumber parser dan nomor baris untuk reproduksibilitas. Bangun daftar pengecualian untuk tanggal borderline (mis. serial Excel yang salah tafsir) dan dokumentasikan aturan transformasi yang diaplikasikan.
7. Pembersihan gaji: hapus pemisah ribuan, validasi range numerik, dan hitung basis kontribusi turunan bila perlu. Terapkan aturan pembulatan/pemotongan dan catat versi aturan dalam salary_transformation_log. Untuk gaji negatif atau nol, kirim ke salary_investigation_queue.

Jika duplikat ditemukan, terapkan aturan: jika NIK sudah ada di daftar aktif saat ini, perlakukan sebagai mutasi bukan pendaftaran baru; catat prior active upload IDs dan set action_suggested = mutation atau merge.

Klasifikasikan error sebagai critical (blok), major (perbaikan HR), atau info (hanya dokumentasi). Berikan petunjuk remediasi per kode error (mis. ERR_FASKES_MAP: daftar 3 kandidat teratas dengan skor similarity).

Wajibkan sign-off HR + Payroll direkam melalui tanda tangan digital atau event SSO sebelum unggah portal. Simpan semua artifak di arsip terenkripsi dengan metadata retensi.

Regex & validator ringan (PCRE)
- NIK: ^\d{16}$
- KK: ^\d{16}$
- Tanggal (dd/mm/yyyy sederhana): ^([0-2]\d|3[0-1])\/(0\d|1[0-2])\/\d{4}$ (gunakan parser tanggal untuk validasi nyata)
- Postal: ^\d{5}$
- Gaji: ^\d+$ (tanpa pemisah)
- Email (sederhana): ^[^\s@]+@[^\s@]+\.[^\s@]+$
Simpan regex di file validator terpusat dan versioning. Untuk cek kritis gunakan parser deterministik alih-alih hanya regex (mis. parsing tanggal). Unit-test regex dengan contoh edge-case dan sertakan vector uji di repository.
Strategi pemetaan fuzzy (praktis)
Saat HRIS menyimpan nama fasilitas sebagai teks bebas, bangun tabel mapping kanonik dengan field: source_name, canonical_faskes_code, canonical_faskes_name, similarity_score, manual_override_flag, approved_by, approved_on. Gunakan trigram similarity atau Jaro-Winkler dengan ambang konservatif (mis. 0.85); di bawah ambang, antri untuk review manual. Persist mapping yang disetujui ke tabel kanonik untuk mengurangi pekerjaan manual ke depan. Catat siapa yang menyetujui setiap mapping dan timestamp untuk audit. Pertahankan mapping_decay_policy untuk menilai ulang mapping berkala (mis. triwulan) terhadap kamus terbaru.
Proses unggah — langkah demi langkah (operasional, diperluas)
1. Pastikan Form_34_ready.xlsx final dan disign-off (HR + Payroll). Gunakan tanda tangan digital atau event SSO tercatat.
2. Simpan salinan final dengan nama file kanonik dan hitung checksum sha256; catat di manifest.txt dan upload_log.csv dengan extraction_time dan operator.
3. Login ke portal unggah pemberi kerja menggunakan akun BU (gunakan 2FA dan SSO perusahaan bila tersedia).
4. Navigasi ke Upload → pilih "Format 34 kolom" → pilih unit kerja dan unggah file. Simpan screenshot navigasi portal dengan timestamp untuk audit.
5. Portal mengembalikan laporan validasi (langsung atau background). Unduh dan analisis laporan: mencantumkan kesalahan & peringatan per baris. Otomasi parsing laporan portal untuk memetakan kode kesalahan portal ke taksonomi internal.
6. Koreksi kesalahan di sumber dan regenerasi Form_34_ready.xlsx (atau buat file delta terpisah untuk baris gagal). Simpan file delta dan cross-reference ke file part asli.
7. Upload ulang sampai diterima; jika terjadi penerimaan parsial, catat baris yang diterima dan kirim ulang baris gagal di file terpisah (gunakan penamaan part##). Pastikan file part tidak saling tumpang tindih dan sertakan rows_range di manifest.
8. Unduh konfirmasi penerimaan portal dan acceptance ID; arsipkan laporan penerimaan dengan manifest yang ditandatangani, sha256 penerimaan, dan screenshot atau tiket portal. Buat acceptance_record.json berisi upload_id, accepted_rows, rejected_rows, timestamp, operator_id.
Menginterpretasi kode kesalahan portal & remedinya (diperluas)
Pemetakan kesalahan portal ke tiga bucket remediasi: Perbaikan Data, Update Kamus/Mapping, Masalah Portal/Proses. Langkah dan remedinya:
- ERR_NIK_NOT_FOUND / NIK_MISMATCH — penyebab: NIK tidak cocok registri atau invalid. Remediasi: verifikasi KTP karyawan, jalankan validasi registri, unggah baris yang dikoreksi atau eskalasi dengan scan ID pendukung. Simpan rantai bukti (scan, pernyataan HR). Jika registri menunjukkan multiple record, siapkan tiket rekonsiliasi termasuk dukcapil_response dan affidavit karyawan.
- ERR_FASKES_INVALID — penyebab: kode faskes tidak ditemukan atau nama mismatch. Remediasi: cek kamus, peta ke kode benar, upload ulang. Jika faskes hilang di kamus, buka permintaan kamus_extension dengan metadata wajib.
- ERR_LAYOUT — penyebab: header berubah atau kolom hilang. Remediasi: pulihkan urutan kolom kanonik dan re-map; selalu jalankan header-checker pra-unggah. Sediakan skrip header_checker.py untuk validasi kesetaraan header case-sensitive.
- ERR_DUPLICATE — penyebab: duplikat NIK atau multiple pendaftaran. Remediasi: verifikasi apakah record aktif; jika aktif gunakan flow mutasi bukan pendaftaran baru; sertakan prior acceptance IDs di catatan remediasi. Untuk duplikat dalam file gunakan rule deterministik (mis. pilih baris terbaru berdasarkan source_timestamp) dan log override.
- ERR_DATE_INVALID — penyebab: tanggal malformed atau di luar rentang. Remediasi: koreksi format & logika bisnis. Untuk masalah parsing sistemik, cek serial Excel vs string dan pastikan locale DD/MM/YYYY.
- ERR_ATTACHMENT_REQUIRED — penyebab: mutasi memerlukan dokumen pendukung. Remediasi: lampirkan scan sesuai manifest & upload ulang; gunakan zip password jika portal tidak mendukung lampiran aman dan catat metode pertukaran password.
- ERR_KAMUS_MISMATCH — nilai tidak ada di kamus untuk field tertentu. Remediasi: konsultasikan kamus terbaru; jika perbedaan dikonfirmasi buka tiket support portal dan snapshot versi kamus & bukti untuk permintaan penambahan.

Strategi pengujian, golden file & regresi

Gunakan CI untuk mengendalikan rilis pipeline mapping. Tambahkan smoke upload otomatis (sample 10 baris) ke portal staging bila tersedia dan sertakan langkah verifikasi penerimaan otomatis yang mem-parse respons portal. Pertahankan regression_dashboard yang menunjukkan pass/fail per golden test lintas versi template.

Rollback & workflow koreksi (terkendali)
Jika data yang salah diterima: siapkan Form_34 korektif menggunakan kode mutasi yang sesuai dan sertakan dokumen pendukung wajib. Unggah file korektif hanya untuk baris terpengaruh. Pertahankan ledger rollback: erroneous_upload_id → corrective_upload_id → reason → operator → timestamp. Untuk permintaan revert skala besar, koordinasikan dengan kantor regional — jangan coba operasi destruktif di portal tanpa panduan eksplisit. Lacak metrik rollback dan waktu hingga koreksi (time-to-correct) dan sediakan ringkasan mingguan untuk stakeholder. Bila diizinkan, otomasi pembuatan file korektif menggunakan diff antara dataset diterima dan master sumber lalu arahkan ke HR untuk sign-off.
Rekonsiliasi & QA pasca-acceptance

Otomatiskan job rekonsiliasi periodik yang membandingkan dataset diterima portal dengan master lokal dan tandai divergensi. Simpan artefak reconciliation_run dengan run_id, sample_size, pass_rate, dan tindakan remediasi.

6. Struktur template: aturan header, sheet, dan data row (kebijakan teknis)

Prinsip utama: jangan mengubah struktur dasar template.

- Sheets & artefak tidak relevan: hanya sertakan sheet yang diminta. Hapus sheet lainnya, styles yang tidak perlu, komentar, gambar, OLE objects, dan custom properties yang bisa mengacaukan parser portal.

- Merged/hidden cells: hindari penggabungan sel (merged cells) dan hidden rows/cols berisi data — beberapa parser portal mempertimbangkan struktur internal Excel.

7. Field-level aturan & remediasi praktis (detil implementable rules + examples)

Untuk setiap kolom, sediakan: field_name, type, regex, auto-fix rules, escalation policy, example_in, example_out. Berikut subset lengkap untuk field krusial:

▪ NIK
▪ Type: Text
▪ Rule: digits only, exactly 16. Regex: ^\d{16}$.
▪ Auto-fix: strip_non_digits(value); value = unicodedata.normalize('NFKC', value); trim leading/trailing whitespace; replace NBSP (U+00A0) with space then strip.
▪ Escalate: jika length != 16 setelah auto-fix → ERROR and request scanned KTP.
▪ Example: 32 0123 4567 890123 → 3201234567890123 (pass).

▪ NOKA (No KPJ)
▪ Type: Text
▪ Rule: optional for new participants; if present, must match registry pattern (validate via registry API when available).
▪ Auto-fix: trim & normalize; if mismatch → WARNING and mark for reconciliation.

▪ Nama Lengkap
▪ Type: Text
▪ Rule: length 3–100 characters; normalized to Unicode NFC; no C0 control characters.
▪ Auto-fix: collapse repeated whitespace; map known diacritics normalization.

▪ Tempat/Tgl Lahir
▪ Type: Text/Date
▪ Rule: target format DD/MM/YYYY; allowable input formats YYYY-MM-DD, DD-MM-YYYY, DD/MM/YYYY. Age must be between 15 and 120.
▪ Auto-fix: parse and normalize; if parse fails or age outside range → ERROR.
▪ Example: 1990-02-28 → 28/02/1990; 31/02/1990 → ERROR.

▪ Jenis Kelamin
▪ Type: Lookup
▪ Rule: map common variants to codes required by template (L=Laki-laki, P=Perempuan).
▪ Auto-fix: normalize case and known mappings (M→L, F→P, male→L, female→P). Unmapped → WARNING/ERROR per business rule.

▪ Nama Ibu Kandung
▪ Type: Text
▪ Rule: mandatory; if missing → ERROR and HR remediation required.

▪ Alamat
▪ Type: Text
▪ Rule: minimally contain street + kel/desa + kecamatan; prefer postal code.
▪ Auto-fix: trim, collapse whitespace, remove control chars. If components missing → WARNING (unless required by business rule).

▪ No HP (phone)
▪ Type: Text
▪ Rule: digits only after transform; length 9–15; preserve leading zero when used.
▪ Auto-fix: remove (, ), spaces, dashes; replace +62 with 0; convert Arabic/Persian digits to ASCII. Regex after transform: ^\d{9,15}$.

▪ NPWP
▪ Type: Text
▪ Rule: if present, 15 digits (strip non-digit).
▪ Auto-fix: strip non-digits; if length mismatch → WARNING.

▪ Upah / UDI
▪ Type: Numeric (integer)
▪ Rule: no thousands separator in file; business min/max validation applies.
▪ Auto-fix: remove thousands separators (. or , in thousand position), convert decimal comma to dot if used for decimal; coerce to integer; if not numeric → ERROR.

▪ TMT (Tanggal Mulai Tugas)
▪ Type: Date
▪ Rule: DD/MM/YYYY and must be <= today. Future date → ERROR unless business rule allows future effective dates.
▪ Auto-fix: parse common formats and normalize.

▪ Status / Unit / Kode Cabang
▪ Type: Lookup
▪ Rule: mapping must be maintained in version-controlled kamus (lookup). Unmapped values → WARNING or ERROR depending on requiredness.
▪ Escalate unmapped candidates to kamus team with sample rows.

Catatan operasional: semua auto-fix harus dicatat di error_report.xlsx sebagai processed_by=auto_fix dan corrected_value diisi. Perubahan identity (NIK, name canonicalization beyond whitespace) memerlukan manual sign-off dan approval artifact.

12. Strategi batch, sizing & performance (operational patterns & tuning)

Saran chunking: produksi chunk 1.000–2.000 rows per file untuk keseimbangan throughput dan traceability. Untuk organisasi besar, gunakan parent-child batch pattern: satu parent batch yang menyertakan sub-batch manifest entries.

Concurrency & rate-limiting: jika portal support parallel uploads, limit client concurrency to 2–4 parallel uploads. Implement token-bucket or leaky-bucket algorithm client-side to avoid bursts.

Retry/backoff: implement exponential backoff for transient HTTP 429/503 errors. Pseudocode:
delay = initial_delay
for retry in 1..max_retries:
attempt_upload()
if response == 200: break
elif response in [429,503]: sleep(delay); delay = min(delay * factor, max_delay)
else: fail and create incident

Batch_id convention: BATCH-{YYYYMMDD}-{unit}-{seq} and include run_id linking parent runs.

Monitoring: track per-batch latency, server response codes, and throughput. Alert when error rate >5% or SLA breaches.

40. Appendix C — Contoh transform & mapping rules (pseudocode & examples)

Pseudocode common transforms:
- employee.nik -> NIK:
value = strip_non_digits(value)
value = unicodedata.normalize('NFKC', value)
if len(value) != 16: flag ERROR

- join_date -> TMT:
if join_date is None: TMT = contract.start_date
else: TMT = parse_to_ddmmyyyy(join_date)

- salary_components -> UpahDasar:
UpahDasar = round(basic + fixed_allowances)

- phone:
value = value.strip()
value = replace_prefix(value, '+62','0')
value = remove_non_digits(value)
if len(value) not in 9..15: flag WARNING/ERROR

Include unit-tests per transform in mapping_test_cases.csv.
41. Appendix D — Regex & validation reference (comprehensive)

Canonical regex patterns:
- NIK: ^\d{16}$.
- HP (after transform): ^\d{9,15}$.
- Upah (numeric integer): ^\d+$ (or ^\d+(\.\d{1,2})?$ if decimals allowed).
- Tanggal DD/MM/YYYY: ^\d{2}\/\d{2}\/\d{4}$.
- Phone detection (pre-normalize): ^\+?[\d\-\s()]{9,20}$.

Edge-case handling: before regex checks, run unicodedata.normalize('NFKC', value), strip NBSP (\u00A0), remove zero-width joiner/zero-width non-joiner, and trim control characters. Convert non-ASCII digits to ASCII equivalents.
