1. What this query is doing in operational terms
This Power Query is a two-source payroll consolidation and comparison pipeline. It takes one table representing Gross Up staff cost data and another representing Gross staff cost data, standardizes both, computes internal totals for each compensation model, and then merges the two versions by exact staff name so they can be compared side by side. In practical terms, this is not just a data-cleaning query. It is a business-reporting query that converts raw payroll components into a structured analytical view.

The query is doing four things at once. First, it normalizes messy input into a predictable schema. Second, it calculates derived totals from detailed payroll components. Third, it removes intermediate detail columns that are no longer needed for the final report. Fourth, it joins the two prepared versions into one output table. The result is a single comparison dataset that shows how the same employee looks under two compensation structures.
2. The query is a controlled ETL pipeline, not a simple reshaper
The structure of the M code makes it clear that this is a multi-stage ETL routine. It extracts source tables from the workbook, transforms them through a series of named steps, and loads a final table. Each named step has a specific purpose: clean inputs, compute totals, join datasets, and control output shape. That is very different from a one-off formula or a shallow table cleanup.

This matters because the query has to survive the kinds of imperfections common in real payroll files. It is designed as a predictable pipeline rather than a fragile chain of assumptions. That is exactly the kind of structure you want when the output will be used for tax, salary, or cost analysis.
3. Why the let ... in structure is the correct architecture here
The let ... in structure is ideal for this kind of work because it allows each transformation stage to be named and reused. It also makes the logic easier to read and audit. Instead of nesting a large set of expressions inside each other, the query creates intermediate results with names such as Promoted, Selected, Renamed, and Final. Those names are not cosmetic. They document the transformation sequence.

In a payroll query, readability is a form of correctness. If a future reviewer needs to understand why a value is being summed or why a column disappears, the named steps tell the story. That is one of the biggest advantages of Power Query over a pile of worksheet formulas.
4. The query is built around safe assumptions, not blind trust
Almost every major step in this query is defensive. It checks whether the source has generic column names before promoting headers. It checks that Nama exists before proceeding. It uses MissingField.UseNull and MissingField.Ignore so the query can continue even when columns are absent. It converts values through a helper that tolerates nulls and text. These are all signs that the author expects messy source data and has planned for it.

That is important because payroll data often contains blanks, formatting inconsistencies, and varying schemas across sources. The query does not assume perfect input. It builds protections around imperfection.
5. ToNumberOrZero is the foundation of all downstream math
ToNumberOrZero is the numeric safety net for the entire query. Its job is to turn any value into a usable number. If the value is null, it returns zero. If the value is already numeric, it returns it unchanged. If the value is text, it trims the text and attempts to interpret it as a number in two different ways. Only if both parsing paths fail does it fall back to zero.

This is critical because every total column in the query depends on being able to sum source fields without crashing. A single text entry, blank cell, or weirdly formatted import should not bring down the report. This helper prevents that. It turns a messy spreadsheet world into arithmetic-friendly inputs.
6. Why the helper tries both Number.From and Number.FromText
The use of two parsing attempts is a deliberate robustness strategy. Some source values can be converted directly using Number.From, while others behave better if you first convert them to trimmed text and then parse them. This gives the query two chances to salvage a number from a value that is not already cleanly typed.

The redundancy is useful because Excel-originated data can be surprisingly inconsistent. A value may look numeric to a user but still arrive in Power Query as text, locale-formatted text, or a mixed-type cell. The helper handles those cases without needing separate cleanup passes.
7. Why zero is the correct fallback for missing or invalid numeric values
For aggregation logic, zero is the safest fallback because the query is computing totals, not averages or ratios. If a component is missing or invalid, treating it as zero preserves the rest of the row and allows the report to complete. That is usually the right business choice in a salary-cost summation context, because one missing component should not block the whole dataset.

This does mean the query can silently hide data issues if a source field unexpectedly becomes invalid. But in a report-oriented payroll model, continuity is often more valuable than hard failure. The query chooses availability over brittleness.
8. EnsureHeaders protects the query from double-promoting or under-promoting headers
EnsureHeaders looks at the table’s column names and only promotes the first row when the table still appears to have generic Column1, Column2, etc. names. This is a practical way to deal with mixed input conditions. Some Excel sources arrive already promoted, others do not. If the query always promoted headers, it could corrupt a table that was already properly formed. If it never promoted headers, it could leave the table unusable.

The function resolves that problem by checking for the generic pattern first. That means the query adapts to the shape of the source instead of forcing a single assumption on every input.
9. Why the first-column test works well enough in this context
The query tests the first column name using Text.StartsWith(cols{0}, "Column"). That is a lightweight heuristic. It does not try to inspect every column name or infer headers from content. It simply checks whether the table still looks like a raw imported table. In the context of Excel-generated data, that is usually enough.

The method is not universal, but it is a practical compromise. It avoids unnecessary complexity and keeps the query fast. For a controlled workbook with predictable data sources, that is the right tradeoff.
10. AddTotalColumn is the query’s reusable arithmetic engine
AddTotalColumn is the general-purpose function that sums a list of fields into a new derived column. It takes three arguments: the table, the name of the new total column, and the list of source fields to sum. For each row, it reads each listed field from the current record, passes it through ToNumberOrZero, and then sums the result with List.Sum.

This function eliminates repetition. Instead of writing separate summing logic for each payroll total, the query defines the pattern once and reuses it several times. That improves consistency and makes the business logic easier to change later.
11. Why Record.FieldOrDefault is especially useful here
Record.FieldOrDefault(_, f, null) allows the total function to ask for a field by name without failing if the field is missing. If the source record does not contain that field, the function simply returns null. The numeric helper then converts that null to zero. This means a missing column no longer destroys the row calculation.

This is a very strong defensive pattern in Power Query. It allows the query to tolerate slight schema drift or missing optional fields, which is common when different payroll extracts are not perfectly identical.
12. PrepareStaffTable is the main normalization layer
This function is the core of the query. It takes one source table and a suffix that identifies whether the table belongs to the Gross Up or Gross branch. Inside the function, the source is promoted if necessary, validated, reduced to the columns needed for that branch, renamed into a suffix-based naming scheme, enriched with derived totals, stripped of intermediate columns, and finalized with a total staff cost figure. In other words, this function turns raw payroll input into a standardized analytical dataset.

That is why most of the important business logic lives here. It is the step where the query moves from raw extraction to structured reporting.
13. Why checking for Nama is a critical validation gate
The first thing PrepareStaffTable does after header promotion is verify that the column Nama exists. If it does not, the query raises an error that includes the column names it did find. This is a strong validation step because Nama is the row identity field that everything else depends on. Without it, the merge later in the query cannot work properly.

By failing early and explicitly, the query prevents misleading output. It is better to stop with a clear error message than to continue with a broken identity key and produce a report that looks valid but is actually wrong.
14. Why the error message is useful rather than generic
When Nama is missing, the query does not just throw a vague failure. It tells the user exactly which columns were found. That is operationally helpful because it narrows the problem immediately. The user can compare the expected schema to the actual schema and spot import mistakes, renamed fields, or missing headers.

This kind of error reporting is important in workbook automation because users often do not have the time or patience to debug opaque failures. A descriptive message saves time and reduces support friction.
15. Table.SelectColumns is used to define the business schema
After validation, the query selects only the columns that matter for the given suffix. That means the table becomes intentionally narrow. For Gross Up, the selected set includes the tax-related and contribution-related components that belong in that model. For Gross, the selected set is slightly smaller. This is not accidental; it reflects the business logic of each compensation structure.

This step is important because it removes irrelevant clutter and forces the transformation to operate on a controlled schema. The query is not trying to be generic. It is trying to be correct for a specific payroll model.
16. MissingField.UseNull means the query prefers continuation over failure
When selecting columns, the query uses MissingField.UseNull. If a requested column is absent, Power Query creates it as null rather than halting. That is a forgiving behavior. It allows the rest of the pipeline to continue even when the source table is incomplete. Since the later totals use ToNumberOrZero, those nulls become zero and the row can still be processed.

This approach is appropriate when the source is expected to be mostly correct but not perfectly uniform. It is less appropriate in a strict validation context, but here the query’s purpose is to create a usable comparison table. Continuation is a better fit.
17. Why the renaming step matters more than it first appears
Renaming the selected columns is not just a cosmetic change. It creates a schema that can survive the later join. Both the Gross Up and Gross datasets contain fields with the same business meaning, such as Basic Salary or PPh 21. If they were not renamed with suffixes, the merge would produce ambiguous duplicate names. The suffixing creates a clean “before/after” style dataset in one result table.

This is what allows the final output to show both versions side by side without confusion. The renaming stage is therefore a central part of the comparison design.
18. Nama becomes Staff Name to standardize the identity field
The query renames Nama to Staff Name in both branches. That is a normalization step that makes the key column explicit and consistent across both source tables. It also creates a more readable final report because the output label is clearer than the source-language raw field name.

This matters because the join later uses Staff Name on both sides. A common naming convention reduces the chance of mistakes and makes the final output easier to read by end users.
19. Why PTKP is preserved as its own field
PTKP appears only in the Gross Up path and is retained in the final output. That suggests it plays a role in the tax classification or tax treatment of the staff record. The query keeps it visible rather than hiding it inside a total. That is useful because classification fields are often needed for explanation, audit, or reporting alongside numeric totals.

Preserving it in the output adds context to the cost numbers. It tells the user something about why the Gross Up calculation is structured the way it is.
20. The Taxable and Non-taxable BPJSTK split is a deliberate business rule
The query calculates employer BPJSTK contributions in two separate totals: one marked taxable and one marked non-taxable. This is a major design decision. It means the query is not simply summing all contribution components into a single bucket. It is preserving the tax treatment distinction explicitly.

That separation improves transparency. It allows a reviewer to see which components are considered taxable employer burden and which are not. In payroll modeling, that distinction can matter for compliance and cost analysis.
21. Why the taxable BPJSTK set differs between Gross Up and Gross
In the Gross Up branch, the taxable employer total includes JKK 0,24%, JKM 0,3%, JP 1%, and JHT 2%. In the Gross branch, the taxable employer total includes only JKK 0,24% and JKM 0,3%. That asymmetry shows the query is encoding distinct payroll logic for the two compensation models.

In other words, the query is not assuming that the same employer burden applies identically across the two source datasets. It adjusts the formula to match the model.
22. The non-taxable BPJSTK total is always the same structure
The non-taxable portion is computed from JP 2% and JHT 3,7% in both branches. That consistency suggests those components are treated in the same way regardless of gross-up or gross arrangement. Keeping them separate from the taxable total still matters, because the final output can distinguish the two categories without recomputing them later.

This separation also makes auditing easier. A reviewer can verify the classification and total structure directly from the table.
23. Why BPJSKes employer is handled separately from BPJSTK
Total BPJSKes Employer ... is created as its own column and not merged into the BPJSTK totals. That means the query treats health insurance as a distinct cost bucket. This is sensible because BPJSKes often needs to be visible on its own in payroll and insurance reporting. By keeping it separate, the output remains more informative and easier to analyze.

The query therefore preserves a meaningful domain distinction rather than forcing all employer contributions into one generic total.
24. The query removes intermediate payroll components after using them
Once the derived totals have been created, the original component columns are no longer needed in the output, so the query removes them. This is a common and wise data-modeling practice. It keeps the final report narrow and prevents the user from having to mentally separate construction fields from finished report fields.

Those detail columns were necessary during computation but are not necessary for the final comparison table. Removing them makes the output clearer and more presentation-ready.
25. The remove list differs by suffix because the source logic differs
The Gross Up branch removes a larger set of detail columns because it used more of them to build the taxable employer total. The Gross branch removes fewer columns because fewer source fields were needed in that path. That difference is another sign that the query is respecting the actual payroll model rather than forcing a one-size-fits-all structure.

This is good design. The removal step mirrors the calculation step rather than using a generic hardcoded cleanup list.
26. Total Staff Cost is the main business output of each branch
The final derived column in each prepared table is Total Staff Cost .... That is the headline metric. It aggregates the salary and relevant employer burdens into one cost number per staff row. In the Gross Up branch, it also includes PPh 21 and Tunjangan Asuransi in the total. In the Gross branch, the formula is slightly shorter because the underlying business logic differs.

This is the number that the rest of the comparison is built around. The query’s purpose is not merely to list components. It is to arrive at a comparable total cost figure for each compensation model.
27. Why the Gross Up total includes PPh 21 but the Gross total does not
This is one of the most important business-rule differences in the query. The Gross Up version explicitly adds PPh 21 into staff cost, which reflects the idea that tax cost is being absorbed into the employer cost structure. The Gross version omits it from the final staff cost formula. That difference is central to the meaning of “gross up” versus “gross.”

The query encodes the compensation logic directly, so the final comparison is not just arithmetic—it is a representation of how the two pay structures differ.
28. Why Tunjangan Asuransi is preserved as an included cost component
Tunjangan Asuransi is part of the selected columns and part of the final total. That means the query treats it as an intentional staff cost element rather than a side note. Since it is retained in both branches, it likely represents a recurring employer allowance or benefit that belongs in the cost comparison regardless of compensation model.

By keeping it in the calculation, the query ensures that the final total reflects the full staff burden rather than only base salary and mandatory contributions.
29. The source tables are read through named workbook objects
Gross_Up_Data and Gross_Data are loaded from Excel.CurrentWorkbook(). That means the query depends on named tables or named ranges in the workbook, not on sheet positions. This is a strong design choice because it decouples the query from physical worksheet layout. The data can move around visually as long as the named objects remain intact.

Named source objects also make the query easier to maintain. The input contract is explicit, which reduces ambiguity for future edits.
30. Why Table.Buffer is used on the prepared tables
Both prepared tables are buffered before the join. Buffering forces Power Query to materialize the table in memory, which can help when the same table is about to be read during a nested join and later expansion. It also helps lock in the prepared state before the merge executes.

This is a practical stability measure. It reduces the chance of repeated re-evaluation and can improve consistency in more complex query chains.
31. The merge is a left outer join because Gross Up is the primary table
Table.NestedJoin is performed with JoinKind.LeftOuter. That means every row from the Gross Up table is preserved, and matching rows from the Gross table are attached where the staff name matches. This tells you something about the intended reporting logic: Gross Up is the anchor dataset, and Gross is the comparison dataset appended to it.

That is the right direction if the Gross Up table is the main source of truth for the report. No Gross Up employee should disappear just because the Gross table lacks a matching row.
32. Why exact name matching is both useful and risky
The join uses exact Staff Name equality. That is deterministic and easy to reason about. If two names are exactly the same, they match. If not, they do not. That simplicity is valuable because it avoids accidental fuzzy matches that could pair the wrong people together. In payroll reporting, incorrect joins are usually worse than missing joins.

The downside is that small differences in spelling, capitalization, or trailing spaces could prevent a match. The query appears to prioritize integrity over guesswork, which is a reasonable choice in financial data.
33. Nested join before expansion is a standard Power Query pattern
Rather than flattening the Gross table immediately, the query stores the matched rows in a nested column called Gross. This intermediate structure is a normal Power Query pattern because it gives the author control over which fields get expanded and how they are named. The query only expands the columns that matter for the report.

This avoids clutter and preserves a clear transformation chain. It also keeps the final schema tight and intentional.
34. Why the expand step is explicit and selective
The query lists the exact Gross columns to expand, rather than expanding the whole nested table blindly. That is important because the goal is a carefully curated comparison table, not a dump of everything from the source table. The explicit list also helps preserve naming consistency and ensures that only the relevant cost fields appear in the final report.

This is good reporting discipline. It reduces noise and makes the output more readable.
35. The final type conversion is not just housekeeping
After joining and expanding, the query explicitly converts column types. This is important because Power Query can carry nulls, mixed values, or temporary type states through transformations. By the time the data reaches the final stage, the query wants every field to have a stable, known type. Text fields remain text, and all cost-related columns become numeric.

This is important for downstream use. It makes the output reliable for charts, pivoting, export, and any later formulas.
36. Why PTKP is typed as text instead of number
PTKP is set to text, which strongly suggests it functions as a code or category label rather than an arithmetic value. Typing it as text preserves its exact representation. That matters because values that look numeric may actually carry categorical meaning. In payroll contexts, tax classification fields often belong in this category.

Keeping it as text also prevents accidental numeric coercion that could erase leading zeros or alter a code-like value.
37. The final Table.SelectColumns defines the report view
The last step is a deliberate report-shaping stage. After all calculations and joins are complete, the query selects only the columns that should be visible to the user. Those include identity fields, Gross Up totals, and Gross totals. Internal helper columns, detail components, and temporary construction fields are not shown. That means the final output is both cleaner and easier to interpret.

This is the point where the query shifts from computation to presentation.
38. Why the final selection includes both Gross Up and Gross values side by side
Placing the two sets of values in the same row gives the user an immediate comparison view. There is no need to jump between sheets or manually line up two datasets. The report can be read horizontally: one staff member, two compensation models, same output row. That is exactly what a comparison report should do.

This design makes cost differences easy to inspect and analyze. It is practical and audit-friendly.
39. MissingField.Ignore in the final selection is another stability choice
Even at the end, the query remains forgiving. If a column expected in the final output is unavailable for some reason, MissingField.Ignore prevents the entire query from failing. That means the query prefers to return a mostly complete report rather than halt because of one missing field. In a production workbook, that can be a reasonable continuity choice.

The tradeoff is that schema issues may be less visible. But the query is clearly optimized for operational usability.
40. The query reveals a stable workbook contract
The entire transformation assumes a predictable workbook contract: two named source tables, staff rows keyed by name, a common payroll component vocabulary, and a final comparison layout with one Gross Up row matched to one Gross row. That means the query is not a free-form data wrangler. It is a workbook-specific payroll reporting engine designed for a known template.

This is good engineering when the template is controlled, because it reduces unnecessary complexity and improves reliability.
41. What the query is doing mathematically
Mathematically, the query performs row-wise aggregation followed by relational joining. Each row is converted from a set of component values into one or more totals. Then the rows are matched across the two datasets using an exact key. So the query is not doing statistical analysis or optimization. It is performing deterministic arithmetic and key-based consolidation.

This matters because the correctness standard is different. The query needs to preserve row identity and sum values accurately, not estimate or infer anything.
42. Why the query is well suited to payroll-style data
Payroll data is usually structured, repetitive, and sensitive to component definitions. This query matches that environment well. It handles repeated row patterns, uses fixed field names, distinguishes tax-related and non-tax-related contributions, and computes totals in a way that is easy to verify. That makes it suitable for salary comparisons, gross-up analysis, and staff cost modeling.

It is not a general-purpose data science query. It is a business-rule query for a structured workbook. That specialization is a strength.
43. Why the query is more robust than a worksheet formula approach
A worksheet formula approach would likely require many copied formulas, manual adjustments, and more room for broken references.
