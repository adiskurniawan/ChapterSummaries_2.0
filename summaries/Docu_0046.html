Option A • Safe Preprocessor in server.py (Immediate Mitigation)
Aspect
Detailed Explanation
Core Concept Add a lightweight pre-processing layer inside server.py that detects and escapes list markers (like 1. or -) when they appear at the start of table cells. It prevents Markdown from converting them into lists, while keeping visible text unchanged.
Location in Flow Implemented directly before _render_text_with_fallbacks() or within convert() just before text is sent to the Markdown or HTML renderers.
Advantages - Immediate fix — no deep refactor.
- Zero dependency changes (pure text-level).
- Reversible — remove one function to revert.
- No behavioral regressions expected.
- Safe even if renderer changes later.
Risks / Limitations - It’s a symptom-level fix, not a root cause solution.
- The escaping step runs per request, so it adds a small (negligible) CPU overhead.
- If the Markdown parser later improves internally, this preprocessor could become redundant.
Testing Plan - Input: a mix of normal tables, list-containing tables, and non-table lists.
- Verify: rendered HTML should show correct table structure, no unintended <ol>/<ul>.
- Edge Cases: nested tables, mixed tabs/spaces, escaped pipes.
Rollback Simply comment out or delete _escape_leading_list_markers_in_table_cells() and its invocation.
Implementation Effort ≈ 10–15 lines of Python (already conceptually written). Integration < 10 min.
Best For Quick mitigation, production-safe, requires no Markdown parser modification, minimal regression risk.

Option B • Renderer-Level Fix (Long-Term Semantic Solution)
Aspect
Detailed Explanation
Core Concept Fix the issue at the source — the Markdown → HTML conversion pipeline (e.g., convert.py, render.convert, or the Markdown renderer functions). Modify the parser so that table cell content is treated as literal text, not as a top-level Markdown block.
Location in Flow Deep inside convert.py or the renderer module (e.g., render/core_table.py, or wherever Markdown is parsed).
Mechanism - Adjust how each table cell’s text is parsed: treat each cell’s content as inline-only Markdown, disallowing block-level syntax such as lists, headings, or quotes.
- Alternatively, wrap cell text with “literal” or “verbatim” guards before rendering (e.g., escape list patterns before parser tokenization, not after).
Effect on Output Properly scoped Markdown behavior: tables render cleanly with <td> cells, but still allow inline formatting (like *italic*, **bold**). Lists inside cells stay literal unless explicitly escaped.
Scope of Impact Applies globally to all Markdown table rendering — ensures consistent literal treatment across the app.
Advantages - Root cause elimination.
- Cleaner, semantically correct behavior (Markdown lists belong outside tables).
- No pre/post string manipulation needed in server.py.
- More efficient (one parse pass).
Risks / Limitations - Requires touching Markdown parser internals or render conversion functions.
- Potential regression if Markdown library updates or if other modules depend on current behavior.
- Slightly longer development/testing cycle (~1–2 hours of careful testing).
Testing Plan - Test all Markdown constructs in cells (lists, headings, quotes).
- Confirm inline formatting still works.
- Compare output with Table Viewer to ensure parity.
- Regression-test non-table Markdown rendering.
Rollback Restore the original Markdown parsing function or flag; changes localized to one renderer.
Implementation Effort Moderate (1–2 hours), requires knowledge of render.convert internals.
Best For Permanent, clean architectural solution; projects expecting future Markdown extensions or renderer reuse.

Option B (Detailed)
Topic
Details
Summary Protect table-cell content from markdown list parsing by escaping leading list markers inside cells before markdown→HTML. Minimal, reversible, and renderer-level.
Rationale Parser-level fix is invasive. Escaping only leading markers inside cells preserves visuals and prevents list recognition. Safe and toggleable.
Scope of change render/convert.py (render_table), html_renderer.py (fallbacks), utils.py (shared helper). Add tests and optional config flag.
Helper (place) protect_leading_list_markers_in_cell(text) in utils.py. Conservative checks for <pre>, <code>, triple backticks, leading backtick, <script>, <textarea>.
Helper behavior If cell starts (after whitespace) with N. or N) or -/*/+ convert marker characters to numeric HTML entities. Leave code/pre/HTML blocks intact. Return string.
Integration point (render_table) Before calling markdown_to_html on a cell, call helper on the raw cell text. Preserve existing newline-collapse logic. Import helper defensively.
Integration point (html_renderer) Apply helper to each cell value (string) before stash/inline processing or before calling md. Honor fallback renderers.
Defensive import Try from utils import protect_leading_list_markers_in_cell and degrade gracefully if import fails.
Config toggle Add renderer config key protect_cell_list_markers default True. Check before applying helper.
Tests (file) tests/test_cell_list_protection.py with pytest cases: ordered, unordered, leading spaces, inline code, HTML blocks, multi-cell rows, CSV/pipe variants.
Example test asserts No <ol>/<ul> generated. Original marker characters still visible in rendered output. Code/pre blocks unaffected.
Edge cases Intentional sub-lists in cells blocked. Workaround: use explicit HTML <ul>/<ol> or a config flag to allow cell lists. Engines that run after entity conversion remain unaffected because entities break list tokenization.
Performance Per-cell regex checks. Negligible for typical tables. Heuristic: only process cells starting with whitespace + digit or bullet.
Migration & rollout Implement helper. Patch render.convert and html_renderer. Add tests. Run unit and integration tests. QA feature branch, then merge to main.
Validation checklist Unit tests pass. Regression tests for code/pre/HTML pass. Visual check in browser. End-to-end server→renderer test. Feature flag present.
Rollback & mitigations Remove helper calls or set config flag protect_cell_list_markers=False. Change is isolated to table-cell rendering.
Risk profile Low. No filesystem or external deps. Change is small, reversible, and well-scoped.
Why safe (legal/ops) No destructive I/O. No new dependencies. Preserves visual fidelity via entities.
Confidence High. Fix is local, testable, and addresses symptom at the renderer layer.
Notes for authors If a genuine list inside a cell is needed, authors must use explicit HTML lists or enable an opt-in config to allow cell lists.
