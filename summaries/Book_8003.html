Purpose Render per-table HTML fragments concurrently and write them atomically to disk.

Produce a consistent return structure so upstream renderer can stream fragments or fall back to inline rendering.
Primary responsibilities - Accept an iterable of table-like objects.
- Render each table to an HTML fragment using the preferred core_table.render_table when present, otherwise use a safe fallback.
- Write fragments atomically to a temporary fragments directory.
- Collect metadata (path, title id/text, sections) and row counts per fragment.
- Return (fragments_map, frag_row_counts, total_rows, warnings) for caller consumption.
Public API render_fragments_concurrently(all_tables, cfg, renderer, tmp_frag_dir, max_workers=1) returning:
- fragments_map: mapping idx → metadata and path.
- frag_row_counts: mapping idx → integer row count.
- total_rows: aggregate rows across fragments.
- warnings: list of warning dicts describing non-fatal issues.
Concurrency model Uses concurrent.futures.ThreadPoolExecutor with worker_count = max(1, min(len(tables), max_workers)).

Each table is submitted as a job that renders, writes atomically, and returns its metadata.

As jobs complete they are collected with as_completed and their results incorporated.
Render function selection Prefers core_table.render_table if available.

Wraps calls in try/except; on failure falls back to _simple_render_table which uses to_html if present, or a minimal placeholder otherwise.

This retains visual parity when the project provides a richer rendering helper.
Atomic write strategy Writes fragments via _atomic_write_text, which writes to a .tmp file then os.replace to swap into place.

If an optional _atomic_stream_write is available it attempts to use it; otherwise falls back to _atomic_write_text.

Ensures partial writes do not leave corrupt fragment files and supports cross-platform atomic replacement.
Metadata extraction _extract_meta_from_html parses a fragment string to find:
- first <h3 id="..."> to obtain title_id and title_text.
- any <div class="section-heading" id="..."> entries and extracts their text.

Defaults to Table{idx} when headers are absent. This allows building a Table of Contents.
Row counting _count_rows_in_fragment attempts to count <tr occurrences within a <tbody> when present, otherwise counts <tr across the fragment.

Provides an approximate row count used for diagnostics and to decide whether fragments contain content.
Error handling & warnings - Per-fragment exceptions are captured and returned as frag_warnings so a single table failure does not abort the whole run.
- General warnings list accumulates non-fatal issues: no tables provided, write failures, render failures.
- Logs info per rendered fragment (idx and row count).
Resilience & idempotence - Ensures tmp_frag_dir exists.
- Writes are idempotent via atomic replace semantics.
- If writing fails after rendering, a safe placeholder fragment is attempted.

Design favors completing as much work as possible and surfacing problems rather than failing fast.
Input handling - Attempts to coerce all_tables into a list(tables) for indexing.
- Falls back to iterative consumption if direct conversion fails.
- If no tables available, returns an empty fragments map and emits a warning.
Performance characteristics - Concurrency reduces wall-clock time for multi-core systems and I/O-bound fragment writes.
- Worker count is bounded by max_workers making resource usage controllable.
- Per-fragment memory footprint is limited because each job writes its output to disk immediately.
Scalability limits - ThreadPoolExecutor is suitable for CPU-light rendering tasks or I/O-heavy jobs.
- If render_table is CPU-heavy, consider ProcessPoolExecutor or a hybrid approach.
- The design streams fragments to disk but the coordinator still collects metadata and counts in memory; extremely large numbers of fragments may need paging.
Security considerations - The module does not interpret or sanitize fragment HTML; it treats rendered HTML as opaque text.
- Upstream renderers must enforce sanitization policies if inputs are untrusted.
- Fragment file write paths are constructed from tmp_frag_dir and table indices; no user-derived filenames are used, minimizing path-injection risk.
Observability & diagnostics - Returns structured warnings for programmatic checks.
- Logs success messages with fragment row counts at INFO level which aids run-summary reporting.
- Does not emit stack traces into warnings; exceptions are logged/debugged at the call site.
Interoperability contracts - Assumes renderer exposes render callable via getattr(renderer,"render", renderer).
- Assumes core_table.render_table signature (df, idx, markdown_to_html, left_width, right_width) when available.
- Upstream code expects fragments_map entries to include path, content, title_id, title_text, sections.
Test recommendations - Unit tests: single small table, missing to_html, and render_table raising exceptions.
- Concurrency tests: many small tables with max_workers>1 to assert ordering and no data loss.
- Failure tests: simulate _atomic_stream_write raising and ensure fallback writing succeeds.
- Metadata tests: fragments with and without <h3 id=...> and section divs.
Immediate, low-risk improvements - Add a per-fragment debug log when a placeholder is written to make failures visible.
- Ensure warnings include exception messages (with controlled verbosity) to aid triage.
- Normalize returned fragments_map keys to a deterministic sequence even if jobs complete out-of-order (current code uses idx so this is OK but ensure callers rely on sorted keys).
Medium-term enhancements - Add optional back-pressure: limit in-flight fragment files if disk I/O becomes a bottleneck.
- Add a pluggable executor strategy (threads vs processes) chosen by a config flag based on workload.
- Provide an optional manifest file describing produced fragments and counts for downstream tooling.
Corner cases & gotchas - If all_tables is a generator that can only be consumed once, list(all_tables) may exhaust it unexpectedly; current code falls back to iterative collection but document this behavior.
- If render_table writes its own files or relies on shared global state, concurrent execution may cause race conditions; prefer pure functions or per-job isolation.
Summary judgement Well-constructed concurrency helper that balances robustness, atomicity, and graceful degradation.

Minor additions to diagnostics and optional executor strategy would improve operability under varied workloads.

Purpose Provide reliable, cross-platform atomic write and directory utilities used by the rendering pipeline. Act as a small compatibility layer that re-uses implementations from io_utils when available and otherwise supplies safe defaults.
Primary responsibilities - Offer _default_write_atomic(path, writer) to write files atomically via a temporary file and os.replace.
- Ensure directory existence with _ensure_dir_path.
- Validate files with _is_valid_file.
- Provide tmp_frag_dir_for(out_dir) to create a unique fragment directory.
- Re-export _atomic_write, _atomic_stream_write, and _ensure_dir selecting project-provided implementations when present.
Design intent Small, well-scoped helper module that centralizes IO primitives used by higher-level render code.

Intentionally conservative: prefer built-in safe defaults while allowing projects to provide optimized implementations in io_utils.
Atomic write semantics _default_write_atomic writes to path.tmp then os.replace(tmp, path) to ensure atomic replacement across platforms that support os.replace.

This prevents partially written target files on crashes or interrupts and is the recommended pattern for atomic updates.
Directory handling _ensure_dir_path uses Path.mkdir(parents=True, exist_ok=True) with an os.makedirs fallback.

This dual attempt maximizes compatibility with non-standard Path objects and older environments.
Temporary fragment directory tmp_frag_dir_for(out_dir) creates a .frags_<ms timestamp> subdirectory under out_dir and ensures it exists.

Provides a safe, non-colliding location for fragment files used by the fragment renderer. On failure it gracefully returns out_dir.
Implementation selection / re-export At import time attempts to import _atomic_write, _atomic_stream_write, and _ensure_dir from .io_utils.

If present those implementations are preferred. Otherwise the module falls back to its built-in defaults (_default_write_atomic, None for stream write, and _ensure_dir_path).

Consumers import _atomic_write and get the best available implementation.
Error handling & resilience Functions use try/except only where necessary and prefer explicit fallbacks rather than swallowing exceptions silently.

_is_valid_file intentionally returns False on any exception to avoid raising in health checks.

Overall behavior is to degrade gracefully if platform or fs quirks occur.
Portability Uses pathlib and os.replace which are portable across modern POSIX and Windows Python runtimes.

Fallbacks to os.makedirs ensure compatibility with older or non-standard Path-like objects.
Security & safety Atomic replacement minimizes window for serving partially-written content.

Temporary file naming uses a .tmp suffix in the same directory which avoids cross-device rename issues.

tmp_frag_dir_for uses a millisecond timestamp for uniqueness; collisions are extremely unlikely but not cryptographically random—suitable for local workflows but not for adversarial multi-tenant contexts.
Performance characteristics IO-bound operations only.

_default_write_atomic requires writing the full content to a temp file then renaming; this doubles write traffic but is necessary for atomicity.

For extremely large outputs consider streaming atomic implementations (_atomic_stream_write) or filesystem-specific atomic helpers.
Maintainability Module is small and easy to read.

Good separation of concerns and clear fallback order.

Consider adding brief docstrings to the exported names to aid discoverability.
Testing recommendations - Unit tests for _default_write_atomic verifying atomicity semantics (simulate interruption scenarios).
- Tests for _is_valid_file with zero-byte files and permission errors.
- Tests for tmp_frag_dir_for ensuring unique directories and correct fallback when creation fails.
- Integration test ensuring _atomic_write points to project-provided implementation when io_utils is present.
Immediate, low-risk improvements - Expose a tiny diagnostics helper that reports which implementation is active (default vs project-provided) to aid debugging.
- Document expected signature for a pluggable _atomic_stream_write so authors of io_utils know how to integrate.
Summary judgement Pragmatic, reliable IO helper module.

It provides correct atomic write semantics and robust directory helpers while permitting optimized overrides.

Minor documentation and diagnostics additions would improve developer ergonomics without changing behavior.

Chapter 3 — Seasonality Analysis
Topic
Details
Purpose Render a single table-like object into a self-contained HTML fragment suitable for inclusion as a table "card" in the viewer.

Provides a conservative fallback rendering path if project-specific table renderers are unavailable.
Primary responsibilities - Accept a table data object and produce accessible HTML with header, body, controls, and row counts.
- Prefer the richer convert.render_table when present to preserve visual parity with project styling.
- Sanitize and format cell contents using the provided markdown renderer or safe fallbacks.
- Expose hooks and structure that the fragment pipeline and final page assembly expect.
Public API render_table(df, idx, markdown_to_html, left_width, right_width)

Inputs: arbitrary table-like df, an integer index, an inline markdown-to-HTML callable, and layout width percentages for left/right columns.

Output: HTML fragment string containing controls, table, and row-count block.
Render selection logic 1) If convert.render_table is available call it and return its output.
2) Else, if df looks like a pandas DataFrame, attempt df.to_html(index=False, border=0, escape=False) and return a simple heading + that HTML.
3) Otherwise, normalize the input into rows and optional header then manually compose HTML for controls, thead, tbody, and row counts.
Data normalization _rows_from_iterable attempts multiple safe strategies:
- Pandas-like objects via .columns and .itertuples.
- List-of-dicts detects keys as headers.
- List-of-lists and single-value rows handled uniformly.
- If first row looks like a header (strings and multiple rows) it is treated as header and skipped from body.

Design goal: accept many input shapes without raising.
Cell rendering & sanitization - markdown_to_html input is first normalized into a callable via _get_md_callable.
- _render_cell_text runs the inline renderer, falls back to html.escape on errors, and then runs sanitize_html to strip or allow safe tags.

This layered approach isolates inline formatting from structural HTML and reduces XSS surface when sanitize_html is available.
Structure & accessibility - Table fragment includes a header <h3 id="...">, a control toolbar with buttons, the table with <thead> and <tbody>, and a .row-count element.
- Column headings are rendered as <th><button class="sort-btn">...</button></th> to make sorting controls keyboard-focusable.
- Data cells include a data-label attribute for responsive or assistive presentation.

These choices improve keyboard access and responsive labeling for screen readers.
Controls & JS integration Buttons reference global UI functions (toggleTable, copyTablePlain, copyTableMarkdown, exportTableCSV, exportTableJSON, exportTableXLSX, exportTablePDF).

The fragment assumes the page-level JS shim or full script.js will wire handlers.

If script.js is missing the renderer injects an inline shim elsewhere to ensure minimal interactivity.
Header detection & labeling When a header is present it is rendered with inline markdown conversion for each header cell.

When absent the renderer generates default column labels like "Col 1".

Column labels are used both visually and as data-label for responsive layouts.
Fault tolerance - Try/except guards around all major operations.
- On unexpected failure the function logs the exception and returns a short failure fragment.
- Avoids raising to keep the broader pipeline resilient.
Performance characteristics - Fast for small-to-medium tables.
- Manual HTML composition is O(rows × cols) string work; overhead grows linearly.
- Uses in-memory assembly of the entire fragment before returning; very large tables will increase memory usage.

For very large datasets prefer the fragment pipeline that streams per-table fragments to disk.
Security & sanitization risks - Sanitization depends on sanitize_html availability. If absent, the fallback uses html.escape which is safe but removes intended inline HTML.
- The function passes escape=False to pandas.DataFrame.to_html when used; upstream code must ensure DataFrame cell content is already sanitized or trusted.
- Data-labels and header content are html-escaped or sanitized, reducing injection risk for those attributes.
Compatibility & extensibility - Pluggable: prefers project convert.render_table if available.
- Accepts markdown renderer objects or simple callables.
- Keeps signature stable so other modules can call it directly or via adapters.
Testing suggestions - Unit tests for many input shapes: pandas DataFrame, list-of-dicts, list-of-lists, single-column lists, and header-as-first-row.
- Tests for markdown rendering integration including failure modes where markdown callable raises.
- Accessibility checks: verify aria attributes, button focusability, and presence of data-label.
- Stress test with large row/column counts to measure memory and rendering time.
Immediate, low-risk fixes - Add small docstrings to _get_md_callable, _rows_from_iterable, and _render_cell_text to aid maintainers.
- Ensure pandas.to_html path documents the safety expectation about cell content or forces escape=True unless sanitized.
- Log at debug level when falling back to manual rendering to make behavior visible in runs.
Medium-term improvements - Stream tbody generation for very large tables instead of building full fragment in memory.
- Offer a max_cells or pagination option so the renderer can render partial tables and delegate full-table browsing to virtualization.
- Provide an explicit sanitizer configuration allowing safe tags while preventing risky attributes like on* handlers.
UX considerations - Generated controls match the page shim so users get consistent actions (copy/export/sort).
- data-label inclusion improves readability on narrow viewports and assists screen readers.
- Column sort buttons are present but their state and icons depend on script integration; consider ARIA state updates when sorting occurs.
Summary judgement Practical and well-engineered fallback table renderer.

It accepts many data shapes, integrates with optional project renderers, and emphasizes safety and accessibility.

Small documentation and observability improvements plus optional streaming for big tables will further strengthen it.

Chapter 4 — Interim Disclosure Examples

Details
Purpose Small collection of conservative utility helpers used across the renderer.

Provides ID sanitization, output path normalization, minimal HTML whitelist pass-through, SRI attribute generation, fragment row detection, and a tolerant "take-and-stream" utility for iterables.
Public surface / functions - _sanitize_id(x, idx=None) → normalized id string safe for HTML ids.
- _sanitize_output_path(x) → defensive normalization of an output path string.
- _sanitize_html_whitelist(s) → placeholder pass-through; real work delegated to sanitize.py.
- _sri_attr(url, cdn_integrity=None) → returns integrity attribute when available.
- _frag_contains_rows(content) → quick check whether a fragment has at least one <tr> inside <tbody>.
- _take_and_stream(rows_iterable, n) → returns (buffer, iterator) where buffer has up to n items and iterator yields the remainder, tolerant of many input shapes.
Design intent Defensive and minimal.

Functions avoid throwing by catching exceptions and returning safe defaults.

They are helpers, not policy enforcers; heavier sanitization and validation happen elsewhere.
Behavioral details - ID sanitizer replaces non-alphanumeric characters with hyphens, collapses multiple hyphens, and produces a fallback id-{idx} when input empties out.
- Output path sanitizer currently is a thin wrapper returning the given string or a safe filename on error.
- HTML whitelist is a no-op placeholder here to allow callers to depend on it even if sanitize.py is unavailable.
- SRI attr builder returns a string with integrity and crossorigin when cdn_integrity maps the exact URL.
- Fragment row detector looks for <tbody>\s*<tr to avoid false positives.
- take_and_stream robustly handles iterators, lists, generators and recovers when partial reads fail.
Correctness risks - _sanitize_output_path does not validate or normalize paths (no path traversal checks, no expansion). Caller must ensure path safety.
- _sanitize_html_whitelist is a placeholder; relying on it for security is unsafe.
- _sri_attr requires exact URL key matches in the provided dict; relative vs absolute mismatch may miss integrity entries.
- _frag_contains_rows is heuristic-based and may mis-classify fragments that use different table markup or non-<tbody> structures.
Security considerations - Nothing here sanitizes user content meaningfully; do not use _sanitize_html_whitelist as a security control.
- _sanitize_id prevents many injection vectors in id attributes but does not enforce uniqueness; callers should ensure unique ids.
- _sanitize_output_path should be extended or callers must perform path validation when outputs are exposed or derived from user input.
Performance characteristics - All helpers are O(length-of-input) or O(1).
- _take_and_stream may materialize a list in fallback mode which increases memory use if the iterable is large; this happens only on certain exception paths.
- _frag_contains_rows uses regex on string content; cost is linear in fragment size but acceptable for typical fragment lengths.
Robustness & fault tolerance - Each function catches exceptions and returns a safe default.
- This prevents single malformed inputs from bubbling up and breaking pipeline runs.
- The tradeoff is weaker immediate observability of root causes unless logs are added at call sites.
Integration contract - Designed to be used by renderer modules that already perform higher-level validation.
- _sanitize_id and _sri_attr are intended to be small deterministic helpers callers can rely on.
- _take_and_stream is useful for fragment writers or streaming renderers that need the first N rows and the remaining iterator.
Test recommendations - Unit tests for _sanitize_id with unicode, empty, and punctuation-heavy inputs.
- Tests for _sri_attr with exact/near-miss URL keys.
- Tests for _frag_contains_rows with fragments containing <tbody>, fragments with <tr> only, and fragments with no table markup.
- Tests for _take_and_stream covering generators, lists, finite and empty iterables, and failure paths that force fallback materialization.
Immediate, low-risk improvements - Implement path normalization and validation in _sanitize_output_path (resolve, forbid .., ensure parent exists or is within sandbox).
- Make _sanitize_html_whitelist explicitly raise or warn when used if sanitize.py is missing to avoid silent security gaps.
- Add optional logging on exception branches to aid debugging while preserving safe defaults.
Medium-term enhancements - Accept a sanitizer callable via config or dependency injection rather than a module-level placeholder.
- Make _sri_attr tolerant to URL variants (strip timestamps or query strings, support base-name lookup).
- Provide an id-registration helper to ensure uniqueness across a page (helpful for automated id generation).
Usage notes & gotchas - These helpers are intentionally conservative; do not assume they perform full validation.
- Always pair _sanitize_html_whitelist with a real sanitizer in production.
- _take_and_stream can consume an iterator; if the caller needs to reuse the original iterable, materialize it first.
Summary judgement Practical, minimal utilities that enable safe defaults for the rendering pipeline.

They prioritize resilience and simplicity.

Key action: harden _sanitize_output_path and replace the HTML-whitelist placeholder with a documented sanitizer injection point before exposing this renderer to untrusted inputs.

Purpose This file exists as a compatibility layer. It exposes a simplified and unified API entry point by re-exporting core rendering functions. Its main goal is to provide external callers with stable names (render_html, render_table, render_page) without requiring them to know the internal module structure.
Imports It imports two key functions: render_html from core_renderer (for full HTML document rendering) and render_table from core_table (for individual table rendering).
Alias Creation Defines render_page = render_html, meaning that both names point to the exact same function. This allows older or external codebases to call either name without breaking.
Public API Exposure The __all__ list explicitly declares which names should be available when another module uses from core_compat import *. This ensures only render_html, render_table, and render_page are exported as the official interface.
Design Implications - Keeps a backward compatibility bridge, useful if function names changed in earlier versions.
- Provides clarity for external integration: consumers of the library can just import from core_compat and don’t need to navigate the internal module layout.
- Maintains future flexibility: internal refactors won’t break downstream usage, as long as core_compat keeps stable references.
Portability & Stability This pattern is highly portable: no OS or Python-version dependencies. Stability is also strong, since it’s just aliasing and re-exporting functions without modification.
Security No direct input/output processing, no file or network access—security risk is negligible.
Performance Zero overhead: simple imports and one assignment alias.
User Experience Improves developer experience by offering a clear, stable import path (core_compat.render_page) regardless of deeper refactors. Reduces coupling to internal structure.
Recommendation - Maintain this file as a stable interface contract.
- If more functions are likely to be renamed in the future, extend this file as the official compatibility surface.
- Document this module clearly as the preferred entry point for external consumers.
