1. What this query is doing at the highest level
This query is a workbook-wide consolidation routine. It searches the current workbook for every named table or named range whose name begins with SC_Data_, loads each one, standardizes it to one common schema, and appends them into a single output table. The result is one combined dataset rather than many separate blocks.

That means the query is acting like a controlled ingestion layer for multiple similar staff-cost tables. It is not just reshaping one table. It is normalizing many source objects and merging them into a standard analytical output.
2. Why this is more than a simple append
A simple append would assume the source tables are already compatible. This query does more than that. It repairs the shape of each source table before combination, trims text noise, enforces types, adds provenance, and handles missing columns safely. The append happens only after all source objects have been normalized.

That structure makes the combined result dependable. The query is therefore a transformation pipeline, not a raw concatenation.
3. The Source = Excel.CurrentWorkbook() line is the discovery point
Excel.CurrentWorkbook() gives the query access to the workbook’s named tables and named ranges. This is the key discovery mechanism. Instead of pointing to one fixed sheet range, the query examines the workbook object list and filters from there.

That approach is especially useful when the workbook contains repeated blocks of similar data. It lets the query scale with the workbook as long as the naming convention is respected.
4. Why object-based sourcing is powerful in Excel
Named objects in Excel are stable references. They survive sheet rearrangements better than direct cell references. Because the query reads workbook objects by name, it is less fragile than a formula-based or cell-address-based system. The workbook can move visually without breaking the consolidation logic.

This is a good fit for operational workbooks that evolve over time but keep a consistent naming contract.
5. The SC_Data_ prefix is the query’s inclusion rule
The query only includes workbook objects whose names start with SC_Data_. That prefix acts like a namespace. It says: these are the source tables I want; everything else should be ignored. This is a very effective way to prevent accidental ingestion of unrelated sheets, helper tables, or calculation ranges.

The result is targeted consolidation rather than blind ingestion.
6. Why a prefix convention matters more than hardcoding names
If the query listed every source table by exact name, every new table would require a code update. With a prefix convention, new qualifying objects are picked up automatically as long as they are named correctly. That makes the design scalable and easier to maintain.

For a workbook that may gain new monthly or departmental data blocks, that is a practical advantage.
7. ExpectedColumns defines the canonical schema
The large ExpectedColumns list is the schema contract for the final output. It specifies exactly which fields every source object must be normalized to. This list includes identity fields, period fields, job data, salary data, reimbursement data, percentages, and indirect cost fields. That means the query is standardizing a rich cost model, not a simple flat table.

This is important because the query can only combine tables cleanly if they all share the same field structure.
8. Why the schema list is so detailed
The query appears to support a staff-cost model with many cost categories. The list includes not only salary and total staff cost but also detailed breakdowns for tax, BPJS, backpay, contract compensation, JSHK, bonus at contract end, insurance, Jobstreet reimbursement, and THR. That suggests the workbook is designed for a real compensation or manpower budgeting model with multiple allocations.

The schema is therefore domain-specific and intentionally comprehensive.
9. The schema includes both amounts and percentages
Fields like Salary Pct, Tax Pct, and THR Pct appear alongside monetary values. This means the query preserves not just raw costs but also allocation rates or cost proportions. That is valuable because percentages often explain how amounts were derived or distributed.

By retaining both, the output becomes more auditable and more useful for later analysis.
10. Source is not just a technical field; it is a provenance field
The query later adds a Source column to each row, storing the original workbook object name. This is useful because once many tables are combined, the original origin of each row would otherwise be lost. Source preserves where each row came from.

That means the output remains traceable. A user can later filter or audit by source object.
11. Why provenance matters in a combined table
Once multiple source blocks are appended, it becomes difficult to see which rows came from which original range unless that metadata is retained. Source solves that. It lets the final table support auditing, debugging, and source-level filtering.

In combined workbook reporting, traceability is often as important as the numbers themselves.
12. TextColsToTrim identifies the fields most likely to have whitespace noise
The query trims Source, Period, Group, Name, Job Title, and EC Sequence. These are fields that are typically entered or imported as text and are vulnerable to leading or trailing spaces. Such spaces can break matching, grouping, or even make identical labels look different in the output.

The query trims only these fields because they are the most likely to need text cleanup. Numeric fields do not need the same treatment.
13. Why trimming matters in spreadsheet data
Excel data often carries hidden formatting issues. A row may look clean to the user but still contain trailing spaces that affect equality checks or grouping. By trimming these text columns, the query avoids subtle data-quality bugs.

This is especially important for Name, Group, and Job Title, where whitespace can create duplicates or mismatches.
14. TextTrim is a defensive helper
TextTrim converts any non-null value into text, trims it, and returns nullable text. If the value is null, it stays null. This means the query does not force missing data into an empty string, which would erase the distinction between missing and blank values.

That is a good Power Query practice because nulls and empty strings have different meanings.
15. Why Text.From(x) is used inside the trim helper
The conversion to text allows the function to handle values that may not already be text. That makes the helper more robust. It can trim text-like content even if the source cell was imported in a different type representation.

This reduces fragility in workbook data, which often mixes types more than expected.
16. TrimTransforms converts a simple list into a reusable transformation definition
List.Transform(TextColsToTrim, each {_, TextTrim, type text}) produces the list structure expected by Table.TransformColumns. This is a compact way to say: for each listed column, apply the same trimming function and treat the result as text.

It is cleaner and easier to maintain than manually writing separate transform instructions for each column.
17. TypeConversions is the schema enforcement layer
This list assigns the final type of each column. No becomes Int64.Type, the date field becomes type date, and the cost fields become type number. That is important because the query wants the final output to behave predictably as a typed analytical table.

Types matter later when the table is sorted, filtered, grouped, or used downstream in other formulas or queries.
18. Why explicit typing is better than implicit inference
Power Query can infer types, but inference can vary with source content. Explicit typing removes ambiguity. It tells the engine exactly how each column should behave. That is especially important in a reporting model where date and number columns must remain reliable across refreshes.

This reduces the risk of accidental text values being treated like numbers or vice versa.
19. PresentSCObjects is the first filter gate
This step filters the workbook object list down to only those with names beginning SC_Data_. It excludes everything else. That is how the query decides which workbook objects are candidates for loading.

This gate is critical because it prevents unrelated workbook objects from being appended into the combined dataset.
20. Why the null check is included before the prefix test
The expression checks [Name] <> null first. That is a small but useful safeguard. It avoids applying Text.StartsWith to a null value, which could cause errors or unnecessary failures. The query prefers to filter safely from the start.

This is another sign of defensive workbook programming.
21. LoadRange is the per-object normalization function
This function takes one workbook object record and turns it into a normalized table if possible. It is the most important internal function in the query because every source object must pass through it before being appended.

The function is responsible for promoting headers, adding the source label, selecting the canonical schema, trimming text fields, and applying types.
22. Why LoadRange accepts a record rather than a table
Table.ToRecords(PresentSCObjects) converts the filtered workbook object list into records. Each record contains metadata like the object name and the content table. That makes it easy for LoadRange to extract both the table data and its origin label.

This is ideal for adding a provenance column later.
23. name = r[Name] captures the object name for lineage
The loader stores the workbook object name in a variable so it can be written into the final table as Source. This is how the query preserves lineage. Without that step, the final combined result would lose track of the original source object.

That lineage is especially useful when multiple source blocks exist in one workbook.
24. raw = r[Content] gets the actual table data
The Content field of each workbook object is the raw table or range. This is the data that the query transforms. It may already have headers or may still need promotion depending on how the workbook object was created.

Everything that follows is about standardizing this raw content.
25. Table.PromoteHeaders turns the first row into field names
Promoting headers makes the raw content behave like a real table with column names. The query uses PromoteAllScalars = true, which helps when the header row includes non-text scalars that need to be interpreted as field names.

This is important because the query expects structured column names before it can select, trim, and type them.
26. Why header promotion is done inside the loader
Each source object can be normalized individually. This is better than assuming all source tables are already in the correct structure. If one table is a raw range and another is already a table, they still get treated consistently because the loader repairs each one before combination.

This is a strong design choice for mixed workbook inputs.
27. withSource = Table.AddColumn(promoted, "Source", each name, type text) adds origin tracking
This step inserts the source object name into every row. That means the final output can trace each row back to the source object from which it originated. It also makes it possible to group or filter by source after consolidation.

This is a small line with a big reporting value.
28. Why the source column is added before selection
Because Source is added after header promotion and before the select step, it is guaranteed to exist in the final standardized table. That is useful because it is not something the original source necessarily contained. The query creates it as a consistent metadata field for all loaded tables.

This ensures the final output always retains lineage information.
29. Table.SelectColumns(..., ExpectedColumns, MissingField.UseNull) is the schema alignment step
This is one of the central lines in the query. It forces every loaded source table to fit the same expected field list. If a source object lacks a column, Power Query inserts that field as null rather than failing. If it has extra columns, they are discarded.

The result is a standardized table shape for every source. That is what makes the later append possible.
30. Why MissingField.UseNull is the right compromise here
Some source blocks may not contain every possible field. Instead of stopping on those tables, the query fills missing fields with null and continues. That makes the consolidation process robust to partial input while still preserving a consistent schema.

This is a practical choice in workbook models where source tables are similar but not perfectly identical.
31. trimmed = Table.TransformColumns(selected, TrimTransforms) cleans selected text fields
Once the schema is aligned, the query trims the text fields that matter most. This ensures the final combined table does not carry hidden whitespace into the output. The trimming happens only after the selection step, which means it runs on the canonical set of fields rather than arbitrary source extras.

That keeps the normalization focused and predictable.
32. typed = Table.TransformColumnTypes(trimmed, TypeConversions) finalizes each loaded table
This step turns the selected and trimmed table into a properly typed analytical table. It gives the query stable behavior for dates, integers, and numeric values. That matters because the next step will combine tables, and combined tables are only as good as the consistency of their component tables.

Explicit typing also helps with downstream reports and calculations.
33. Why the LoadRange function uses try ... otherwise null
The query does not allow one bad source table to break the whole consolidation process. If any step in the loader fails, the function returns null. That means the bad table is skipped later, while valid tables still contribute to the final result.

This is a very practical error-isolation strategy for multi-source workbooks.
34. The failure scope is per-object, not global
Because each object is loaded independently, a single malformed table does not ruin the entire workbook query. That is a major reliability advantage. It lets the query continue processing the good sources even if one source is broken or malformed.

This is better than a hard fail in environments where workbook inputs are managed manually.
35. Ranges = List.RemoveNulls(List.Transform(...)) gathers only successful loads
This step applies LoadRange to every candidate workbook object and removes any null results. The outcome is a list of valid normalized tables. That list becomes the input to Table.Combine.

This means the query automatically discards bad source tables while preserving good ones.
36. Why List.RemoveNulls is necessary after error-tolerant loading
Since LoadRange returns null for failures, the query must filter those nulls out before combining. Otherwise, the append step would not receive a clean list of tables. This is the glue that connects tolerant loading with a valid final append.

It is a small structural step, but it is essential.
37. The query raises a custom error if no valid source tables were found
If the resulting list is empty, the query does not return a blank table. It raises a descriptive error: no named ranges or tables starting with SC_Data_ were found. That is helpful because it tells the user exactly what setup problem needs fixing.

This is better than a silent failure or an empty result that looks valid but contains nothing.
38. Why a custom error is better than an empty table in this case
An empty output could be misread as “no data today” instead of “wrong workbook setup.” The custom error avoids that confusion. It makes the problem explicit and actionable.

This is a good user-experience choice for workbook automation.
39. Table.Combine(Ranges) is the final append step
Once all source tables are normalized, trimmed, typed, and filtered for success, the query appends them into one combined table. Because each table already follows the same schema, the combine step is straightforward and stable.

The append is therefore the final structural assembly of the pipeline.
40. Why normalization must happen before combination
If the query combined raw source tables first, the result would likely contain inconsistent columns, mixed types, and extra fields. Normalizing first ensures the combine step is simple and predictable. This sequence is the correct order for robust consolidation.

That is a key design point: repair first, append second.

1. What this query is in operational terms
This is a full ingestion-and-curation Power Query designed to turn a folder of messy transaction files into one standardized reporting table. It reads multiple file formats, normalizes the schema, enriches the rows with optional dimensions, applies data-quality checks, detects duplicates, optionally deduplicates to the latest record per transaction key, and finally returns a clean analytical output.

It is not a simple import query. It is a complete ETL pipeline embedded inside Power Query.
2. The core design philosophy
The query is built around the idea that raw files are unreliable, inconsistent, and incomplete. Instead of assuming every source file is perfect, it anticipates schema drift, type mismatch, late data, missing tables, bad text, duplicated rows, and parsing failures.

That is why the code has so many helper functions. Each helper reduces fragility in one specific area so the whole pipeline can survive imperfect real-world inputs.
3. Why the code is organized into named blocks
The query is segmented into configuration, helper functions, workbook dimension handling, file discovery, schema normalization, type coercion, derivations, joins, data-quality flags, duplicate detection, deduplication, and final shaping. Each section has a distinct responsibility.

This structure matters because Power Query can become unreadable very quickly if all logic is mixed together. Here, the author has separated logic into phases that mirror a real data-engineering workflow.
4. Why the Config record is important
Config acts as a single control panel for the query. It holds the source folder, archive folder, accepted extensions, CSV parsing settings, culture, preferred sheet names, incremental lookback window, dedupe behavior, archive filtering behavior, and debug mode.

This is a strong design choice because it makes the query easier to maintain. The behavior of the whole pipeline can be changed by adjusting one record rather than rewriting many lines of logic.
5. SourceFolder and ArchiveFolder as operational boundaries
The source folder tells the query where active files live. The archive folder gives the query a place to exclude if archived files should not be ingested.

This separation is useful in production setups where active and historical files are stored in the same broad file structure but should not always be treated the same way.
6. Why accepted extensions are explicit
The query only accepts .csv, .xlsx, .xls, .xlsb, and .txt. That limits the ingestion surface to known data-bearing file types and avoids accidental parsing of unrelated files.

This is a small but important guardrail. It reduces garbage input before the parser even starts.
7. Why CSV delimiter and encoding are configurable
CSV files vary across systems, especially in delimiter choice and encoding. By placing the delimiter and encoding in the config, the query can be adapted for different source systems without touching the main ingestion logic.

That is a practical sign that the author expects the pipeline to be reused in more than one workbook or environment.
8. Why Culture = "en-US" matters
Culture affects how numbers and dates are interpreted. A culture-sensitive parser is important because the same text can be read differently depending on locale conventions for decimal separators, date formats, and thousand separators.

By setting culture explicitly, the query reduces ambiguity and keeps parsing behavior predictable.
9. The purpose of PreferredSheetNames
Workbook files often contain many sheets, but only one or two are the real data source. The preferred sheet list tells the query which names should be favored first, such as data, raw, transactions, sheet1, and input.

This is a very practical heuristic. It helps the query avoid loading title sheets, summary tabs, or decorative tabs by mistake.
10. Why IncrementalDaysBack is operationally useful
This setting defines how far back the query should look when filtering recent files by modified date. It lets the pipeline behave like a rolling incremental ingestion process rather than a full folder scan every time.

That helps both performance and control. Large folders remain manageable when the query only considers a recent window of files.
11. KeepOnlyLatestPerKey as a business switch
This setting decides whether the final result should collapse duplicates and keep only the latest record per transaction key. That is a major business behavior toggle, not just a technical parameter.

It means the same query can support either audit-style full retention or reporting-style latest-state extraction.
12. Why UseArchiveFilter is optional
Not every deployment needs archive exclusion. Some file structures separate active and archived data cleanly; others do not. Making archive filtering optional allows the query to adapt to both styles.

This is a good example of avoiding hardcoded assumptions that might not fit every workbook.
13. Why DebugMode exists
Debug mode lets the query preserve more diagnostic detail instead of hiding error rows. That is extremely useful while developing, testing, or troubleshooting the pipeline.

In production, the user may want only clean rows. In development, the user may want the error rows and metadata preserved so they can see what failed.
14. RefreshUtc as a stable timestamp anchor
The query captures the exact UTC refresh time once and then reuses it throughout the pipeline. That means all rows produced in the same refresh share the same ingestion timestamp.

This is important for auditability. It tells you when the dataset was built without relying on volatile recalculation inside later steps.
15. MinModifiedDate and incremental scope
The minimum modified date is calculated by subtracting the configured lookback window from the current date. Files older than that cutoff are excluded from active ingestion.

This is a performance and governance feature. It keeps the query focused on recent data and prevents unnecessary reprocessing of old files.
16. Why the query starts with helper functions
Helper functions turn repeated micro-patterns into reusable building blocks. Instead of writing custom conversion, normalization, and fallback logic over and over, the query defines them once and reuses them throughout the pipeline.

This is what makes the query look long but still maintainable. The length is mostly due to structure, not duplication.
17. fnTry as a safe function wrapper
This helper calls another function and returns either the result or a default value if an error occurs. It is a generic safety wrapper.

That is useful when a step may fail for benign reasons and the query should continue rather than halt. It is a tiny function with broad defensive value.
18. Why fnTry is useful in Power Query specifically
Power Query can fail on type issues, malformed text, missing rows, or unexpected source structures. A wrapper like fnTry helps keep those errors local rather than letting them crash the entire query.

In a multi-file ingestion pipeline, controlled failure handling is a major advantage.
19. fnCleanText as a universal text sanitizer
This function converts a value to text safely, trims whitespace, cleans nonprinting characters, and turns blank strings into nulls.

That is essential because messy input often contains invisible characters, leading/trailing spaces, or blank text values that look empty but are not logically null.
20. Why blank-to-null conversion matters
In Power Query, blank text and null are not always treated the same way. Converting empty text to null makes downstream logic cleaner, especially for joins, filters, and required-field checks.

It reduces ambiguity and prevents “empty but not really empty” data from slipping through quality rules.
21. fnNormalizeName as the schema-shaping engine
This helper forces a header or field name into a canonical internal form. It lowercases the text, cleans it, removes line breaks, replaces spaces and punctuation with underscores, strips brackets and quotes, collapses repeated underscores, and trims leading or trailing underscores.

This is one of the most important helpers in the entire query because schema drift usually begins with inconsistent naming.
22. Why schema normalization is a central strategy here
Real-world source files do not always use the same header spelling, spacing, or punctuation. One file may use Invoice No, another invoice_no, another Invoice-No, and another Invoice.No.

Normalizing the name to a shared internal pattern makes later alias mapping and joins far more reliable.
23. fnNormalizeHeader as a thin wrapper
This helper simply delegates to fnNormalizeName. It exists for semantic clarity, because headers and general names are conceptually related but not always discussed the same way in maintenance work.

That may seem minor, but it helps readers understand that the function is intended for column names specifically.
24. fnToNumber as a culture-aware number parser
This is the query’s main numeric conversion function. It first tries direct numeric coercion, then tries text parsing using the configured culture, then tries alternate cleanup strategies for text that contains different decimal or thousand separators.

That makes it much more resilient than a naive Number.From call. It can handle values that arrive as actual numbers, numeric text, or locale-formatted text.
25. Why fnToNumber tries multiple parsing paths
One source may output 1,234.56, another may output 1.234,56, and another may output a plain 1234.56. A single parser can fail on some of those forms even though the value is clearly numeric to a human.

Multiple parsing attempts greatly improve the chance of successful conversion without manual cleanup.
26. Why the query uses Number.FromText with fallback cleanup
If a string cannot be parsed directly, the function tries to remove or swap separators in a way that increases the chance of a valid number. That is especially useful when source systems use different locale conventions than the workbook culture.

It is a practical compromise between strict parsing and over-permissive coercion.
27. fnToInt as controlled integer coercion
This function converts a value to a number and then rounds it down to an integer. The final cast to Int64 gives a stable integer output type.

This is useful for line numbers, counts, and identifiers that should not remain as decimals.
28. Why rounding down is used for integers
Some imported values may be decimal-like even if they conceptually represent whole numbers. Rounding down before conversion avoids type failure while keeping the result deterministic.

This is not a rounding-to-nearest rule. It is a deliberate normalization rule for numeric fields that should behave like whole numbers.
29. fnToLogical as boolean normalization
This helper converts common truthy and falsy text values such as yes, no, true, false, 1, 0, x, and their variants into actual logical values.

That is very useful because boolean source fields are often stored as text in flat files or Excel sheets.
30. Why boolean normalization matters for operational flags
Many file feeds use text markers rather than actual logical types. Normalizing them early lets the query use consistent boolean tests later in quality checks or business logic.

Without this, the query would need repetitive text comparisons scattered throughout the pipeline.
31. fnToDate and date parsing resilience
This helper tries direct conversion first and then attempts text-based parsing in several cultures. That is important because date strings are one of the most variable things in multi-source ingestion.

A date may come in as a true date, a local-language string, or an English-style text date depending on the source file.
32. Why multiple date cultures are attempted
The query tries the configured culture first and then other likely formats. That increases coverage when files come from different systems or exports with different locale assumptions.

It helps the query survive date inconsistency without forcing a manual standardization step before ingestion.
33. fnToDateTime and fnToDateTimeZone as separate layers
Date, datetime, and datetimezone are not interchangeable in Power Query. The query correctly treats them as distinct target types and uses specialized helpers for each.

This avoids the common error of forcing all temporal fields through one generic coercion path that silently loses information.
34. Why separate temporal helpers are good design
Source metadata often arrives as datetime or datetimezone, while business dates often arrive as date only. Keeping these categories separate preserves meaning and prevents unwanted truncation or zone loss.

That matters for auditing file-modified times, ingestion timestamps, and transaction dates.
35. fnToCurrency for accounting-style numeric text
This helper strips a string down to digits, separators, minus signs, and parentheses, then interprets parentheses as negative values. That is a classic accounting format rule.

It is especially useful when source systems export currencies with commas, dots, symbols, or bracketed negatives.
36. Why parentheses are treated as negative values
In financial reporting, (123.45) often means -123.45. The helper recognizes that convention directly and converts it into a signed number.

This is a strong sign the query is built with finance-style file data in mind.
37. fnAddIfMissing as a schema stabilizer
This function adds a null column only if the column is absent. That means the query can guarantee that important fields exist, even when some files do not carry them.

It is one of the most useful functions in the pipeline because it prevents missing-column failures later.
38. Why adding null columns is safer than failing immediately
Not every source file will contain every field. If a field is optional for the report, it is often better to create it as null and continue than to stop the entire refresh.

This keeps the pipeline operational while still making the missing data visible.
39. fnEnsureColumns and canonical schema enforcement
This helper first adds any missing columns from a required list, then reorders the table so those columns appear first in the expected sequence, followed by any extras.

That is a very strong pattern for schema drift management. It lets the query establish a predictable column contract without destroying additional columns that may still be useful.
40. Why canonical ordering matters
Stable output column order is important for readability, downstream formulas, and repeatable refresh behavior. If the same report changes column order every time, it becomes harder to maintain and compare.

This helper prevents that problem by explicitly shaping the table layout.
41. fnDistinctNonNull as a small list utility
This helper removes nulls and returns distinct values from a list. It is a basic utility, but it is the kind of thing that supports higher-level metadata handling and filtering patterns.

It fits the overall style of the query: small reusable helpers for common defensive tasks.
42. fnContainsAnyText for text pattern detection
This helper checks whether a text value contains any pattern in a list. It lowercases the text and evaluates pattern hits across the list.

It is useful for classification logic, filename checks, sheet-name checks, or label-based routing.
43. Why having extra helpers is good even if not all are heavily used
In production query work, utility functions often get reused across versions. A helper may not be central in the visible flow today but may become essential in a later refinement.

Having a richer helper library shows the query was built as a maintainable framework rather than as a one-off script.
44. fnGetWorkbookTable as optional workbook access
This function tries to load a named table from the current workbook. If the table does not exist, it returns an empty table, optionally with fallback columns.

That is an excellent way to make optional dimensions truly optional. The whole pipeline does not break just because a reference table is absent.
45. Why empty fallback tables matter
If a dimension is missing, the query can still complete the fact load and simply leave enrichment fields null. That is much better than forcing the whole dataset to fail because one lookup table was not supplied.

This is the difference between a brittle report and a resilient reporting framework.
46. fnStandardizeTable as a general cleanup pass
This function normalizes column names and trims all text columns in the table. It applies the same hygiene principles to both workbook dimensions and file-loaded data.

The result is a more stable base for joins, aliases, and type conversion.
47. Why text columns are cleaned after column names are normalized
Column names affect schema behavior; text values affect row behavior. They need different cleanup logic, but both should be handled before the data reaches the business-rule layer.

That is exactly what this helper does.
48. fnApplyAliases and source-name harmonization
This function maps many different source header variants into canonical business names such as document_no, transaction_date, customer_code, product_code, quantity, unit_price, discount, tax, gross_amount, and net_amount.

This is one of the most important parts of the query because it turns many possible source schemas into one shared internal schema.
49. Why alias mapping is essential in file ingestion
Real source systems rarely agree on exact naming. Some use abbreviations, some use full phrases, some use inconsistent underscores, and some use different export conventions entirely.

Alias mapping collapses that diversity into one canonical vocabulary, which makes the rest of the query far easier to write and maintain.
50. The alias list reveals the business domain
The names point to transaction-level sales or order data: invoices, vouchers, document dates, customer codes, item codes, quantities, unit prices, amounts, discounts, tax, and units of measure.

This is not a generic file ingestion model. It is clearly designed for transactional reporting and analysis.
51. Why doc_no, invoice_no, and similar variants all point to the same internal field
When different systems produce different labels for the same concept, the ingestion layer needs to unify them. The alias list does that by mapping variants to one internal name.

This is one of the best ways to absorb schema drift without rewriting downstream logic for each source variant.
52. fnPickBestItem as the workbook sheet selection engine
For workbook files, the query does not blindly take the first sheet or table. It filters visible workbook items, normalizes their names, checks for preferred sheet names, and selects the best match if one exists.

That is a very practical way to deal with workbooks that contain multiple tabs, summary pages, or staging sheets.
53. Why visible items are filtered first
Hidden workbook objects may be helper tabs, temporary calculations, or non-source content. Filtering them out reduces the chance of importing the wrong thing.

This is an important safety step in workbooks that contain both data and presentation content.
54. Why normalization is applied before comparing sheet names
Sheet names may contain spaces, uppercase letters, or punctuation. Normalizing them before comparison allows the preferred list to match more reliably even if the workbook naming style is inconsistent.

This prevents minor naming differences from breaking the selection logic.
55. Why preferred sheets are selected when available
Not every workbook should be treated as a generic data container. Often the correct sheet has a recognizable name like raw or transactions. Prioritizing those names gives the query a simple and robust heuristic for finding the main data source.

If no preferred item exists, the query still falls back to the first available item, so it remains functional.
56. fnLoadBinary as the file parser and metadata enricher
