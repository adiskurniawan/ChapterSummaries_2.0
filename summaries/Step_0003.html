1. What this query is doing in operational terms
This Power Query is a folder-based ingestion, harmonization, enrichment, and validation pipeline. It continuously scans a designated folder, loads all supported files inside it, standardizes their structures into a single canonical transaction schema, enriches the rows with optional customer and product dimension data from the current workbook, flags incomplete or suspicious records, detects duplicate line combinations, and returns a final report-ready table. The query is doing the work of a compact data warehouse staging layer inside Excel.

That matters because the query is not merely “opening files.” It is imposing order on heterogeneous raw source material. It is deciding which files count, which worksheets count, which columns mean the same thing, how values should be typed, what counts as an incomplete record, and what counts as a duplicate. In other words, it is operationalizing a repeatable ingestion policy.
2. Why the query is best understood as a pipeline rather than a single transformation
The query uses a chain of staged steps, and each stage has a distinct responsibility. The configuration section sets parameters. The helper section defines reusable cleaning logic. The loader section resolves file and workbook variation. The alias section normalizes schema differences. The normalization section imposes a canonical structure. The enrichment section attaches dimensional context. The quality section adds flags and duplicate counts. The final section controls presentation order.

That structure is important because each layer depends on the previous layer being sane. If the source is messy, the query does not collapse immediately. It gradually converts chaos into structure. That is the hallmark of a well-designed ingestion pipeline.
3. The configuration block defines the operational contract
SourceFolder, CsvDelimiter, AcceptedExtensions, PreferredSheetNames, and RefreshUtc are not decorative constants. They define the execution contract. They tell the query where to look, what kinds of files to accept, how to parse CSV, how to choose among multiple workbook sheets or tables, and how to timestamp the refresh.

This is a strong design choice because it separates environment-specific assumptions from the logic itself. If the folder changes, the delimiter changes, or the preferred sheet names change, the query can be adapted without rewriting the core transformations.
4. Why SourceFolder is the first true dependency
The entire ingestion starts with a filesystem folder. Everything downstream depends on files being present there. That means the query is designed for a drop-zone workflow: users or other systems place files in the folder, then refresh the query to rebuild the dataset. This is a common and practical pattern for operational Excel models.
5. Why the query accepts several file extensions
Supporting .csv, .xlsx, .xls, and .xlsb means the query can ingest text exports, modern Excel workbooks, legacy Excel workbooks, and binary Excel workbooks. That flexibility is very useful because source systems often do not export in one ideal format. The query therefore reduces the burden on upstream users and can absorb mixed source formats in the same folder.
6. Why PreferredSheetNames matters
Excel workbooks can contain multiple visible sheets and tables, and only some of them are relevant. The priority list {"data", "raw", "transactions", "sheet1"} gives the query a soft heuristic for choosing the likely data-bearing object. That is much better than selecting a random sheet blindly. It reflects the common naming conventions used in operational files.
7. Why RefreshUtc is captured once at the beginning
DateTimeZone.FixedUtcNow() returns one stable refresh timestamp for the whole evaluation. That is the correct choice for lineage fields because every row produced by a single refresh should share the same ingestion timestamp. If the query called a live current-time function later on in separate steps, the timestamps could drift across rows. The fixed timestamp avoids that inconsistency.
8. fnCleanText is the basic sanitation primitive
This helper converts any value to text where possible, cleans control characters, trims whitespace, and turns empty strings into null. It is used everywhere text cleanliness matters. That includes source cleanup, dimension cleanup, alias handling, and value normalization after loading.

This kind of helper is essential because raw workbook and CSV data often contains invisible formatting debris. A cell that looks blank may actually contain spaces or nonprinting characters. Those can break joins, comparisons, and duplicate logic if not handled early.
9. Why blank-looking text becomes null
Power Query distinguishes between blank text and null, and that distinction matters. Blank text can still behave like a value in comparisons or joins, while null behaves like missing data. Turning empty or whitespace-only text into null makes later validation and filtering more accurate. It also improves semantic clarity in the final table.
10. fnNormalizeHeader is doing schema canonicalization, not just pretty formatting
This function lowercases headers, removes control characters, replaces spaces and punctuation with underscores, removes brackets, collapses repeated underscores, and trims excess underscore boundaries. The goal is not cosmetic prettiness. The goal is to create a stable, machine-friendly header vocabulary that can support alias mapping and downstream selection.

That is a key difference. The function is standardizing the structural meaning of headers so that variation in user-facing formatting does not disrupt the query.
11. Why the header normalization approach is robust
Source headers may arrive as Invoice No, INVOICE NO, invoice-no, invoice.no, or even Invoice No with trailing spaces. fnNormalizeHeader collapses all of those into a consistent form that can later map to document_no. That reduces the amount of special-case handling required later in the query and makes cross-file interoperability much stronger.
12. The CR/LF cleanup in headers is important in real-world files
Some Excel exports embed line breaks inside header cells. Those line breaks are invisible in the UI but problematic in Power Query because they create unpredictable field names. Replacing CR/LF markers with spaces reduces that risk. It is a small step that prevents a surprisingly annoying class of schema bugs.
13. fnToNumber is designed for mixed-format numeric values
This helper is more than a simple cast. It tries direct numeric conversion first, then attempts text parsing using English rules, and finally tries a fallback that strips thousands separators and converts decimal commas to decimal points. That makes it tolerant of values like 1,234.56, 1234.56, 1.234,56, and numbers that are stored as text.

This is exactly the kind of resilience a folder ingestion query needs because the same business quantity may be exported differently across systems or user sessions.
14. Why the helper tries multiple parsing strategies instead of one
Different source systems and locales represent numbers differently. One system may use commas as thousand separators and dots as decimals. Another may reverse the convention. Some files may already have numbers typed correctly, while others may preserve text. The helper gives the query several chances to recover a numeric value before surrendering to null. That is practical, not theoretical.
15. The en-US parsing attempt is a deliberate anchor
Even when files come from mixed environments, an English-style parse is often the safest first text conversion because it handles common export formats and standard numeric text shapes. The fallback after that handles some of the common locale variants. The helper is not trying to solve every locale on earth. It is trying to be robust for the workbook environment it expects.
16. fnToDate is deliberately tolerant of source ambiguity
The date helper first tries Date.From directly, then attempts text parsing in English, then Indonesian. That means typed dates, text dates, and locale-specific dates all have a chance to become valid dates. This is important because transaction date is one of the most important fields in the final dataset, and date formats are one of the most common import problems.
17. Why supporting both English and Indonesian date styles is practical
In an Indonesian workbook environment, files may arrive from local sources and international sources. Date text may therefore appear in different conventions. This helper accepts both without forcing an upstream pre-cleaning step. That makes the query more reusable across mixed source origins.
18. The alias block is a schema unification layer
fnApplyAliases takes a pile of source vocabulary variants and maps them to canonical business names. doc_no, invoice_no, voucher_no, and trx_no all become document_no. trx_date, date, and posting_date all become transaction_date. cust_code and the customer ID variants all become customer_code. This is critical because different source files often use different labels for the same concept.

Without this step, the query would have to keep writing source-specific branches. With it, the rest of the logic can operate on one stable vocabulary.
19. Why aliasing is better than custom logic scattered throughout the query
When schema differences are localized in one alias mapping table, the query becomes easier to maintain and extend. Adding a new synonym is a single-line edit. If alias handling were embedded in many later steps, the logic would become fragmented and much harder to debug. Centralizing schema variance is a strong data-engineering habit.
20. The alias list reveals the business domain
The synonyms suggest a transaction or sales-style dataset: document number, transaction date, customer code, product code, quantity, unit price, and net amount. Those fields are typical of line-level commercial records. That tells you the query is not just loading arbitrary files. It is preparing a transactional fact table for analysis.
21. fnPickTable is the workbook-object resolver
Excel files can contain multiple sheets, tables, hidden objects, and miscellaneous metadata entries. fnPickTable filters candidates to visible tables and sheets, then gives preference to names that match the priority list. If nothing preferred exists, it falls back to the first available visible candidate. That is a practical and general way to choose a likely source object from a workbook file.
22. Why hidden workbook objects are excluded
Hidden sheets are often support tabs, staging tabs, or accidental leftovers. Ingestion queries should usually avoid them unless explicitly intended. By excluding hidden objects, the query reduces the risk of pulling in the wrong worksheet. That lowers the chance of invalid data entering the combined dataset.
23. Why the loader promotes headers after selecting a workbook object
Some Excel objects are already structured, and some are not. Promoting headers only after selecting the candidate table or sheet keeps the process localized. If promotion fails, the query leaves the selected data alone rather than crashing. That tolerance is very useful in mixed-source workbook environments.
24. Why the workbook loader normalizes headers again after promotion
Even after selecting a workbook table, the headers may still be formatted inconsistently. Applying fnNormalizeHeader after promotion ensures that all column names become machine-friendly and comparable before the rest of the pipeline proceeds. This second normalization pass is necessary rather than redundant.
25. fnLoadFile is the core file-to-table converter
This function receives the binary content of one file plus its metadata, determines how to parse it, normalizes the resulting table, adds lineage fields, and returns a clean ingest result or an error row. It is the engine of the folder ingestion pipeline. Without this function, the query would not be able to process each file uniformly.
26. Why CSV and Excel are treated differently inside fnLoadFile
CSV is a flat text format, so it uses Csv.Document. Excel is a workbook format, so it uses Excel.Workbook. This distinction matters because the parse strategy depends on the physical structure of the file. The query respects that difference rather than forcing one parser onto both formats.
27. Why UTF-8 and CSV quote handling matter
Using UTF-8 encoding and CSV quote style increases the chance that special characters and quoted delimiters are interpreted correctly. That is particularly important when source text contains non-ASCII characters or commas inside text values. These details reduce import errors in text-based sources.
28. Why CSV headers are promoted immediately
Most CSV exports are row-based with the first row as headers. Promoting headers immediately makes the rest of the CSV path compatible with the canonical schema pipeline. If header promotion fails, the query preserves the raw table so the load attempt still has a chance to proceed.
29. Why the loader trims text columns before adding metadata
It is cleaner to standardize the row-level source values before the lineage columns are added. That ensures the business fields are cleaned first, then the metadata is appended. The resulting output is more consistent. The lineage columns themselves do not need the same text trimming because they are being generated by the query rather than imported from the source.
30. The lineage fields provide traceability
The query adds source_file, source_extension, source_folder, source_full_path, source_file_size, source_modified_utc, ingested_utc, load_status, and load_error. These fields let the user trace each row back to its source file and understand how the row entered the system. That is valuable for auditability and debugging.
31. Why source_full_path is especially useful
This field combines the folder path and file name, so a row can be traced directly to a physical file path. When a problem row appears in the final table, that path is often the fastest way to locate the original file on disk.
32. Why load_status and load_error are so important
Instead of failing the whole refresh on one bad file, the query tries to preserve an error record with status ERROR and the associated message. Successful loads get OK. That is operationally much better than a brittle all-or-nothing approach because it preserves visibility into failed files while still loading the rest of the folder.
33. The error fallback is structured, not ad hoc
The fallback for load failures returns a one-row table with the same lineage columns as a successful row would have. That means the combined dataset can still be processed uniformly later. This is a strong design choice because even failed loads remain visible in the final pipeline.
34. Why the query uses an explicit typed error table
Returning a typed table with nullable columns ensures the downstream Table.Combine step can still work across successful and failed file loads. If the error fallback returned a differently shaped object, the combine step could become unstable. The typed fallback preserves schema compatibility.
35. The current workbook dimension tables are optional enrichment sources
tblCustomerDim and tblProductDim are read from the workbook only if they exist. If not, the query substitutes empty typed tables. That means the core ingestion can still run without the dimensions. The enrichment is helpful but not mandatory. This makes the solution flexible and modular.
36. Why the dimensions are read from the current workbook instead of external files
That keeps the enrichment close to the report model. Users can maintain small lookup tables directly in the workbook without managing separate files. It also makes the whole model easier to refresh and distribute because the lookup data travels with the workbook.
37. Why the dimension tables are normalized the same way as the source files
Applying header normalization and text cleaning to the lookup tables ensures they use the same canonical naming and value style as the transactional data. That consistency is important because joins are sensitive to exact key matches. If the lookup tables remained messy, the enrichment would be unreliable.
38. Why the dimension tables are reduced to a narrow set of columns

1. What this query is in operational terms
This is a full ingestion-and-curation Power Query designed to turn a folder of messy transaction files into one standardized reporting table. It reads multiple file formats, normalizes the schema, enriches the rows with optional dimensions, applies data-quality checks, detects duplicates, optionally deduplicates to the latest record per transaction key, and finally returns a clean analytical output.

It is not a simple import query. It is a complete ETL pipeline embedded inside Power Query.
2. The core design philosophy
The query is built around the idea that raw files are unreliable, inconsistent, and incomplete. Instead of assuming every source file is perfect, it anticipates schema drift, type mismatch, late data, missing tables, bad text, duplicated rows, and parsing failures.

That is why the code has so many helper functions. Each helper reduces fragility in one specific area so the whole pipeline can survive imperfect real-world inputs.
3. Why the code is organized into named blocks
The query is segmented into configuration, helper functions, workbook dimension handling, file discovery, schema normalization, type coercion, derivations, joins, data-quality flags, duplicate detection, deduplication, and final shaping. Each section has a distinct responsibility.

This structure matters because Power Query can become unreadable very quickly if all logic is mixed together. Here, the author has separated logic into phases that mirror a real data-engineering workflow.
4. Why the Config record is important
Config acts as a single control panel for the query. It holds the source folder, archive folder, accepted extensions, CSV parsing settings, culture, preferred sheet names, incremental lookback window, dedupe behavior, archive filtering behavior, and debug mode.

This is a strong design choice because it makes the query easier to maintain. The behavior of the whole pipeline can be changed by adjusting one record rather than rewriting many lines of logic.
5. SourceFolder and ArchiveFolder as operational boundaries
The source folder tells the query where active files live. The archive folder gives the query a place to exclude if archived files should not be ingested.

This separation is useful in production setups where active and historical files are stored in the same broad file structure but should not always be treated the same way.
6. Why accepted extensions are explicit
The query only accepts .csv, .xlsx, .xls, .xlsb, and .txt. That limits the ingestion surface to known data-bearing file types and avoids accidental parsing of unrelated files.

This is a small but important guardrail. It reduces garbage input before the parser even starts.
7. Why CSV delimiter and encoding are configurable
CSV files vary across systems, especially in delimiter choice and encoding. By placing the delimiter and encoding in the config, the query can be adapted for different source systems without touching the main ingestion logic.

That is a practical sign that the author expects the pipeline to be reused in more than one workbook or environment.
8. Why Culture = "en-US" matters
Culture affects how numbers and dates are interpreted. A culture-sensitive parser is important because the same text can be read differently depending on locale conventions for decimal separators, date formats, and thousand separators.

By setting culture explicitly, the query reduces ambiguity and keeps parsing behavior predictable.
9. The purpose of PreferredSheetNames
Workbook files often contain many sheets, but only one or two are the real data source. The preferred sheet list tells the query which names should be favored first, such as data, raw, transactions, sheet1, and input.

This is a very practical heuristic. It helps the query avoid loading title sheets, summary tabs, or decorative tabs by mistake.
10. Why IncrementalDaysBack is operationally useful
This setting defines how far back the query should look when filtering recent files by modified date. It lets the pipeline behave like a rolling incremental ingestion process rather than a full folder scan every time.

That helps both performance and control. Large folders remain manageable when the query only considers a recent window of files.
11. KeepOnlyLatestPerKey as a business switch
This setting decides whether the final result should collapse duplicates and keep only the latest record per transaction key. That is a major business behavior toggle, not just a technical parameter.

It means the same query can support either audit-style full retention or reporting-style latest-state extraction.
12. Why UseArchiveFilter is optional
Not every deployment needs archive exclusion. Some file structures separate active and archived data cleanly; others do not. Making archive filtering optional allows the query to adapt to both styles.

This is a good example of avoiding hardcoded assumptions that might not fit every workbook.
13. Why DebugMode exists
Debug mode lets the query preserve more diagnostic detail instead of hiding error rows. That is extremely useful while developing, testing, or troubleshooting the pipeline.

In production, the user may want only clean rows. In development, the user may want the error rows and metadata preserved so they can see what failed.
14. RefreshUtc as a stable timestamp anchor
The query captures the exact UTC refresh time once and then reuses it throughout the pipeline. That means all rows produced in the same refresh share the same ingestion timestamp.

This is important for auditability. It tells you when the dataset was built without relying on volatile recalculation inside later steps.
15. MinModifiedDate and incremental scope
The minimum modified date is calculated by subtracting the configured lookback window from the current date. Files older than that cutoff are excluded from active ingestion.

This is a performance and governance feature. It keeps the query focused on recent data and prevents unnecessary reprocessing of old files.
16. Why the query starts with helper functions
Helper functions turn repeated micro-patterns into reusable building blocks. Instead of writing custom conversion, normalization, and fallback logic over and over, the query defines them once and reuses them throughout the pipeline.

This is what makes the query look long but still maintainable. The length is mostly due to structure, not duplication.
17. fnTry as a safe function wrapper
This helper calls another function and returns either the result or a default value if an error occurs. It is a generic safety wrapper.

That is useful when a step may fail for benign reasons and the query should continue rather than halt. It is a tiny function with broad defensive value.
18. Why fnTry is useful in Power Query specifically
Power Query can fail on type issues, malformed text, missing rows, or unexpected source structures. A wrapper like fnTry helps keep those errors local rather than letting them crash the entire query.

In a multi-file ingestion pipeline, controlled failure handling is a major advantage.
19. fnCleanText as a universal text sanitizer
This function converts a value to text safely, trims whitespace, cleans nonprinting characters, and turns blank strings into nulls.

That is essential because messy input often contains invisible characters, leading/trailing spaces, or blank text values that look empty but are not logically null.
20. Why blank-to-null conversion matters
In Power Query, blank text and null are not always treated the same way. Converting empty text to null makes downstream logic cleaner, especially for joins, filters, and required-field checks.

It reduces ambiguity and prevents “empty but not really empty” data from slipping through quality rules.
21. fnNormalizeName as the schema-shaping engine
This helper forces a header or field name into a canonical internal form. It lowercases the text, cleans it, removes line breaks, replaces spaces and punctuation with underscores, strips brackets and quotes, collapses repeated underscores, and trims leading or trailing underscores.

This is one of the most important helpers in the entire query because schema drift usually begins with inconsistent naming.
22. Why schema normalization is a central strategy here
Real-world source files do not always use the same header spelling, spacing, or punctuation. One file may use Invoice No, another invoice_no, another Invoice-No, and another Invoice.No.

Normalizing the name to a shared internal pattern makes later alias mapping and joins far more reliable.
23. fnNormalizeHeader as a thin wrapper
This helper simply delegates to fnNormalizeName. It exists for semantic clarity, because headers and general names are conceptually related but not always discussed the same way in maintenance work.

That may seem minor, but it helps readers understand that the function is intended for column names specifically.
24. fnToNumber as a culture-aware number parser
This is the query’s main numeric conversion function. It first tries direct numeric coercion, then tries text parsing using the configured culture, then tries alternate cleanup strategies for text that contains different decimal or thousand separators.

That makes it much more resilient than a naive Number.From call. It can handle values that arrive as actual numbers, numeric text, or locale-formatted text.
25. Why fnToNumber tries multiple parsing paths
One source may output 1,234.56, another may output 1.234,56, and another may output a plain 1234.56. A single parser can fail on some of those forms even though the value is clearly numeric to a human.

Multiple parsing attempts greatly improve the chance of successful conversion without manual cleanup.
26. Why the query uses Number.FromText with fallback cleanup
If a string cannot be parsed directly, the function tries to remove or swap separators in a way that increases the chance of a valid number. That is especially useful when source systems use different locale conventions than the workbook culture.

It is a practical compromise between strict parsing and over-permissive coercion.
27. fnToInt as controlled integer coercion
This function converts a value to a number and then rounds it down to an integer. The final cast to Int64 gives a stable integer output type.

This is useful for line numbers, counts, and identifiers that should not remain as decimals.
28. Why rounding down is used for integers
Some imported values may be decimal-like even if they conceptually represent whole numbers. Rounding down before conversion avoids type failure while keeping the result deterministic.

This is not a rounding-to-nearest rule. It is a deliberate normalization rule for numeric fields that should behave like whole numbers.
29. fnToLogical as boolean normalization
This helper converts common truthy and falsy text values such as yes, no, true, false, 1, 0, x, and their variants into actual logical values.

That is very useful because boolean source fields are often stored as text in flat files or Excel sheets.
30. Why boolean normalization matters for operational flags
Many file feeds use text markers rather than actual logical types. Normalizing them early lets the query use consistent boolean tests later in quality checks or business logic.

Without this, the query would need repetitive text comparisons scattered throughout the pipeline.
31. fnToDate and date parsing resilience
This helper tries direct conversion first and then attempts text-based parsing in several cultures. That is important because date strings are one of the most variable things in multi-source ingestion.

A date may come in as a true date, a local-language string, or an English-style text date depending on the source file.
32. Why multiple date cultures are attempted
The query tries the configured culture first and then other likely formats. That increases coverage when files come from different systems or exports with different locale assumptions.

It helps the query survive date inconsistency without forcing a manual standardization step before ingestion.
33. fnToDateTime and fnToDateTimeZone as separate layers
Date, datetime, and datetimezone are not interchangeable in Power Query. The query correctly treats them as distinct target types and uses specialized helpers for each.

This avoids the common error of forcing all temporal fields through one generic coercion path that silently loses information.
34. Why separate temporal helpers are good design
Source metadata often arrives as datetime or datetimezone, while business dates often arrive as date only. Keeping these categories separate preserves meaning and prevents unwanted truncation or zone loss.

That matters for auditing file-modified times, ingestion timestamps, and transaction dates.
35. fnToCurrency for accounting-style numeric text
This helper strips a string down to digits, separators, minus signs, and parentheses, then interprets parentheses as negative values. That is a classic accounting format rule.

It is especially useful when source systems export currencies with commas, dots, symbols, or bracketed negatives.
36. Why parentheses are treated as negative values
In financial reporting, (123.45) often means -123.45. The helper recognizes that convention directly and converts it into a signed number.

This is a strong sign the query is built with finance-style file data in mind.
37. fnAddIfMissing as a schema stabilizer
This function adds a null column only if the column is absent. That means the query can guarantee that important fields exist, even when some files do not carry them.

It is one of the most useful functions in the pipeline because it prevents missing-column failures later.
38. Why adding null columns is safer than failing immediately
Not every source file will contain every field. If a field is optional for the report, it is often better to create it as null and continue than to stop the entire refresh.

This keeps the pipeline operational while still making the missing data visible.
39. fnEnsureColumns and canonical schema enforcement
This helper first adds any missing columns from a required list, then reorders the table so those columns appear first in the expected sequence, followed by any extras.

That is a very strong pattern for schema drift management. It lets the query establish a predictable column contract without destroying additional columns that may still be useful.
40. Why canonical ordering matters
Stable output column order is important for readability, downstream formulas, and repeatable refresh behavior. If the same report changes column order every time, it becomes harder to maintain and compare.

This helper prevents that problem by explicitly shaping the table layout.
41. fnDistinctNonNull as a small list utility
This helper removes nulls and returns distinct values from a list. It is a basic utility, but it is the kind of thing that supports higher-level metadata handling and filtering patterns.

It fits the overall style of the query: small reusable helpers for common defensive tasks.
42. fnContainsAnyText for text pattern detection
This helper checks whether a text value contains any pattern in a list. It lowercases the text and evaluates pattern hits across the list.

It is useful for classification logic, filename checks, sheet-name checks, or label-based routing.
43. Why having extra helpers is good even if not all are heavily used
In production query work, utility functions often get reused across versions. A helper may not be central in the visible flow today but may become essential in a later refinement.

Having a richer helper library shows the query was built as a maintainable framework rather than as a one-off script.
44. fnGetWorkbookTable as optional workbook access
This function tries to load a named table from the current workbook. If the table does not exist, it returns an empty table, optionally with fallback columns.

That is an excellent way to make optional dimensions truly optional. The whole pipeline does not break just because a reference table is absent.
45. Why empty fallback tables matter
If a dimension is missing, the query can still complete the fact load and simply leave enrichment fields null. That is much better than forcing the whole dataset to fail because one lookup table was not supplied.

This is the difference between a brittle report and a resilient reporting framework.
46. fnStandardizeTable as a general cleanup pass
This function normalizes column names and trims all text columns in the table. It applies the same hygiene principles to both workbook dimensions and file-loaded data.

The result is a more stable base for joins, aliases, and type conversion.
47. Why text columns are cleaned after column names are normalized
Column names affect schema behavior; text values affect row behavior. They need different cleanup logic, but both should be handled before the data reaches the business-rule layer.

That is exactly what this helper does.
48. fnApplyAliases and source-name harmonization
This function maps many different source header variants into canonical business names such as document_no, transaction_date, customer_code, product_code, quantity, unit_price, discount, tax, gross_amount, and net_amount.

This is one of the most important parts of the query because it turns many possible source schemas into one shared internal schema.
49. Why alias mapping is essential in file ingestion
Real source systems rarely agree on exact naming. Some use abbreviations, some use full phrases, some use inconsistent underscores, and some use different export conventions entirely.

Alias mapping collapses that diversity into one canonical vocabulary, which makes the rest of the query far easier to write and maintain.
50. The alias list reveals the business domain
The names point to transaction-level sales or order data: invoices, vouchers, document dates, customer codes, item codes, quantities, unit prices, amounts, discounts, tax, and units of measure.

This is not a generic file ingestion model. It is clearly designed for transactional reporting and analysis.
51. Why doc_no, invoice_no, and similar variants all point to the same internal field
When different systems produce different labels for the same concept, the ingestion layer needs to unify them. The alias list does that by mapping variants to one internal name.

This is one of the best ways to absorb schema drift without rewriting downstream logic for each source variant.
52. fnPickBestItem as the workbook sheet selection engine
For workbook files, the query does not blindly take the first sheet or table. It filters visible workbook items, normalizes their names, checks for preferred sheet names, and selects the best match if one exists.

That is a very practical way to deal with workbooks that contain multiple tabs, summary pages, or staging sheets.
53. Why visible items are filtered first
Hidden workbook objects may be helper tabs, temporary calculations, or non-source content. Filtering them out reduces the chance of importing the wrong thing.

This is an important safety step in workbooks that contain both data and presentation content.
54. Why normalization is applied before comparing sheet names
Sheet names may contain spaces, uppercase letters, or punctuation. Normalizing them before comparison allows the preferred list to match more reliably even if the workbook naming style is inconsistent.

This prevents minor naming differences from breaking the selection logic.
55. Why preferred sheets are selected when available
Not every workbook should be treated as a generic data container. Often the correct sheet has a recognizable name like raw or transactions. Prioritizing those names gives the query a simple and robust heuristic for finding the main data source.

If no preferred item exists, the query still falls back to the first available item, so it remains functional.
56. fnLoadBinary as the file parser and metadata enricher
This is the main function that turns one file into one standardized table. It handles CSV/TXT via Csv.Document, workbook files via Excel.Workbook, standardizes the resulting table, and then adds source metadata columns like file name, extension, folder, path, size, modified time, ingestion time, load status, and load error.

This is the heart of the ingestion pipeline.
57. Why CSV and TXT are parsed together
Both are text-based flat files and can be handled by the same CSV document parser with an adjustable delimiter and encoding. Treating them together keeps the logic simple and avoids duplicated branches.

That is a sound design decision when both formats are conceptually flat text sources.
58. Why Table.PromoteHeaders is attempted after CSV parsing
Many CSV exports have a first row that should become the column names. The query attempts to promote headers but wraps that in try because some files may not be perfectly structured.

This prevents a single malformed file from breaking the entire ingestion run.
59. Why workbook parsing is different from CSV parsing
Workbook files contain multiple sheets, tables, and metadata layers. They must be opened as a workbook structure before the actual data can be selected.

That is why the workbook branch uses Excel.Workbook and then passes the workbook content to the sheet-selection helper.
60. Why source metadata columns are so important
Adding file name, extension, folder, path, file size, modified time, and ingestion time gives every row provenance. You can later trace where a row came from and when it was loaded.

That is critical for audit, debugging, and operational monitoring.
61. Why load_status and load_error matter
These fields encode whether the file loaded successfully and, if not, why it failed. That means file-level failures become visible in the data itself rather than hidden in refresh logs.

This makes the query much easier to troubleshoot in practice.
62. fnLoadBinarySafe as the error-preserving wrapper
This function runs the parser in a try block. If parsing succeeds, it returns the parsed table. If parsing fails, it returns a one-row error table that captures the file metadata and the error message.

This is a highly mature pattern because it preserves failure context instead of losing the failed file completely.
63. Why returning error rows is better than failing silently
If a file cannot be parsed, the user still needs to know which file caused the problem and what the error was. The one-row error table preserves that information in a structured way.

This makes operational debugging much faster.
64. Why the error row has the same metadata shape
Even failed files carry the same provenance fields as successful ones. That consistency makes the combined table easier to inspect and filter.

It also means the failure information can coexist with successful rows in one pipeline output if debug mode is enabled.
65. Loading optional workbook dimensions
The query loads customer, product, and store dimensions from named workbook tables if they exist. Each dimension is standardized, ensured to contain a target schema, cleaned, and deduplicated by its natural key.

This gives the fact data enrichment attributes without making the dimensions mandatory.
66. Why the dimensions are separate from the fact pipeline
Separating dimensions from fact ingestion is a star-schema-style design. It keeps the reference data distinct from the transaction data and makes the final join logic clearer.

It also means the dimensions can be maintained independently from the source file folder.
67. CustomerDim and key deduplication
The customer dimension is deduplicated on customer_code. That prevents duplicate lookup rows from causing row multiplication during the join.

This is a very important defensive measure in dimensional enrichment.
68. Why customer_code is the natural key here
Code fields are usually more stable than display names. Using the code as the key reduces the chance of mismatched joins caused by formatting differences or name variants.

That is standard and sensible practice for lookup tables.
69. The same pattern for ProductDim
The product dimension carries product code, brand, category, product group, and pack size, then deduplicates on product_code.

This means the final transaction table can be enriched with descriptive product hierarchy fields without changing the underlying fact records.
70. The same pattern for StoreDim
The store dimension provides store code, store name, and store region, then deduplicates on store_code.

Again, this is classic lookup-table design and helps the final report speak in business terms instead of only raw codes.
71. Why text cleaning on dimensions matters
Dimension joins are fragile when key text has trailing spaces or hidden characters. Cleaning and trimming the dimension values improves match rates and reduces false nonmatches.

This is one of the simplest ways to increase join reliability.
72. File discovery starts with Folder.Files
The query enumerates all files in the source folder, then filters them according to visibility, extension, and modified date.

This is the expected pattern for a folder-based ingestion pipeline. It allows new files to be picked up automatically on refresh.
73. Why hidden files are excluded
Hidden files are often temporary artifacts, internal system files, or non-source material. Excluding them reduces the risk of loading noise into the fact table.

That keeps the ingestion layer focused on user-relevant files only.
74. Why extension filtering is necessary
The folder may contain many file types. The query should only attempt to parse files that make sense in the ingestion context.

This avoids wasting time on unsupported files and lowers the chance of parse errors.
75. Why the date-modified filter is a practical incremental approach
By only looking at relatively recent files, the query behaves like a rolling window ingestion process. That can dramatically reduce refresh time in large folders.

It also helps prevent repeated reprocessing of old files that have not changed.
76. Archive filtering as an optional second gate
If archive filtering is enabled, the query excludes any file whose folder path contains the archive folder string. That allows old or finalized files to remain in the file structure without entering active ingestion.

This is useful in environments that keep active and archived data under the same parent directory.
77. Why archive filtering is path-based rather than name-based
Folder path checks are often more reliable than file-name checks for distinguishing archive areas. A file name may change, but a folder placement rule is usually more stable.

That makes the archive filter a practical operational control.
78. AddParsed applies the file parser row by row
Each file record gets a new parsed column that contains the table created by fnLoadBinarySafe. That means every file becomes a nested table inside the file list.

This is a clean way to stage multiple parsed tables before combining them.
79. Table.Combine(AddParsed[parsed]) merges all parsed file tables
After each file is parsed into a table, the query combines them into one larger table. That is the point where the pipeline moves from file-level processing to record-level processing.

It is a crucial consolidation step because everything that follows assumes a unified row set.
80. Why combining after parsing is better than trying to parse everything at once
Parsing each file independently allows the query to isolate file-specific errors, preserve metadata, and standardize each source before merging. That is more robust than trying to force all files through one monolithic parse step.

It also makes debugging much easier.
81. Schema normalization begins after file combination
Once the files are combined, the query applies alias mapping to harmonize field names. This means downstream steps operate on one canonical vocabulary rather than many source-specific variants.

That is a major turning point in the pipeline. Raw file diversity becomes standardized fact data.
82. CanonicalColumns as the final target schema
This list defines the expected business, metadata, and quality columns that should exist in the standardized output. It includes transactional fields, source lineage fields, and diagnostic columns.

That list acts like the schema contract for the rest of the query.
83. Why the canonical list includes both business and metadata fields
The final table is not just for reporting business values. It also needs to support audit, lineage, and quality monitoring.

That is why the schema contains fields like source_file, source_extension, source_folder, ingested_utc, load_status, and load_error alongside transaction fields.
84. fnEnsureColumns is applied to the aliased table
After aliasing, the query forces the table to contain the canonical columns. Missing ones become null, and the table is reordered into a stable shape.

This guarantees that later coercion and derivation steps can reference a known column set.
85. Why this is safer than relying on whatever columns happened to exist
In a schema-drift environment, relying on accidental presence is dangerous. A missing column can break a transformation or produce inconsistent output.

Ensuring the column set first is one of the best ways to stabilize the pipeline.
86. Type coercion is grouped by expected output types
The query defines lists for numeric, date, datetime, datetimezone, and text fields. It then builds transformation operations only for columns that actually exist.

This is a clean and systematic way to apply types without hardcoding a separate transform for every field.
87. Why coercion is conditional on column existence
Optional columns may not exist in every source. If the query tried to coerce a missing field, it could fail unnecessarily.

Checking the actual table columns before building type operations makes the pipeline more resilient.
88. Number coercion through fnToNumber
All target numeric fields are passed through the robust numeric parser. This means textual numeric values, culture-specific numbers, and mixed formatting are all more likely to normalize correctly.

That is very important for fields like quantity, unit price, discount, tax, gross amount, net amount, and file size.
89. Date coercion through fnToDate
Transaction dates are normalized into actual dates. That lets the later derivations compute calendar year, month, quarter, and lateness accurately.

Without this step, the query would not be able to do date-based analytics reliably.
90. Why source_modified_utc is coerced as datetime and not date
File modified time is more granular than a day. Keeping it as datetime preserves the time-of-day information needed for auditing and dedupe decisions.

That is useful when multiple files are modified on the same date.
91. Why ingested_utc is a datetimezone
The refresh timestamp is recorded in UTC with zone awareness. That makes it consistent and safe for comparison across systems and time zones.

Using datetimezone also prevents ambiguity about the exact ingestion moment.
92. Text coercion and cleanup are applied after structural normalization
Text fields such as document numbers, codes, labels, notes, and source metadata are cleaned and trimmed again. That ensures any residual whitespace or nonprinting characters are removed before business logic uses them.

This second pass is valuable because earlier transformations may have introduced or preserved text inconsistencies.
93. Why the query applies both schema cleaning and value cleaning
Header normalization solves structure problems; text cleaning solves row-value problems. Both are necessary because bad data can live in both places.

This is one reason the query is robust. It addresses schema and content separately.
94. Converting blank text values to nulls
After trimming, any text cell that becomes an empty string is turned into null. That simplifies later field existence checks and quality rules.

It also reduces the risk that a blank-but-not-null field will be mistaken for a meaningful value.
95. Why blank-to-null conversion improves joins and filters
Null has a clear meaning in Power Query: missing value. Blank text can be more ambiguous, especially when coming from file imports. Converting to null makes later logic more direct and consistent.

This is especially useful for required-field checks and dimension matching.
96. Removing fully blank rows
The query removes rows where all fields are null. That is a cleanup step for malformed file fragments or accidental empty rows inside the source files.

It keeps the final table from being cluttered with non-data records.
97. row_hash as a record fingerprint
The query builds a hash-like row identity string from document number, line number, customer code, store code, product code, transaction date, and net amount. This gives each row a compact fingerprint for duplicate detection and traceability.

It is not a cryptographic hash, but it behaves like a practical row signature.
98. Why row_hash includes several business fields
A single transaction key may not be enough to detect whether two rows are truly the same or merely share one identifier. Including several fields creates a stronger identity check.

This reduces the chance of treating different rows as duplicates just because they share a document number.
99. transaction_key as the primary business key
The query defines the business key as document_no|line_no. That is a classic transaction-row identifier in document-based data models.

It gives the dedupe logic a simple, stable key to group on.
100. Why a composite key is better than a single field here
Many documents have multiple line items. A document number alone may identify the header, not the row. Adding the line number makes the key row-specific.

That is why the query combines the two fields instead of relying on one.
101. source_day as a file-age proxy
The query extracts the date portion of the source file modified timestamp. That creates a simple day-level marker for file recency and operational monitoring.

It can be useful when comparing file arrival patterns or identifying late deliveries.
102. Calendar derivations create reporting-ready fields
Year, month, month name, and quarter are all derived from transaction date. These fields are standard dimensions for reporting and grouping.

By generating them inside the query, the table becomes immediately usable for pivots and time-based summaries.
103. Why both month number and month name are useful
Month number is useful for sorting and numeric analysis, while month name is useful for display. Having both means the report can serve both machine and human needs without extra formulas later.

This is a classic reporting convenience pattern.
104. net_amount_signed applies document semantics to value sign
The query changes the sign of the net amount when the document type indicates a return. This means the table reflects business meaning rather than just raw source arithmetic.

That is important because returns typically subtract value from sales rather than add it.
105. Why document type controls sign
Some systems store returns as positive amounts with a separate type flag. Others store them as negative amounts directly. The query handles the business rule explicitly so the downstream data is consistent regardless of source convention.

This is a practical normalization strategy in transaction reporting.
106. gross_amount_signed follows the same sign rule
Gross amount receives the same return-aware sign treatment. That means both value measures remain internally aligned when returns are present.

Consistency between net and gross signed values is important for correct margin calculations.
107. Gross margin calculation as a derived measure
The query defines gross margin as signed gross amount minus signed net amount. That creates a row-level profitability or spread measure.

It is a core analytical field because it shows how much margin exists after the base amount is considered.
108. Gross margin percentage adds scale normalization
The query divides gross margin by signed gross amount when gross amount is nonzero. This turns the raw difference into a ratio that is easier to compare across rows of different sizes.

Ratio-based metrics are very useful in reporting because they help normalize large and small transactions.
109. Why zero gross amount is guarded against
Division by zero would cause an error or invalid result. The query avoids that by returning null when gross amount is absent or zero.

This is a basic but necessary numerical safeguard.
110. unit_price_check as a row-level validation metric
This derived value estimates the effective unit amount by dividing signed net amount by quantity. It can be used to validate source pricing consistency or detect anomalies.

It is not always a formal business measure, but it is very useful for audit and data-quality analysis.
111. Why quantity zero is guarded against in the unit price check
If quantity is zero, a unit price estimate is meaningless. Returning null is safer than forcing a mathematical error or a misleading result.

This preserves the integrity of the derived metric.
112. days_from_file_modified as a freshness measure
The query compares the transaction date to the file modified date and calculates the number of days between them. That gives a rough measure of how old the transaction appears relative to when the file changed.

This can help detect late arrivals, backdated entries, or operational delays in posting.
113. Why date freshness is useful in transaction pipelines
Transactions that are very old relative to the file modification date may indicate delayed batch loads, corrections, or manual backposting. The query surfaces that difference explicitly.

This is valuable for both operational monitoring and audit review.
114. is_late_record as a simple lateness flag
Any record more than 30 days away from file modification is marked late. This is a binary operational signal that can be easily filtered or highlighted in a report.

It is a straightforward threshold-based rule that translates a numeric age into an actionable status.
115. Why 30 days is a practical threshold
The threshold is arbitrary in the strict mathematical sense, but it is operationally meaningful. Many organizations treat older-than-30-day records as potentially late or delayed enough to merit attention.

This kind of threshold is common in data-quality and reporting workflows.
116. sign_flag as a simple categorical output
This column labels signed net amounts as NEGATIVE or POSITIVE. It gives a quick human-readable sign classification without requiring the user to inspect numeric values directly.

That can help with filtering and conditional formatting in reports.
